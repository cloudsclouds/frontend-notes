# 封装组件
## 封装 FollowTable 组件
1. 封装组件的背景是这样的，原生的 sticky 定位在处理嵌套容器滚动的时候，会出现吸顶失效的问题。
为了保证在复杂场景下的稳定性，放弃了纯 CSS 方案，采用 JavaScript 监听滚动事件，通过选择滚动容器、计算元素位置、复制表头来实现吸顶。

虽然成本稍高一些，但能够解决原生方案在复杂场景下的稳定性问题。

2. 另一个问题是，在做多表格页面的时候，发现当用户对最后一个表格进行筛选，数据变少时，页面总高度会突然收缩，导致底部内容跳动，体验很差。

为了解决这个问题，我在组件初始化时，专门计算了最后一个表格到底部的距离，并将其锁定。无论表格数据如何变化，都强制保持着底部间距不变，从而实现了视觉上的防抖动。

# 100+接口
这个报表页面的首屏需要渲染多个接口，如果全量并发会阻塞浏览器，使用promiseAll全部成功不允许失败，串行又太慢，难点在于部分列的数据依赖前一列的查询结果。

因此，我设计了一个并发池，这个并发池主要是通过一个任务队列和一个计数器来实现的，计数器就是记录当前正在执行的任务数量，具体实现的方式是拿到一个任务的时候，先去并发池里找这个计数器是否小于当前的一个限制并发数，如果小于的话说明任务可以执行，就可以继续发送请求，否则的话把它放在任务队列里面。当这个任务执行完成的时候，会在 finally 中执行释放这个计数器，并从任务队列里面拉去下一个任务进行执行。

在处理依赖任务的时候，利用 async、await串行等待前置的任务请求返回，保证数据准确。
对于请求失败的情况，做了兜底处理，就是展示默认横线，防止页面奔溃。

# 虚拟列表
针对长列表渲染导致的页面卡顿，引入了虚拟渲染技术，只渲染可视区域的节点，大幅减少了 DOM 操作，显著提升了滚动流畅度。

原理是计算可视窗口，监听滚动，用滚动距离除以固定行高，计算出起始索引，再结合窗口高度计算出结束索引，只渲染这一小段，配合上下留白撑高度，减少页面的卡顿。

为了防止滚动频繁出发重排，使用了RAF节流渲染，能让计算逻辑紧跟着浏览器的刷新频率。
