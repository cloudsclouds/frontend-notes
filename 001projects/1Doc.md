# BioNova AI
## 项目总述
### 1. 这个项目介绍一下，有哪些核心亮点，负责哪些内容？
这个项目是一个面向生物信息科研团队的协同文档与智能问答平台，主要解决两个问题：一是科研协作里多人同时改文档；二是专业知识密集、模型容易幻觉，写作和问答效率都不高。

我在项目里主要负责前后端开发和协同/AI链路设计，覆盖了实时协同编辑、流式问答、长上下文记忆管理、RAG 检索增强问答，以及文档内 AI 辅助创作的任务编排，帮助科研团队更快完成文献检索、知识整理、问答和内容生成。

## 简历串讲

### A. 实时协同编辑：Yjs + WebSocket
> 在协同编辑上，基于 Yjs + WebSocket 搭建了 CRDT 协同架构，
> 把文档操作建模成增量 update，同步到各客户端后自动完成冲突合并；
> 同时结合 Awareness 机制实现在线状态和光标同步，保障多人同时编辑时的一致性和流畅性。
> 针对异常场景，考虑了离线编辑后的状态收敛、网络抖动下的重连补偿等问题。

#### 0. TipTap 编辑器
Tiptap 是基于 ProseMirror 的富文本编辑器，和 Yjs 有官方适配的 y-prosemirror 插件，能直接把编辑器的操作映射成 Yjs 的共享状态修改，不用自己写复杂的转换逻辑。而且它支持自定义扩展，比如科研文档常用的公式、表格、引用块，都能方便地集成到协同编辑里。

#### 1. yjs 是什么，为什么要使用 yjs？
Yjs 是一个基于 CRDT 的实时协同编辑框架，它已经把文档状态同步、冲突合并、离线编辑、增量更新、最终一致性这些最难的问题封装好了。

#### 2. 实现“多人同时编辑”的流程。
1. 每个客户端打开文档后，会先从数据库拉取文档全量内容并初始化一个 Yjs 文档实例，Tiptap 绑定这个 Y.Doc文档；

2. 用户本地编辑时，Tiptap 的操作会转成对 Yjs 共享状态的修改，Yjs 生成增量更新；

3. 客户端通过 WebSocket 发给服务端；服务端广播给同一文档房间里的其他客户端；其他客户端收到更新后，合并到自己的本地文档状态中。

4. 因为 Yjs 基于 CRDT：所以即使 update 乱序到达，或者用户离线后重新上线，或者多人同时修改，最终所有客户端都会收敛到同一个状态。

#### 2.1 首次进入文档时，怎么保证所有人看到的是同一个版本？
一般会由后端提供文档的最新持久化内容作为基准版本。
客户端进入房间后，先加载这个基准内容，再连接协同通道，后续增量更新都在这个版本基础上进行合并。这样可以保证新加入的用户不会拿到过时状态。

#### 2.2 文档持久化怎么做？
服务端按照定时保存或事件触发的方式（比如每10分钟、或者操作了50次），将最新全文持久化回数据库里中。
如果需要历史版本，可以额外维护版本表，保存快照，方便回滚。

#### 3. 为什么用 WebSocket，而不是 HTTP 轮询或者 SSE？ 
因为协同编辑需要高频、双向、低延迟通信。
HTTP 轮询实时性差，开销也大；
SSE 虽然适合服务端推送，但它本质是单向的，客户端还要单独发消息，不适合协同编辑这种双方都在持续改状态的场景。
WebSocket 建连后就是全双工长连接，更适合多人协作、光标同步、增量更新这些实时交互。

#### 4. WebSocket 里传的是什么内容？
WebSocket 里通常传的是协同相关的增量消息，比如：
- 房间管理：维护每个文档对应的协同房间
- 文档更新的增量数据：接收某个客户端的更新后广播给同房间其他用户
- 光标位置和选区状态
- 重连后的状态同步请求：处理用户加入、离开、重连、心跳等连接管理

#### 4.1 Yjs update 的格式是什么？
Yjs 会把操作序列化为二进制格式，通常会包含操作类型、修改范围、内容、来源标记 origin、时间戳等信息。还会包含唯一的操作 ID、依赖的前置操作 ID，以及客户端 ID 等，这样比 JSON 更节省带宽，适合高频传输。

Yjs 的二进制编码有校验机制，服务端收到消息后会先验证格式，若解析错误会丢弃该消息，客户端重连时会重新同步完整状态。

一般会检查二进制数据的完整性，比如是否符合 Yjs 定义的 update 结构，有没有缺失关键的操作 ID、依赖 ID 这些字段，客户端 ID 是否合法，时间戳是否在合理范围，避免恶意构造的无效数据干扰同步。

```js
{
  id: string
  type: 'insert' | 'delete' | 'replace' | 'format' | 'batch'
  from: number
  to: number
  content?: string | object
  attrs?: Record<string, any>
  origin: 'local' | 'remote' | 'system' | 'ai'
  timestamp: number
}
```

##### 4.2 为什么不用每次都存全量内容？
因为全量存储会带来：
多人高频编辑时会频繁写入大文件，MySQL 存储压力大，且同步时传输全量数据会占用更多带宽，导致延迟。

而 Yjs 的增量操作数据量小，服务端定时或按操作次数触发全量保存，既能减少存储和传输成本，又能通过增量日志保证离线重连时的状态合并，最终一致性也由 CRDT 算法保障。

所以更合理的方式是“实时协同 + 定期落库 / 版本快照”结合起来。

#### 4.3 在高并发场景下，Websocket 广播会不会产生延迟或丢包，如何处理？
Java 服务端用 Netty 框架处理 WebSocket。
对丢包，Netty 的 TCP 可靠传输会自动重传，此外还开启心跳机制检测连接状态。
进一步的，如果用户量增加，还可以设置服务端对每个文档房间维护独立的消息队列，确保广播顺序和完整性，避免乱序导致的合并问题。

#### 5. Yjs 底层为什么能解决冲突？如果多人同时改同一段内容，最后怎么决定谁生效？
冲突的解决依赖 CRDT 的设计。
Yjs 底层是基于双向链表的，每个操作都有唯一 ID 和依赖关系。不管操作是顺序还是离线产生的，他都能自动合并。

- 不同客户端的编辑操作都可以作为独立更新存在
- 更新之间可以按顺序到达，也可以乱序到达
- 即使离线期间产生了修改，重新联机后也能合并
- 最终所有客户端会收敛到同一个状态

比如两个人同时编辑一句话：
- A 把“生物实验数据分析”改成“生物实验结果分析”
- B 同时在后面补了“流程”
在 Yjs 里，这两个操作会作为独立更新被记录下来，最终会合并成一个一致结果。它的目标不是保留某一个人的完整版本，而是让多个并发操作能稳定收敛，尽量保留各自的有效编辑结果。

双向链表能同时记录每个操作的前驱和后继依赖，这样即使中间某个操作乱序到达，也能通过前后依赖关系找到正确的插入位置。比如 A 操作依赖 B，C 操作依赖 A，就算 C 先到，也能通过依赖关系等到 A 和 B 都同步后再合并，保证最终状态一致。

#### 6. CRDT 和 OT 有什么区别？
两者都是为了解决协同编辑冲突，但思路不同：

- OT：依赖服务器转换操作，逻辑复杂，需要把操作转换成当前状态可执行的形式，对网络延迟较敏感。
- CRDT：通过数据结构自动合并，不太依赖中心转换，天然支持离线而且不需要服务端密集计算。

##### 7. 离线编辑怎么做？断网期间写的内容会不会丢？重新上线后怎么恢复状态？
不会丢。

通过 WebSocket 连接状态变化和心跳机制判断。
当连接断开、超时未响应或服务端检测到心跳失败时，就可以把该用户标记为离线，并从在线用户列表和光标列表中移除。
短暂断开通常会加重连机制和短暂保留策略，比如：

- 短时间内断线不立即清除状态
- 客户端自动重连
- 重连后重新拉取房间状态并恢复同步

离线时用户的修改先写到本地 Yjs 文档里，通常存在浏览器的本地存储里，比如 IndexedDB，产生的更新也是独立的 CRDT 更新。
重新上线后，客户端会重新加入对应房间，通过y-websocket的同步机制，先向服务器请求缺失的操作日志，
再用 CRDT 算法自动合并本地离线操作与远端最新状态自动合并，确保冲突被正确处理后同步到客户端。

#### 8. 光标同步是怎么做的？ 光标同步为什么不直接放进文档状态里？
具体来说，光标同步其实分两步：
yjs 中 有专门的 protocols、awareness模块，每个客户端的光标位置、选择范围会作为 awareness 状态通过 Websocket 广播；
其他客户端收到后解析并渲染对应的光标位置和用户名。

一般会包含用户 ID、光标开始和结束的文本偏移量、选择方向，可能还有用户名、光标颜色这些元数据。比如用户选中了从第 10 个字符到第 20 个字符，那数据里就会有 start:10、end:20，其他客户端收到后就能在对应位置渲染出光标和选区。

它和文档内容同步是分开的，因为光标是“临时协作状态”，不是文档持久内容。
如果把它们直接写进文档，会污染持久化内容，增加同步复杂度，也会让数据体量变大。

#### 8.1 如果用户网络不稳定，光标位置频繁变化，WebSocket 会不会一直广播无效的光标消息？
Awareness 模块会对光标消息做节流处理，比如 300 毫秒内只发送一次最新位置，减少无效广播。

#### 8.2 那如果某个用户的光标长时间不动，服务端会怎么处理？
Awareness 模块会设置超时时间，比如 5 分钟内光标无变化，服务端会标记该用户为 “非活跃”，不再广播其光标状态，直到用户再次移动光标才恢复广播，这样能进一步减少冗余数据。

#### 9. 历史栈记录的是什么？如何维护？
项目里用的是 Tiptap，它底层基于 ProseMirror，自带 history 机制，主要管理本地编辑的 undo / redo，放在前端内存里，页面刷新后历史栈会丢失。

undoStack：记录已执行但可撤销的操作，redoStack：记录被撤销后可重做的操作。

但在协同编辑场景下，撤销会更麻烦，因为远端用户的修改会导致光标位置和操作范围漂移，不能简单按本地历史直接回放，否则可能误删别人的内容，所以需要结合 Yjs 的协同状态来处理撤销范围和一致性。

#### 9.1 协同编辑下本地 undo 怎么避免影响别人？如果要撤销的内容被别人改了怎么办？
核心原则：本地 undo 只撤销“当前用户自己产生的操作”，不直接撤销远端同步内容。

实现上会把本地操作和远端操作分链路处理：
- 本地主动编辑：进入本地 undo/redo 历史（Tiptap / ProseMirror history）
- 远端同步更新：只合并到文档状态，不进入本地可撤销历史

区分方式通常依赖 transaction 的 origin / meta 标记，保证撤销范围始终限定在“我自己的历史链”。

如果本地要撤销的内容已经被远端协作者改动，不能按原位置硬回退，需要先做位置映射和状态校验；
- 能安全撤销：执行最小范围撤销
- 不能安全撤销：提示“该内容已被其他协作者修改，无法直接撤销”，或让用户二次确认

如果用户确认撤销 Yjs 会把本地撤销操作也编码成一个增量 update，和远端操作一起按 CRDT 规则合并，确保撤销后所有用户的文档状态依然一致，同时本地历史栈会更新为撤销后的状态。

#### 10. 多人同时编辑时，怎么保证性能？
多人编辑时，性能优化可以从操作合并、数据传输、文档拆分、渲染优化这几个方面入手。
比如用微批处理把短时间内的操作合并成一个事务，减少 CRDT 合并和重渲染的次数；只传输变化的增量数据，而不是整个文档；
其次是把光标状态和正文状态分离，避免无效更新；
还可以对文档进行分片处理，比如按章节或段落拆分，多人编辑不同分片时只同步对应分片的增量，进一步降低数据传输量。
还有就是避免频繁更新 DOM，用虚拟滚动处理长文档；
同时可以做节流、批量发送和重连补偿，减少 WebSocket 压力。

如果有多个服务端，Yjs 可以通过 “去中心化” 或 “中心化集群” 方案处理。去中心化的话，服务端之间用 Gossip 协议同步操作日志，每个服务端存完整数据；中心化集群的话，选一个主服务端处理合并，其他服务端同步主节点状态。不过多服务端会增加网络开销，需要根据实际并发量和延迟要求选方案。


#### 11. 批量操作、增量命令和 Yjs update 的区别
- 多个命令可以合并进同一个 transaction，一次性提交，它们会成为一条历史记录，undo 时可整体回退（粘贴大段文本、整块格式化）。
- transaction 提交后，会被编码成 Yjs 的增量 update，在协同端之间同步。

#### 11.1 如果批量操作中途失败怎么回滚
我会优先做预检和事务式提交，尽量避免半执行状态；
如果已经部分生效，就通过执行前快照或反向补偿把文档恢复回去。

#### 11.2 事务的大小有没有限制？
通常可以根据实际场景设置一个合理的事务大小限制，比如单次批量操作不超过 1000 个字符或 100 行内容，避免单个事务过大导致同步时阻塞或性能下降。如果超过限制，可以自动拆分成多个小事务处理。

#### 13. 如果文档中包含图片、表格等复杂内容，协同编辑时这些内容的同步和冲突处理和纯文本有区别吗？
图片、表格这类复杂内容通常会被拆分为多个基础操作，比如图片的插入位置、src；表格的行列增减、单元格等，Yjs 会为这些子操作分别生成唯一 ID，冲突时还是按 CRDT 的规则合并，但需要业务层对复杂内容的展示做适配。

比如表格合并单元格，Yjs 会记录合并的范围和顺序，若同时有其他用户拆分单元格，CRDT 会根据操作 ID 确定最终状态，业务层再根据合并后的操作数据渲染表格。

#### 14. 删除为什么不是直接删？
yjs 删除本质是 tombstone（墓碑标记），不会立即物理删除。
原因是别的客户端可能还没同步到，仍引用该节点。
等所有依赖安全后，GC 才会真正清理。

通常是当所有在线客户端都确认收到并应用了这个删除操作，或者经过一段时间后没有客户端再引用这个节点时，Yjs 会自动清理墓碑标记，释放内存。如果之后想优化内存占用，也可以手动配置 Yjs 的清理策略，比如设置一个固定的清理时间间隔。

### B. 大文件上传链路
> 在文件能力上，设计并落地了“切片上传 + 断点续传 + 秒传 + 任务恢复”的完整链路：
> 前端按分片并发上传，服务端记录分片状态并支持按任务 ID 恢复上下文；
> 上传中断后仅补传缺失分片，避免全量重传；
> 通过文件指纹与分片记录校验实现秒传，显著降低重复上传带来的网络和存储成本。

#### 1. 大文件上传链路是怎么设计的？
把它设计成“切片上传 + 断点续传 + 秒传 + 任务恢复”四层能力。前端先按固定大小切片，对每个分片做唯一标识，再并行上传；服务端保存每个分片的接收状态。这样用户断网或者刷新页面后，只需要补传缺失分片，不需要整个文件重传，尤其适合科研场景里几十 MB 甚至更大的资料、图片和附件。

#### 2. 怎么实现秒传？
核心是先算文件指纹，比如文件名、大小、hash 或者分片 hash 的组合。上传前先让前端和服务端做一次校验，如果服务端已经存在同样的文件标识和完整分片记录，就直接返回“已存在”，前端就可以跳过真正的上传流程，这就是秒传。它能显著减少重复文件的网络和存储成本。

#### 3. 断点续传具体怎么做？
每个分片上传成功后，服务端都会记录状态。用户重新打开页面或者上传中断后，前端先请求服务端拿到已上传分片列表，再只补传缺失的部分。上传完成后，服务端再做分片合并、完整性校验，最后把文件元信息写入数据库。这样断点续传不是“从头开始重试”，而是“只补缺口”。

#### 4. 如果上传中途失败，怎么恢复任务？
会把上传任务本身当成一个可恢复状态机来设计，至少包含待上传、上传中、部分完成、已完成、失败可重试这几种状态。前端刷新后可以重新拉取任务状态，服务端根据任务 ID 和分片状态恢复上下文，避免用户重新选择文件。这样用户体验会比“失败后全量重传”稳定很多。


### C. SSE 流式问答
> 前端采用 fetch + ReadableStream 做增量解析，解决了半包粘包、逐 token 渲染性能和请求级中断问题；后端配合连接取消和任务清理实现端到端资源回收；在多轮对话里通过滑动窗口 + Rolling Summary 控制上下文长度和成本。

#### 1. 为什么在问答里选 SSE？SSE 有什么缺点？
因为这个场景本质上是“服务端持续生成，前端持续消费”，服务端单向推流就够了。SSE 基于 HTTP，接入和部署都相对简单，而且浏览器原生支持断线重连，特别适合大模型逐 token 输出的问答场景。相比 WebSocket，它实现成本更低，也更贴合“只需要服务端往前端推文本”的需求。

SSE 的缺点
- 只能服务端单向通信
- 浏览器连接数限制
- 长连接稳定性问题
- 不适合高频双向实时通信

#### 2. SSE 为什么不用 EventSource，而用 fetch 自己读流？
选 fetch + ReadableStream 主要是为了可控性和一致性：
- EventSource 对请求能力受限，自定义 Header、POST body 不灵活，而问答接口需要携带复杂参数（多轮上下文、检索选项、模型参数）。
- 做请求级中断（AbortController）、会话级并发治理，这在 fetch 模式下更统一。
- 前后端已有统一鉴权链路（JWT/Header），fetch 可以复用现有拦截器和错误处理体系。
- 需要对流式分片做精细解析（半包/粘包、增量渲染、统计 token），ReadableStream 更适合做底层控制。

#### 3. SSE 里的粘包、半包你们怎么处理？SSE 数据不是一条条完整到达的，你怎么保证解析正确？
流式响应过来的内容可能一个 chunk 里包含多条事件，也可能一条事件被拆成多个 chunk，所以前端需要维护一个增量缓冲区，持续累积数据
- 每次 reader.read() 拿到 Uint8Array 后，用 TextDecoder(..., { stream: true }) 解码，避免 UTF-8 多字节字符被截断导致乱码。
- 把新文本 append 到 buffer。
- 以 \n\n 作为 SSE 事件分隔符循环切片；不完整尾巴保留到下一轮。
- 每个事件按行解析 data: 字段并拼接，多行 data: 合并后再 JSON.parse。
- 对 [DONE] 或自定义 event: end 做终止收尾。

#### 4. fetch + ReadableStream 逐 token 渲染，具体怎么做？token 一个个来会导致重渲染风暴？
前端不是一次性等接口返回完整文本，而是用 `fetch` 拿到流式响应，再从 `ReadableStream` 里不断读取 chunk。

我做了两层节流：
- 数据层聚合：每个 chunk 到达后先做解析，不是每个 token 都立刻 setState，而是先 push 到内存队列，用来解决 SSE 的半包/粘包（协议层问题）；
- 渲染层合批（分帧消费）：通过 requestAnimationFrame 或 20~50ms 定时窗口批量 flush 到 UI，过程中加入 UI 的 buffer，用来减少 setState + Markdown parse 频率，避免 token 全量 parse 触发高频重排和重绘，

#### 4.1 Markdown 是不完整语法，半截代码块怎么办？
流式中经常出现未闭合结构（如 ````` 尚未闭合、表格只到一半）。
常用的刷新时机：
- 增量刷新时，UI 的 buffer 累积到阈值/到达时间窗口，合并文本，执行一次 Markdown 解析，更新视图。

- 结构闭合时，如果检测到代码块/表格/公式刚闭合，可立即触发一次解析，让展示更稳定。

- 结束收尾时，收到 [DONE] 或流关闭后，做最终一次完整解析与规范化渲染。

#### 4.2 长文本性能优化（解决“单次渲染成本”）
- 增量拼接 + 合批解析：不逐 token parse，按批次解析。
- 分块渲染：把回答拆成稳定块（段落/代码块），只重算“尾块”，前面块冻结。
- 迟重处理：代码高亮、数学公式在块闭合后再做；流结束后做终态精修。
- 自动滚动治理：仅当用户接近底部才跟随滚动，避免每次更新触发布局抖动。
- Worker 下沉（可选）：Markdown parse/高亮放 Worker，主线程只做最小提交。

#### 4.3 XSS 怎么防？
LLM 输出本质是不可信输入。
1. 不直接 v-html/innerHTML 生插，流程应是：Markdown 文本 -> Markdown 解析器 -> HTML -> sanitize 白名单清洗 -> 渲染。
2. 白名单而不是黑名单，只允许安全标签和属性，例如：p, ul, ol, li, strong, em, code, pre, a, h1-h6...，移除全部事件属性、禁止危险标签：script, iframe, object, embed...
3. 如果支持代码高亮，也要纳入清洗链路，最终注入前统一再 sanitize 一次。
4. 链接加固，外链统一加：rel="noopener noreferrer nofollow" target="_blank" 只在可信策略下开启，拒绝 javascript:、可疑 data: URL
5. 配合 CSP 兜底：配置 Content-Security-Policy 内容安全策略，限制脚本来源，尽量禁用内联脚本。

#### 5. 流式输出时怎么支持中断？只 cancel 前端就够了吗？
前端会把当前请求和 `AbortController` 绑定起来。用户点击停止时，直接 abort 当前流式请求，停止读流并更新消息状态为“已中断”。

后端还需要：监听连接关闭，cancel 上游模型请求，清理任务上下文，回收 token stream。这样既能避免继续消耗算力，也能让用户在结果不满意时及时打断，重新发起新的请求。

#### 6. 流式过程中如果 JSON 解析失败或网络抖动怎么办？
把异常分三类处理：

- 可恢复（短时抖动、局部分片异常）：保留已生成内容，提示“连接波动，可重试续答”。
- 不可恢复（鉴权失败、模型报错）：立即终止并展示明确错误码/原因。
- 用户中断：区分为正常业务状态，不记为系统错误。

同时做了 observability：记录首 token 延迟、平均吞吐、流中断率、用户中断率，便于定位瓶颈在网络、检索还是模型。

#### 7. 重连重试怎么做？
在流式请求外面包一层重试控制，断线时先做有限次快速重试，重试失败后再进入退避等待。重新连接后，前端会把上一次已经消费到的位置，比如SSE 事件 id、会话任务的标识带给后端，后端能找到这次生成上下文或缓存，跳过已发部分，尽量从断点继续，而不是完全重新开始。

#### 8. 如何避免“光标抖动”和滚动体验差？
只有用户在底部附近时才自动跟随滚动
用户手动上滑后暂停自动滚动
新内容用锚点恢复，而不是每次 scrollToBottom 暴力拉到底
这样用户看历史内容时不会被“拽回底部”。

### D. L1 / L2/ L3 三层记忆
> 我设计的是 L1/L2/L3 三层记忆架构，分别对应最近原文、阶段摘要和长期语义记忆。
> L1 保证当前推理的细节不丢，L2 通过 Rolling Summary 压缩长对话并保持连续性，L3 负责存储长期稳定的用户偏好和事实。
> 写入上采用分层过滤和置信度控制，避免错误记忆污染；读取上采用 L1 原文、L2 摘要、L3 检索结果的组合。

#### 1.为什么要做 L1 / L2 / L3 三层记忆？一层不行吗？
如果只用一层记忆，把所有历史都塞进 prompt，要么保留太多原文导致 token 成本很高，要么压缩太狠导致信息丢失。
把记忆拆成三层，是为了分别解决“最近上下文细节”“阶段性语义压缩”“长期稳定偏好”这三个问题。
L1 保证当前推理不丢细节，L2 保证长对话不中断，L3 保证跨会话还能记住用户偏好和长期事实。
本质上是用分层来平衡准确性、成本和个性化。

#### 2. L1 / L2 L3 三层记忆分别是什么？
L1 Recent Window，保存最近几轮高相关上下文，比如当前任务、最近一次修改意见；
L2 Rolling Summary：存滚动摘要，主要是目标、约束、已确认决策、未决问题；
L3 Long-term Memory 长期语义记忆：面向跨轮/跨会话的稳定信息，只写长期稳定信息，保存用户偏好、项目背景、关键结论、历史协作习惯等。

#### 3. L1 的“滑动窗口”怎么设计？窗口多大？
设计成基于 token 的滑动窗口，而不是单纯按轮次截断。
它保留最近的原始对话、当前任务关键决策和未解决问题。窗口大小通常控制在 1k 到 4k 左右，具体取决于模型上下文长度和任务复杂度。
会优先保留并同步写入 L2，再从 L1 滑出，这样既控制成本，又不丢关键信息。

#### 3.1 L1 滑出后怎么决定写进 L2 还是 L3？
L1 滑出后，会先判断它的语义类型。
如果是当前任务、阶段性上下文、未决事项，就写进 L2，因为它只对当前会话或当前任务有用。
如果是长期稳定的用户偏好、身份背景、明确确认过的事实，并且置信度足够高，就写进 L3。

#### 4. 为什么 L2 要用 Rolling Summary，而不是固定摘要？Rolling Summary 是怎么做的？
固定摘要通常是把某一段对话一次性总结出来，之后就不再更新，容易丢掉最新决策。
Rolling Summary 是增量更新的，每次只吸收新增信息，并保留当前阶段最关键的状态。
会把早期对话压缩成“任务 - 动作 - 结果”这样的摘要结构，保留最有价值的信息，而不是机械截断文本。每当新对话进来时：旧摘要 + 新增信息 -> 新摘要

#### 4.1 Rolling Summary 会不会越滚越偏？怎么防止摘要漂移？
Rolling Summary 确实存在漂移风险，因为它本质上是持续压缩，误差会逐轮累积。
防护思路是：
采用结构化摘要，固定保留目标、约束、决策和未决项，而不是一段自由文本；
每次更新时，输入上一版摘要和新增原文，输出新摘要，同时保留变更记录和来源引用；
加入一致性校验，对数字、时间、否定句和关键决策做保护，必要时回退到原文。


#### 5. L3 长期记忆怎么写入？是不是每次都写？
L3 只写高价值、长期稳定、可复用的信息，比如偏好、长期目标、明确纠正后的事实。

不是每次都写。先做抽取和过滤，判断这条信息是不是值得长期保存，再写入并附带来源、时间戳（TTL）、置信度、去重（语义相似度）。记录结构：fact, scope, confidence, source_turn, updated_at, ttl，可以使用结构化 KV。

#### 5.1 怎么检索 L3？
L3 的检索通常是先做混合召回，再做重排序。
先通过语义检索召回相关记忆，再用关键词或规则补充精确匹配项，然后结合时间衰减、置信度和冲突状态做重排。最后只把少量高相关、高可信的记忆拼进当前 prompt，避免噪声干扰。

这样做的原因是 L3 存的是长期偏好、稳定事实和用户画像，这类信息既有语义相关性，也有精确匹配需求，单一检索方式容易漏召回或错召回。

#### 5.2 L3 会不会“记错”用户？
不能每句话都写 memory。
需要：memory gating，例如：用户明确确认、多次重复出现、高置信度、长期稳定才写入。

#### 5.3 L3 如何避免无限增长？
TTL：30 天未命中自动过期
热度衰减：长期不用自动降权。
consolidation：合并重复 memory。

#### 6. 检索时 L1 、L2 、L3 怎么融合？三层之间的数据流怎么走？生成回复时，三层记忆是怎么参与的？
流程是：
- 新消息先进入 L1
- L1 超阈值（token/轮次）触发摘要，通过 Rolling Summary 进入 L2
- 从 L2/L1抽取“稳定且可复用事实”写入 L3（带置信度、时间戳、来源）
- 每轮生成前做组装：L1、L2、L3 不是简单拼接，而是按问题类型动态加权：L1 负责最近原文和细节，L2 负责当前阶段的滚动状态，L3 负责长期偏好和稳定事实。

#### 7. L1、L2、L3 分别存放在哪里？
L1 读写频率高，要求延迟低，Redis 很适合做短期记忆缓存；
L2 也放在会话级持久化存储中，Redis 用来保存 Rolling Summary 和当前任务状态；
L3 是长期稳定信息，通常需要按用户维度查询、更新和审计，放在 PostgreSQL 这类关系型数据库更适合管理结构化摘要。

#### 8. 你怎么评估这套记忆机制有效？
我会从四类指标看效果。
第一是连续性，比如长对话里上下文丢失率、指代消解正确率。
第二是个性化，比如用户偏好命中率、重复询问率下降。
第三是成本，比如平均 token、P95 延迟、存储成本。
第四是质量，比如人工盲评的一致性和幻觉率。
这样能比较全面地衡量系统收益。

#### 9. 如果用户前后说法冲突了，怎么处理？L2 和 L3 冲突时怎么办？
新的信息如果和旧记忆冲突，会判断它是明确修正、作用域变化，还是信息不确定。
如果是明确修正，就以最新表述为准，同时把旧记忆降权或标记过期；
如果只是不同场景下的说法，就通过作用域区分，避免误判冲突；
如果仍然不确定，就保留多个版本，并用置信度和时间戳决定生成时优先使用哪条。

给 L2 和 L3 加一个冲突仲裁层，
L2 代表当前会话状态，通常优先级更高；L3 代表长期记忆，只有在用户没有明确新指令时才作为默认偏好。
如果冲突被判定为长期变化，我会更新 L3；如果只是临时覆盖，则保留 L3 但在本轮生成时忽略它。

### E. RAG 检索增强问答
> 我们做了一个 RAG 知识增强链路，先对文档做解析和切分，再进行向量化入库；
> 检索阶段采用 BM25 + 向量检索的混合召回，兼顾关键词精确匹配和语义召回，
> 然后通过重排挑出最相关的证据片段，最后让大模型基于证据生成答案，并把来源回填给用户。

#### 1. 为什么要做 RAG，而不是直接微调模型？
因为科研场景对准确性要求高，通用大模型很容易出现专业幻觉。RAG 的核心价值是把外部知识检索进来，再基于检索内容生成答案，这样模型不完全靠参数记忆，回答会更贴近项目里的真实文档和知识库，可信度也更高。
而微调适合：固定任务格式、风格统一、领域能力强化的场景；更新成本高，知识变更后要重新训练。

#### 2. 你们的检索链路是怎么做的？
我们不是简单拿 TopK 片段直接拼上下文，而是先做文档解析和切片，再做向量化和索引；检索时采用 BM25 关键词稀疏检索 + 向量相似度稠密检索的混合召回，再做重排和上下文压缩，最后把高相关内容注入回答流程。这样能兼顾关键词精确匹配和语义召回。

#### 3. 分片策略是什么样的？
对于普通文本，采用递归语义切片，先按段落分割，若段落长度超过512 token，再按句子递归拆分，确保每个分片在256-512 token之间，且不拆分完整句子。
对于Markdown格式文本，优先按标题层级分片，同一标题下的内容再按代码块、列表块分割，保证每个分片保留标题上下文和完整语义块，这样向量化时能更准确捕捉内容主题，提升召回相关性。

chunk 太大会导致： embedding 语义混杂，recall 不精准，rerank 成本高；
chunk 太小会导致： 上下文断裂、丢语义、retrieval fragmented

#### 3.1 那针对格式化文本，除了按标题和块分片，有没有处理过特殊格式，比如表格、公式或者代码块？
假设处理表格时，会将表头和对应行数据合并成“表头：值”的文本片段，公式保留LaTeX原格式，代码块按函数或逻辑块分割。

处理代码块，按语法树解析，比如借助tree-sitter工具，优先按函数、类这些语法单元切分，每个分片控制在1000到1500字符，同时保留函数签名和注释。

#### 4. 为什么要混合召回，而不是只用向量检索？索引是如何做的？
因为科研文本里既有语义相似，也有非常关键的术语和专有名词。
只用向量检索，有时候会把语义接近但术语不对的片段召回上来，对精确术语、数字、专有名词不够稳定；
只用BM25关键词检索，优点是对关键词、术语特别准，但用户换个说法可能检索不到。

混合召回可以更稳地平衡两者，提高召回质量，融合时采用RRF方法：对每个分片，分别取BM25排名和向量排名，按1/(排名+60)计算分数后相加，最后按总分取Top结果，能稳定平衡关键词匹配和语义相关性。

#### 4.1 向量数据库用的什么？
使用 Redis Stack，自带了 RedisSearch 和 RedisJSON 这些模块，能直接存向量，还支持 HNSW 索引和余弦相似度这些常用的向量检索功能，部署和集成也方便，很适合 RAG 这种需要实时检索的场景。

#### 5. 混合召回后为什么还要重排？重排策略
因为召回阶段追求的是“多而全”，但生成阶段需要的是“少而准”。
混合召回会拿到一批候选片段，但里面不可避免会混入：相关性一般或者重复内容，所以需要重排模型做二次筛选。

重排阶段可以加入交叉编码器，对召回的Top10分片和用户问题做二次语义匹配，进一步提升相关性。
比如先用RRF拿到10个分片，再用交叉编码器打分排序取Top5，这样兼顾效率和准确性。

#### 5.1 混合召回数量和最终 topk 如何取值
混合召回阶段会把 topK 设得相对大一些，通常 BM25 和向量检索各取 20 到 50 条，合并去重后再做重排；
最终送给大模型的 topK 会收敛到 3 到 8 条，常用 5 条左右。

#### 6. 如果文档更新频繁，你们是怎么保证Redis里的向量索引和最新文档同步的？是全量重建还是增量更新？
可以设计增量更新策略，当文档有新增或修改时，只对变化的分片重新生成向量，然后调用Redis的FT.ADD或FT.UPDATE命令更新索引，删除旧分片的向量。
如果是大量文档更新，可先暂停检索，全量重建索引后再切换，这样能减少对服务的影响。

#### 7. 那在检索时，如果用户查询的是长文本问题，比如多轮对话后的复杂问题，向量索引还能准确召回相关分片吗？
可以先对长查询进行语义压缩，提取核心问题生成向量，再去检索。或者用多向量检索，把长查询拆成多个子问题向量，分别召回后合并结果，这样能提升长问题的召回准确性。

#### 8. 在RAG系统里，你怎么评估检索到的文档和用户问题的相关性？有没有什么具体的指标或者方法？
召回率看是否能把所有相关分片都找出来，准确率看召回的分片中真正相关的比例。
除了这两个，还可以用NDCG评估排序质量，或者人工标注Top5分片的相关性打分。

#### 9. 怎么处理“检索不到答案”的情况？
如果召回证据不足，模型必须明确说明当前知识库中未找到足够依据
可以建议用户补充文档、 уточ specific question
或者返回“根据现有资料无法确认”的保守回答

#### 9.1 多文档冲突怎么办？
来源打分、时间、权威性、文档类型；或者告诉模型存在冲突

#### 10. 来源回填是如何做的？
来源回填就是在RAG生成回答后，把引用的文档分片信息附在回答末尾。具体做法是检索时记录每个相关分片的元数据，生成回答时用这些元数据替换掉回答里的占位符，让用户知道信息来源。

因为用户不只想要答案，还想知道答案来自哪份资料、哪段原文。来源回填能提升可解释性，也方便科研团队回查原始依据。

### F. 文档内 AI 辅助创作
## 逐字解说
这个部分做了一套文档内的 AI 多智能体协作系统，主要是想解决传统单 Prompt 模式在写作场景下的问题。

整体上设计的是一个Fast / Swarm 双通道架构。

1. Fast 模式主要处理低复杂度、低时延任务，比如翻译、简单润色等。这类请求会直接经过 Router Agent 路由对应到具体的 Worker Agent，然后调用 LLM 生成结果。
2. Swarm 模式主要处理复杂任务，比如长文档总结、Mermaid 生成等。复杂任务会进入多智能体协作链路。

它会先对请求做复杂度评估，比如：输入文本长度、是否需要知识库索引、是否涉及多步骤生成，然后决定走 Fast 还是 Swarm。

如果进入 Swarm 模式，会先由 Planner Agent 做任务拆解。

比如用户说：

‘请根据当前文档生成技术方案，并补 Mermaid 架构图。’

Planner 会把任务拆成几个步骤：

1. 先总结文档核心模块
2. 提取系统结构
3. 生成 Mermaid 草稿
4. 最后统一润色输出

之后任务会进入多个 Worker Agent 并行执行。

比如：

Summary Agent 负责总结
Retrieval Agent 负责 RAG 检索
Mermaid Agent 负责结构图生成
Writing Agent 负责最终文本生成

中间结果不会直接拼 Prompt，而是写入 Shared Workspace，也就是共享工作区，所有 Agent 都可以读取前序结果，而不需要重新理解全文。

生成完成后，还会经过 Critic Agent。

Critic 主要负责：检查结果是否偏题、引用合法性、Markdown 格式等问题。

如果发现问题，会进入 Reflection Loop，再次修正生成结果。

最后由 Merger Agent 做统一结果聚合，并转换成结构化协议返回前端。

因为我们这个场景不是普通聊天，而是编辑器内 AI 操作，所以前端不仅需要拿到文本，还需要知道：

应该如何回填
替换哪里
插入哪里
是否生成 Mermaid 块

基于此，我们设计了一套统一响应协议。

```
                    User Request
                           │
                   Lead Orchestrator
                           │
               Complexity Scorer Agent
                 ┌─────────┴─────────┐
                 │                   │
              FAST                SWARM
                 │                   │
          Router Agent         Planner Agent
                 │                   │
          Single Worker        Task DAG Planning
                 │                   │
                LLM        ┌─────────┼─────────┐
                           │         │         │
                    Retrieval   Writing   Mermaid
                      Agent      Agent      Agent
                           │         │         │
                           └──── Shared Workspace ────┐
                                                      │
                                                Critic Agent
                                                      │
                                                Reflection Loop
                                                      │
                                                Merger Agent
                                                      │
                                                 Final Output
```


#### 1. 你们为什么要做文档内 AI 辅助创作，而不是单独聊天？
因为用户的真实需求不是“跟模型聊天”，而是在文档编辑场景里直接完成续写、润色、翻译、总结、生成图表这些动作。把 AI 能力嵌进文档里，可以让用户在同一个工作流中完成检索、编辑和改写，减少来回切换工具的成本。

#### 2. 你们的多智能体是怎么分工的？
我们设计了 Orchestrator / Planner / Worker / Critic 这样的链路。Orchestrator 负责总控，Planner 负责拆任务，Worker 负责执行具体步骤，Critic 负责检查结果和反思修正。这样做的好处是把复杂任务拆开，既提升并行效率，也提升输出质量。

#### 3. 为什么要用 ReAct 和 Reflection Loop？
我会回答：ReAct 的好处是让模型按“思考 - 行动 - 观察”的方式推进，而不是一口气瞎生成。Reflection Loop 则是在结果不达标时触发再思考和修正，避免模型陷入无限推理或者带着错误继续输出。对科研写作这种高精度任务来说，这个约束非常重要。

#### 4. Mermaid 图是怎么生成的？
先让 Summary Agent 提炼核心模块，再让 Mermaid Agent 把模块关系映射成图语法，像“模块 A 调用模块 B”就会变成 `A --> B`。如果结构复杂，就拆成子图，尽量保证前端渲染清晰，避免图太大看不懂。

#### 5. 前端怎么知道该替换哪里、插入哪里？
后端不会只返回一段纯文本，而是返回结构化协议，里面包含操作类型、起始位置、结束位置、内容和是否生成 Mermaid 块等信息。前端根据这些元数据去执行替换、插入或高亮，这样 AI 结果才能真正回填到编辑器里，而不是只停留在聊天框。


### 1. 这个框架相比单 Agent 方案，任务处理效率提升了多少？
多 Agent 并行写作能使复杂任务平均处理时间缩短，同时 评审的 Reflection Loop能提高输出正确率，降低 Markdown 格式错误等。

### 2. 为什么选择Planner-Worker-Critic三层架构？
因为科研文档创作任务复杂，Planner负责拆解任务保证逻辑清晰，Worker并行执行提升效率，Critic通过反思循环保证结果质量，三层分工比单一Agent更适配“复杂任务+高精度”的科研场景。

### 3. Blackboard Pattern在高并发下如何保证数据一致性？
我们给共享工作区的每个数据块加了版本号，Agent写入时需校验版本，避免覆盖冲突；同时用读写锁控制访问，Worker读时加共享锁，写时加排他锁，高并发下任务错误率控制在5%以内。

### 4. Critic的Reflection Loop触发阈值是什么？
比如Markdown语法错误≥2处、引用格式错误率＞10%、内容与文档核心主题匹配度＜80%时，会触发反思循环，针对性回溯对应Worker Agent修正。
还能加个收敛检测，要是连续两次反思改进幅度小于5%，也直接停，避免无效循环。

### 5. ReAct范式
简单说就是让模型“思考-行动-观察”循环，先分析任务该做什么，再调用工具执行，然后根据结果调整下一步，像人解决问题一样有逻辑地一步步推进，最后得出答案。

### 6. Memorize Agent如何抽象复杂系统结构生成Mermaid？
先通过Summary Agent的文档总结提取核心模块，再用规则库映射模块关系到Mermaid语法，比如“模块A调用模块B”转化为“ A --> B”，复杂结构拆分为子图表，确保前端渲染清晰。

### 7. 响应式协议如何定义回填位置？
协议包含“操作类型+起始位置+结束位置+内容”，比如“replace:100:200:生成文本”，前端根据起始/结束索引定位文档位置，替换或插入内容，适配编辑器实时更新。

### 8. Worker Agent结果冲突时如何仲裁？
冲突时由Planner Agent根据任务优先级仲裁，比如“生成结构图”任务中，Memorize Agent结果优先级高于Summary Agent，以Memorize结果为准写入共享工作区。

### 9. 基于 ReAct 范式，设置 Reflection Loop 阈值
可以结合任务类型动态设阈值，比如简单的文案生成设3次，复杂的技术文档设5次，超过就终止
