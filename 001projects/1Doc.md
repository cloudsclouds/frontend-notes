# BioNova AI
## 项目总述
### 1. 这个项目介绍一下，有哪些核心亮点，负责哪些内容？
这个项目是一个面向生物科研团队的协同文档与智能问答平台，主要解决两个问题：一是科研协作里多人同时改文档、版本难统一；二是专业知识密集、模型容易幻觉，写作和问答效率都不高。

我在项目里主要负责前端核心能力和协同/AI链路设计，覆盖了实时协同编辑、大文件上传、流式问答、长上下文记忆管理、RAG 检索增强问答，以及文档内 AI 辅助创作的任务编排，帮助科研团队更快完成文献检索、知识整理、问答和内容生成。

## 简历串讲

### A. 实时协同编辑：Yjs + WebSocket
#### 1. yjs 是什么，为什么要使用 yjs？是怎么实现“多人同时编辑”的流程？以及持久化问题。
因为协同编辑的核心是“并发修改怎么稳定收敛”。Yjs 是成熟的 CRDT 协同库，它已经把冲突合并、离线编辑、增量更新、最终一致性这些最难的问题做好了。它定义了文档状态如何表示、如何合并、如何生成增量更新，是协同编辑里的核心一致性层。

具体流程是：
每个客户端打开文档后，会先从 MySQL 拉取文档全量内容并初始化一个 Yjs 文档实例，Tiptap 绑定这个 Y.Doc；
文档内容和编辑操作会被映射到 Yjs 的共享数据结构里；
用户本地编辑时，Tiptap 的操作会转成对 Yjs 共享状态的修改，Yjs 生成增量更新，通过 WebSocket 发给服务端；
服务端广播给同一文档房间里的其他客户端；
其他客户端收到更新后，合并到自己的本地文档状态中。
服务端再按照定时保存或事件触发的方式，将最新全文持久化回 MySQL 中，保证文档可恢复和可追溯。
如果需要历史版本，可以额外维护版本表，保存快照，方便回滚和审计

#### 1.1 为什么不用每次都存全量内容？

因为全量存储会带来：

- 写入频繁、IO 开销大
- 大文档性能差
- 历史版本管理不灵活

所以更合理的方式是“实时协同 + 定期落库 / 版本快照”结合起来。

### 1.2 首次进入文档时，怎么保证所有人看到的是同一个版本？
一般会由后端提供文档的最新持久化内容作为基准版本。
客户端进入房间后，先加载这个基准内容，再连接协同通道，后续增量更新都在这个版本基础上进行合并。这样可以保证新加入的用户不会拿到过时状态。

### 1.3 如果有人正在编辑，基准版本和实时增量怎么对齐？
把“基准版本”理解成进入房间时拿到的快照，把“实时增量”理解成后续不断到达的 Yjs update。客户端先渲染基准版本，再把连接建立之后收到的增量按顺序或按 CRDT 规则合并进去，这样就能追上最新状态。

如果在拉取基准版本和真正加入协同通道之间还有一段时间差，服务端一般会保留这段期间的增量队列，或者在客户端进入后补一次状态同步，确保快照和增量不会断层。这样新用户看到的就是“基准版本 + 最新增量”的结果，而不是一个静态旧副本。


#### 2. 为什么用 WebSocket，而不是 HTTP 轮询或者 SSE？ WebSocket 里传的是什么内容？
因为协同编辑是高频、双向、低延迟通信。
HTTP 轮询实时性差，开销也大；
SSE 虽然适合服务端推送，但它本质是单向的，客户端还要单独发消息，不适合协同编辑这种双方都在持续改状态的场景。
WebSocket 建连后就是全双工长连接，更适合多人协作、光标同步、增量更新这些实时交互。

WebSocket 里通常传的是协同相关的增量消息，比如：
文档更新的增量数据：接收某个客户端的更新后广播给同房间其他用户
房间管理：维护每个文档对应的协同房间
光标位置和选区状态
重连后的状态同步请求：处理用户加入、离开、重连、心跳等连接管理


#### 3. Yjs 底层为什么能解决冲突？如果多人同时改同一段内容，最后怎么决定谁生效？

冲突的解决依赖 CRDT 的设计，它允许并发修改在数学上可合并。

- 不同客户端的编辑操作都可以作为独立更新存在
- 更新之间可以按顺序到达，也可以乱序到达
- 即使离线期间产生了修改，重新联机后也能合并
- 最终所有客户端会收敛到同一个状态

比如两个人同时编辑一句话：
- A 把“生物实验数据分析”改成“生物实验结果分析”
- B 同时在后面补了“流程”
在 Yjs 里，这两个操作会作为独立更新被记录下来，最终会合并成一个一致结果。它的目标不是保留某一个人的完整版本，而是让多个并发操作能稳定收敛，尽量保留各自的有效编辑结果。

#### 3.1 那它和 OT 有什么区别？
两者都是为了解决协同编辑冲突，但思路不同：

- OT：需要把操作转换成当前状态可执行的形式，通常对操作顺序和中心协调依赖更强
- CRDT：通过数据结构和规则保证各端最终一致，不太依赖中心转换


#### 4. 离线编辑怎么做？断网期间写的内容会不会丢？
不会丢。
离线时用户的修改先写到本地 Yjs 文档里，产生的更新也是独立的 CRDT 更新，不依赖实时连接。等网络恢复后，会和远端最新状态自动合并，最终各端会收敛到一致结果，不需要人工处理版本冲突。

#### 5. 光标同步是怎么做的？ 光标同步为什么不直接放进文档状态里？
光标同步其实分两部分：

- 位置状态上报：本地用户的光标位置、选区范围通过 WebSocket 发给其他用户
- 前端渲染展示：其他客户端收到后，在编辑器中渲染对应用户的光标和选区高亮

它和文档内容同步是分开的，因为光标是“临时协作状态”，不是文档持久内容。

因为光标、选区、在线状态这些是临时协作状态，不是文档内容本身。
如果把它们直接写进文档，会污染持久化内容，增加同步复杂度，也会让数据体量变大。
更合理的做法是把正文和 awareness/presence 分开，正文走 Yjs 持久状态，光标和在线信息走单独的 WebSocket 协作状态。

#### 6. 如果一个用户断开连接了，其他人怎么知道他离线了？重新上线后怎么恢复状态？

可以通过 WebSocket 连接状态变化和心跳机制判断。

当连接断开、超时未响应或服务端检测到心跳失败时，就可以把该用户标记为离线，并从在线用户列表和光标列表中移除。

短暂断开通常会加重连机制和短暂保留策略，比如：

- 短时间内断线不立即清除状态
- 客户端自动重连
- 重连后重新拉取房间状态并恢复同步

重新上线后，客户端会重新加入对应房间，服务端会下发当前最新文档状态或缺失的增量更新。
由于 Yjs 支持状态同步和增量合并，客户端可以把本地离线期间积累的更新与远端最新状态自动合并，恢复到一致视图。

#### 7. 增量命令的数据结构是什么？
一次编辑动作在系统里被抽象成的可传输、可回放、可撤销的数据对象。
我理解增量命令本质上是一次编辑操作的结构化描述，通常会包含操作类型、修改范围、内容、来源标记 origin、时间戳等信息。底层在协同编辑里更接近 Yjs 的增量 update，业务层为了更好控制历史、同步和落库，会再封装成统一的数据结构。

```js
{
  id: string
  type: 'insert' | 'delete' | 'replace' | 'format' | 'batch'
  from: number
  to: number
  content?: string | object
  attrs?: Record<string, any>
  origin: 'local' | 'remote' | 'system' | 'ai'
  timestamp: number
}
```

### 8. 历史栈记录的是什么？如何维护？
项目里用的是 Tiptap，它底层基于 ProseMirror，自带 history 机制，主要管理本地编辑的 undo / redo，放在前端内存里，页面刷新后历史栈会丢失。

undoStack：记录已执行但可撤销的操作，redoStack：记录被撤销后可重做的操作。

但在协同编辑场景下，撤销会更麻烦，因为远端用户的修改会导致光标位置和操作范围漂移，不能简单按本地历史直接回放，否则可能误删别人的内容，所以需要结合 Yjs 的协同状态来处理撤销范围和一致性。


### 9. 协同编辑下本地 undo 怎么避免影响别人？如果要撤销的内容被别人改了怎么办？
核心原则：本地 undo 只撤销“当前用户自己产生的操作”，不直接撤销远端同步内容。

实现上会把本地操作和远端操作分链路处理：
- 本地主动编辑：进入本地 undo/redo 历史（Tiptap / ProseMirror history）
- 远端同步更新：只合并到文档状态，不进入本地可撤销历史

区分方式通常依赖 transaction 的 origin / meta 标记，保证撤销范围始终限定在“我自己的历史链”。

如果本地要撤销的内容已经被远端协作者改动，不能按原位置硬回退，需要先做位置映射和状态校验；
- 能安全撤销：执行最小范围撤销
- 不能安全撤销：提示“该内容已被其他协作者修改，无法直接撤销”，或让用户二次确认

所以在正确设计下，Ctrl+Z 不会误删别人的内容。

### 10. 多人同时编辑时，怎么保证性能？

核心是减少全量传输，只同步增量；
其次是把光标状态和正文状态分离，避免无效更新；
同时可以做节流、批量发送和重连补偿，减少 WebSocket 压力。
文档持久化也尽量异步批量处理，避免每次输入都落库。

### 11. 批量操作、增量命令和 Yjs update 的区别
可以分三层理解：

1) 编辑命令层（Tiptap / ProseMirror command）
- 这是业务动作，比如插入文本、设置格式、删除区块。
- 多个命令可以合并进同一个 transaction，一次性提交。

2) 事务层（ProseMirror transaction）
- 文档修改在底层都通过 transaction 落地。
- 如果把一组批量操作放进同一个 transaction，它们会成为一条历史记录，undo 时可整体回退（如 AI 生成整段、粘贴大段文本、整块格式化）。

3) 协同同步层（Yjs update）
- transaction 提交后，会被编码成 Yjs 的增量 update，在协同端之间同步。
- 它关注的是“状态增量同步”，不是面向用户语义的编辑命令。

所以：命令是“做什么”，transaction 是“怎么原子提交”，Yjs update 是“怎么把结果同步给别人”。协同场景下还需要额外处理冲突与一致性。

### 16. 如果批量操作中途失败怎么回滚
我会优先做预检和事务式提交，尽量避免半执行状态；
如果已经部分生效，就通过执行前快照或反向补偿把文档恢复回去。


### B. 大文件上传链路
#### 1. 大文件上传链路是怎么设计的？
把它设计成“切片上传 + 断点续传 + 秒传 + 任务恢复”四层能力。前端先按固定大小切片，对每个分片做唯一标识，再并行上传；服务端保存每个分片的接收状态。这样用户断网或者刷新页面后，只需要补传缺失分片，不需要整个文件重传，尤其适合科研场景里几十 MB 甚至更大的资料、图片和附件。

#### 2. 怎么实现秒传？
核心是先算文件指纹，比如文件名、大小、hash 或者分片 hash 的组合。上传前先让前端和服务端做一次校验，如果服务端已经存在同样的文件标识和完整分片记录，就直接返回“已存在”，前端就可以跳过真正的上传流程，这就是秒传。它能显著减少重复文件的网络和存储成本。

#### 3. 断点续传具体怎么做？
每个分片上传成功后，服务端都会记录状态。用户重新打开页面或者上传中断后，前端先请求服务端拿到已上传分片列表，再只补传缺失的部分。上传完成后，服务端再做分片合并、完整性校验，最后把文件元信息写入数据库。这样断点续传不是“从头开始重试”，而是“只补缺口”。

#### 4. 如果上传中途失败，怎么恢复任务？
会把上传任务本身当成一个可恢复状态机来设计，至少包含待上传、上传中、部分完成、已完成、失败可重试这几种状态。前端刷新后可以重新拉取任务状态，服务端根据任务 ID 和分片状态恢复上下文，避免用户重新选择文件。这样用户体验会比“失败后全量重传”稳定很多。


### C. SSE 流式问答
#### 1. 为什么在问答里选 SSE？SSE 有什么缺点？
因为这个场景本质上是“服务端持续生成，前端持续消费”，服务端单向推流就够了。SSE 基于 HTTP，接入和部署都相对简单，而且浏览器原生支持断线重连，特别适合大模型逐 token 输出的问答场景。相比 WebSocket，它实现成本更低，也更贴合“只需要服务端往前端推文本”的需求。

SSE 的缺点
- 只能服务端单向通信
- 浏览器连接数限制
- 长连接稳定性问题
- 不适合高频双向实时通信

#### 2.  fetch + ReadableStream 逐 token 渲染，具体怎么做？
前端不是一次性等接口返回完整文本，而是用 `fetch` 拿到流式响应，再从 `ReadableStream` 里不断读取 chunk。每个 chunk 到达后先做解析，再把增量文本拼到当前回答里，页面也就能逐步渲染出来。这样用户能第一时间看到模型在“打字”，体感会比等完整结果好很多。

#### 3. SSE 里的粘包、半包你们怎么处理？
我会按事件边界来解析，而不是按固定字节长度切。流式响应过来的内容可能一个 chunk 里包含多条事件，也可能一条事件被拆成多个 chunk，所以前端需要维护一个缓冲区，持续累积数据，直到读到完整分隔符后再解析成单条消息。这样才能保证 token 不丢、不乱序。

#### 4. 流式输出时怎么支持中断？
前端会把当前请求和 `AbortController` 绑定起来。用户点击停止时，直接 abort 当前流式请求，后端也可以根据连接断开停止继续生成。这样既能避免继续消耗算力，也能让用户在结果不满意时及时打断，重新发起新的请求。

#### 5. 多轮上下文是怎么管理的？
我会回答：我们不是简单把所有历史都塞进去，而是结合会话记忆和当前任务上下文做管理。短期内保留最近几轮关键信息，长期则压缩成摘要，避免 token 膨胀。这样既能保持对话连续性，也能让模型在长对话里不容易跑偏。

#### 6. 重连重试怎么做？
我会在流式请求外面包一层重试控制，断线时先做有限次快速重试，重试失败后再进入退避等待。重新连接后，前端会把上一次已经消费到的位置或会话标识带给后端，尽量从断点继续，而不是完全重新开始。这样可以减少重复生成，也能让用户体验更平滑。

#### 7. 为什么要指数退避？
因为流式失败有时候不是业务错误，而是网络抖动、瞬时拥塞或者服务端短暂繁忙。如果一失败就立刻高频重试，会把问题放大成“重试风暴”，既浪费资源，也更容易压垮服务。指数退避可以让重试间隔逐步拉长，给系统恢复时间，同时避免无意义的密集请求。

### D. L1 / L2 双层记忆
#### 1.为什么要做 L1 / L2 双层记忆？
因为长对话里既需要“最近几轮的细节”，也需要“长期稳定偏好”。L1 解决短期连续性，L2 解决长期记忆和用户画像。只做一层的话，要么上下文太短记不住细节，要么全塞进去导致 token 爆炸，所以双层记忆更平衡。

#### 2. L1 和 L2 分别存什么？
L1 存最近几轮对话的详细内容，比如当前任务、临时约束、最近一次修改意见；L2 存经过压缩后的长期摘要，比如用户偏好的输出风格、常用任务类型、历史协作习惯。L1 更像工作台上的便签，L2 更像长期知识库。

#### 3. 为什么 L1 放 Redis，L2 放关系型数据库？
L1 读写频率高，要求延迟低，Redis 很适合做短期记忆缓存；L2 是长期稳定信息，通常需要按用户维度查询、更新和审计，放在 PostgreSQL 这类关系型数据库更适合管理结构化摘要。这样设计可以兼顾实时性和可维护性。

#### 4. Rolling Summary 是怎么做的？
我会把早期对话压缩成“任务 - 动作 - 结果”这样的摘要结构，保留最有价值的信息，而不是机械截断文本。比如用户让系统先总结、再补 Mermaid 图，这个协作过程就会被压缩成一条长期摘要，后面继续对话时模型还能知道之前怎么协作的。

#### 5. 如果用户突然改需求，记忆怎么适应？
L1 会优先记录当前最新需求，快速覆盖旧的临时细节；L2 则保留稳定偏好和历史背景，不会因为一次临时改动就把长期画像冲掉。这样既能响应“这次我想换个方向”，又不会丢掉“这个用户平时喜欢简洁、结构化输出”这种长期信息。

### E. RAG 检索增强问答
#### 1. 你们为什么要做 RAG？
因为科研场景对准确性要求高，通用大模型很容易出现专业幻觉。RAG 的核心价值是把外部知识检索进来，再基于检索内容生成答案，这样模型不完全靠参数记忆，回答会更贴近项目里的真实文档和知识库，可信度也更高。

#### 2. 你们的检索链路是怎么做的？
我们不是简单拿 TopK 片段直接拼上下文，而是先做文档解析和切片，再做向量化和索引；检索时采用 BM25 关键词稀疏检索 + 向量相似度稠密检索的混合召回，再做重排和上下文压缩，最后把高相关内容注入回答流程。这样能兼顾关键词精确匹配和语义召回。

#### 3. 为什么要混合召回，而不是只用向量检索？
因为科研文本里既有语义相似，也有非常关键的术语和专有名词。只靠向量检索，有时候会把语义接近但术语不对的片段召回上来；只靠关键词检索，又容易漏掉表达不同但意思相近的内容。混合召回可以更稳地平衡两者，提高召回质量。

#### 4. 来源回填为什么重要？
因为用户不只想要答案，还想知道答案来自哪份资料、哪段原文。来源回填能提升可解释性，也方便科研团队回查原始依据。对于专业场景来说，这比“模型直接给个结论”更可信，也更容易通过人工审核。

#### 5. 如果文档更新了，索引怎么同步？
我会优先做增量更新，只重建发生变化的分片向量和索引；如果文档变更量特别大，再考虑批量重建。这样可以避免全量重建带来的性能波动，也能保证检索结果尽可能跟随最新文档状态。

### F. 文档内 AI 辅助创作
#### 1. 你们为什么要做文档内 AI 辅助创作，而不是单独聊天？
因为用户的真实需求不是“跟模型聊天”，而是在文档编辑场景里直接完成续写、润色、翻译、总结、生成图表这些动作。把 AI 能力嵌进文档里，可以让用户在同一个工作流中完成检索、编辑和改写，减少来回切换工具的成本。

#### 2. 你们的多智能体是怎么分工的？
我们设计了 Orchestrator / Planner / Worker / Critic 这样的链路。Orchestrator 负责总控，Planner 负责拆任务，Worker 负责执行具体步骤，Critic 负责检查结果和反思修正。这样做的好处是把复杂任务拆开，既提升并行效率，也提升输出质量。

#### 3. 为什么要用 ReAct 和 Reflection Loop？
我会回答：ReAct 的好处是让模型按“思考 - 行动 - 观察”的方式推进，而不是一口气瞎生成。Reflection Loop 则是在结果不达标时触发再思考和修正，避免模型陷入无限推理或者带着错误继续输出。对科研写作这种高精度任务来说，这个约束非常重要。

#### 4. Mermaid 图是怎么生成的？
先让 Summary Agent 提炼核心模块，再让 Mermaid Agent 把模块关系映射成图语法，像“模块 A 调用模块 B”就会变成 `A --> B`。如果结构复杂，就拆成子图，尽量保证前端渲染清晰，避免图太大看不懂。

#### 5. 前端怎么知道该替换哪里、插入哪里？
后端不会只返回一段纯文本，而是返回结构化协议，里面包含操作类型、起始位置、结束位置、内容和是否生成 Mermaid 块等信息。前端根据这些元数据去执行替换、插入或高亮，这样 AI 结果才能真正回填到编辑器里，而不是只停留在聊天框。

# Memory
## 逐字解说
L1记忆是滑动窗口，存最近几轮对话的详细内容，保证短期上下文连贯；
L2记忆用Rolling Summary，定期把L1的内容压缩成摘要存起来，管理长期对话。
比如用户连续聊10轮文档创作，L1存最后3轮的具体需求，L2存前7轮的摘要，这样既能在token限制内记住用户习惯，又不会丢失关键信息。

双层记忆不仅记用户输入和最终输出，也会记录Agent间的关键流转信息。
比如L1滑动窗口会存“Planner拆分出Summary和Memorize任务→Worker执行→Critic检查通过”这样的核心步骤，确保短期对话中Agent能衔接之前的分工；
L2的Rolling Summary会压缩这些流转逻辑，比如“用户需要带架构图的总结，系统调用了Summary和Memorize Agent协作完成”，这样长期对话时，Agent依然知道之前是怎么配合的。

## QA
### 1. 如果用户在长对话中突然改变需求，双层记忆机制如何适应这种变化？
当用户改变需求时，L1滑动窗口会优先清空旧需求的详细信息，存入新需求的内容，确保短期协作聚焦当前任务；L2的Rolling Summary会新增一条“用户当前需求变更为XX”的摘要，同时保留之前的个性化偏好，比如用户之前喜欢简洁风格，新需求下依然会沿用这个偏好，不会完全丢失历史习惯。

新需求可以通过用户输入的关键词和意图来判断，比如之前用户一直说“生成论文总结”，突然说“帮我把总结翻译成英文”，这就是需求从“生成”变成“翻译”。这时候L1会优先存“翻译”或“不要架构图”这些新指令，L2保留“用户需要论文总结”的历史任务背景，帮助Agent理解新需求是基于什么任务来的。

### 2. Rolling Summary 用的什么方法？
用的是“任务-动作-结果”三元组压缩法，明确了要保留哪些类型的信息；
比如把“用户让生成论文总结，Planner调用Summary Agent，生成300字带架构图的内容，Critic检查通过”压缩成“任务：论文总结；动作：Summary+Memorize Agent协作；结果：带Mermaid图的300字总结”。
这种方法能直接保留对后续任务有用的核心要素，对包含Agent流转和用户偏好的对话来说，三元组压缩丢失关键信息的概率更低。

压缩时，还会用LLM对对话内容进行语义分析来填充这个框架，优先保留用户明确提到的偏好和任务核心要素，比如用户说“图表要简洁风格”，压缩时会特意保留“简洁风格”这个偏好词。如果遇到模糊信息，会参考历史对话中用户的选择来补充，比如用户之前每次选300字总结，压缩时就默认“结果：300字总结”，避免丢失隐性需求。

### 3. 如果L2的Rolling Summary积累了多条摘要，怎么决定哪条优先被调用？
会根据时间戳和需求相关性排序，最近的摘要优先，同时用LLM计算当前用户需求和每条摘要的语义相似度，相似度高的摘要排在前面，确保优先唤起和当前任务最相关的历史信息。

### 4. 记忆存在什么数据库里？
L1短期记忆可以存在Redis里，因为它需要高频读写，Redis的速度能满足实时对话需求。L2长期记忆的Rolling Summary适合存在PostgreSQL这类关系型数据库，方便按用户ID查询和更新长期偏好、任务历史这些结构化信息。

### 5. L1 L2 什么时候会被保存
1. L1 是会话级的短期记忆，所以它是高频更新的，基本上每轮对话都会更新。

具体来说：

用户发言后，先把当前轮用户输入写入 L1
模型生成完成后，把 assistant 的输出也写入 L1
如果这一轮涉及 Agent 协作，像 Planner、Worker、Critic 的关键中间结果，也会写入 L1 的滚动摘要里
当上下文接近 token 上限时，会触发 rolling summary，把更早的内容压缩后继续保留在 L1

2. L2 是跨会话的长期事实记忆，不能高频乱写，所以它通常是低频、批量、带条件更新。

常见更新时机有三种：
第一种，会话结束时更新
在一轮会话结束后，系统会从整段对话里抽取稳定事实，比如：

用户偏好正式语气
用户喜欢简洁输出
用户常做技术文档总结
然后写入 L2。

第二种，达到一定命中次数后再更新
为了避免误判，L2 通常不会因为用户说一次就立刻记住，而是要满足：

事实高置信
可验证
出现次数达到阈值，比如 2 次或 3 次
这样可以减少错误记忆污染。

第三种，用户显式确认时更新
比如用户明确说：

“以后都用中文回答”
“默认不要太口语化”
“我在做科研写作”
这种明确指令可以直接进入 L2，或者进入待确认队列后再更新。



# RAG
## 逐字解说
RAG 这一块我们也没有做成固定链路，而是做成 Agentic RAG。

也就是说不是所有请求都默认检索，而是由 Retrieval Agent 判断当前任务是否需要 RAG。

如果需要，它会先做 Query Rewrite，把用户问题改写成更适合检索的查询，再执行 BM25 + 向量检索的混合召回，并做 rerank 和上下文压缩，最后把高相关内容注入 Workspace。

因为长上下文场景里，如果直接把 TopK chunk 全塞进去，很容易导致 token 爆炸，所以我们做了 Context Compression，比如 chunk summary、chunk merge 等。

### 1. 分片策略是什么样的？
对于普通文本，采用递归语义切片，先按段落分割，若段落长度超过512 token，再按句子递归拆分，确保每个分片在256-512 token之间，且不拆分完整句子。
对于Markdown格式文本，优先按标题层级分片，同一标题下的内容再按代码块、列表块分割，保证每个分片保留标题上下文和完整语义块，这样向量化时能更准确捕捉内容主题，提升召回相关性。

### 2. 那针对格式化文本，除了按标题和块分片，有没有处理过特殊格式，比如表格、公式或者代码块？
假设处理表格时，会将表头和对应行数据合并成“表头：值”的文本片段，公式保留LaTeX原格式，代码块按函数或逻辑块分割。

处理代码块，按语法树解析，比如借助tree-sitter工具，优先按函数、类这些语法单元切分，每个分片控制在1000到1500字符，同时保留函数签名和注释。

### 3. 来源回填是如何做的？
来源回填就是在RAG生成回答后，把引用的文档分片信息附在回答末尾。具体做法是检索时记录每个相关分片的元数据，生成回答时用这些元数据替换掉回答里的占位符，让用户知道信息来源。

### 4. 索引是如何做的？
索引用BM25关键词索引和向量相似度索引混合，融合时采用RRF方法：对每个分片，分别取BM25排名和向量排名，按1/(排名+60)计算分数后相加，最后按总分取Top结果，能稳定平衡关键词匹配和语义相关性。

### 5. 向量数据库用的什么？
使用 Redis Stack，自带了 RedisSearch 和 RedisJSON 这些模块，能直接存向量，还支持 HNSW 索引和余弦相似度这些常用的向量检索功能，部署和集成也方便，很适合 RAG 这种需要实时检索的场景。

### 6. 如果文档更新频繁，你们是怎么保证Redis里的向量索引和最新文档同步的？是全量重建还是增量更新？
可以设计增量更新策略，当文档有新增或修改时，只对变化的分片重新生成向量，然后调用Redis的FT.ADD或FT.UPDATE命令更新索引，删除旧分片的向量。如果是大量文档更新，可先暂停检索，全量重建索引后再切换，这样能减少对服务的影响。

### 7. 那在检索时，如果用户查询的是长文本问题，比如多轮对话后的复杂问题，向量索引还能准确召回相关分片吗？
可以先对长查询进行语义压缩，提取核心问题生成向量，再去检索。或者用多向量检索，把长查询拆成多个子问题向量，分别召回后合并结果，这样能提升长问题的召回准确性。

### 7. 在RAG系统里，你怎么评估检索到的文档和用户问题的相关性？有没有什么具体的指标或者方法？
召回率看是否能把所有相关分片都找出来，准确率看召回的分片中真正相关的比例。
除了这两个，还可以用NDCG评估排序质量，或者人工标注Top5分片的相关性打分。

### 8. 重排策略
重排阶段可以加入交叉编码器，对召回的Top10分片和用户问题做二次语义匹配，进一步提升相关性。
比如先用RRF拿到10个分片，再用交叉编码器打分排序取Top5，这样兼顾效率和准确性。


# Agents
## 逐字解说
这个部分做了一套文档内的 AI 多智能体协作系统，主要是想解决传统单 Prompt 模式在写作场景下的问题。

整体上设计的是一个Fast / Swarm 双通道架构。

1. Fast 模式主要处理低复杂度、低时延任务，比如翻译、简单润色等。这类请求会直接经过 Router Agent 路由对应到具体的 Worker Agent，然后调用 LLM 生成结果。
2. Swarm 模式主要处理复杂任务，比如长文档总结、Mermaid 生成等。复杂任务会进入多智能体协作链路。

它会先对请求做复杂度评估，比如：输入文本长度、是否需要知识库索引、是否涉及多步骤生成，然后决定走 Fast 还是 Swarm。

如果进入 Swarm 模式，会先由 Planner Agent 做任务拆解。

比如用户说：

‘请根据当前文档生成技术方案，并补 Mermaid 架构图。’

Planner 会把任务拆成几个步骤：

1. 先总结文档核心模块
2. 提取系统结构
3. 生成 Mermaid 草稿
4. 最后统一润色输出

之后任务会进入多个 Worker Agent 并行执行。

比如：

Summary Agent 负责总结
Retrieval Agent 负责 RAG 检索
Mermaid Agent 负责结构图生成
Writing Agent 负责最终文本生成

中间结果不会直接拼 Prompt，而是写入 Shared Workspace，也就是共享工作区，所有 Agent 都可以读取前序结果，而不需要重新理解全文。

生成完成后，还会经过 Critic Agent。

Critic 主要负责：检查结果是否偏题、引用合法性、Markdown 格式等问题。

如果发现问题，会进入 Reflection Loop，再次修正生成结果。

最后由 Merger Agent 做统一结果聚合，并转换成结构化协议返回前端。

因为我们这个场景不是普通聊天，而是编辑器内 AI 操作，所以前端不仅需要拿到文本，还需要知道：

应该如何回填
替换哪里
插入哪里
是否生成 Mermaid 块

基于此，我们设计了一套统一响应协议。

```
                    User Request
                           │
                   Lead Orchestrator
                           │
               Complexity Scorer Agent
                 ┌─────────┴─────────┐
                 │                   │
              FAST                SWARM
                 │                   │
          Router Agent         Planner Agent
                 │                   │
          Single Worker        Task DAG Planning
                 │                   │
                LLM        ┌─────────┼─────────┐
                           │         │         │
                    Retrieval   Writing   Mermaid
                      Agent      Agent      Agent
                           │         │         │
                           └──── Shared Workspace ────┐
                                                      │
                                                Critic Agent
                                                      │
                                                Reflection Loop
                                                      │
                                                Merger Agent
                                                      │
                                                 Final Output
```

## QA 
### 1. 这个框架相比单 Agent 方案，任务处理效率提升了多少？
多 Agent 并行写作能使复杂任务平均处理时间缩短，同时 评审的 Reflection Loop能提高输出正确率，降低 Markdown 格式错误等。

### 2. 为什么选择Planner-Worker-Critic三层架构？
因为科研文档创作任务复杂，Planner负责拆解任务保证逻辑清晰，Worker并行执行提升效率，Critic通过反思循环保证结果质量，三层分工比单一Agent更适配“复杂任务+高精度”的科研场景。

### 3. Blackboard Pattern在高并发下如何保证数据一致性？
我们给共享工作区的每个数据块加了版本号，Agent写入时需校验版本，避免覆盖冲突；同时用读写锁控制访问，Worker读时加共享锁，写时加排他锁，高并发下任务错误率控制在5%以内。

### 4. Critic的Reflection Loop触发阈值是什么？
比如Markdown语法错误≥2处、引用格式错误率＞10%、内容与文档核心主题匹配度＜80%时，会触发反思循环，针对性回溯对应Worker Agent修正。
还能加个收敛检测，要是连续两次反思改进幅度小于5%，也直接停，避免无效循环。

### 5. ReAct范式
简单说就是让模型“思考-行动-观察”循环，先分析任务该做什么，再调用工具执行，然后根据结果调整下一步，像人解决问题一样有逻辑地一步步推进，最后得出答案。

### 6. Memorize Agent如何抽象复杂系统结构生成Mermaid？
先通过Summary Agent的文档总结提取核心模块，再用规则库映射模块关系到Mermaid语法，比如“模块A调用模块B”转化为“ A --> B”，复杂结构拆分为子图表，确保前端渲染清晰。

### 7. 响应式协议如何定义回填位置？
协议包含“操作类型+起始位置+结束位置+内容”，比如“replace:100:200:生成文本”，前端根据起始/结束索引定位文档位置，替换或插入内容，适配编辑器实时更新。

### 8. Worker Agent结果冲突时如何仲裁？
冲突时由Planner Agent根据任务优先级仲裁，比如“生成结构图”任务中，Memorize Agent结果优先级高于Summary Agent，以Memorize结果为准写入共享工作区。

### 9. 基于 ReAct 范式，设置 Reflection Loop 阈值
可以结合任务类型动态设阈值，比如简单的文案生成设3次，复杂的技术文档设5次，超过就终止
