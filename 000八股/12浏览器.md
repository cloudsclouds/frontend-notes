# 浏览器

## 关键渲染路径

也叫浏览器渲染管线，是**浏览器把 HTML/CSS/JS 代码，一步步转换成屏幕上可见像素的完整流程**，它决定了页面首次显示的速度，也是前端性能优化（重排、重绘、合成）的核心依据，现代浏览器标准流程分为 **6 个核心阶段**：

### 6 步流程

##### 1. 解析 HTML → 构建 DOM 树

- 浏览器读取 HTML 字节流，解码、分词，**生成 DOM 文档对象模型**
- DOM 树描述了页面的**结构与节点关系**（div、p、文本节点等），不包含样式
- 遇到 <script> 会阻塞解析（默认同步 JS），直到 JS 执行完毕

##### 2. 解析 CSS → 构建 CSSOM 树

- 解析所有 CSS（内联、外链、样式表），**生成 CSSOM 样式对象模型**
- CSSOM 包含所有样式规则、继承、层叠、优先级计算，是**带样式的树形结构**
- CSS 同样会阻塞渲染，因为无样式的页面不会展示

##### 3. 合并 DOM + CSSOM → 生成渲染树（Render Tree）

- 只保留**可见节点**：剔除 display:none、head 内节点等不可见元素
- 为每个可见节点**匹配最终计算样式**（继承 + 层叠后的最终样式）
- 渲染树 = 可见节点 + 最终样式，**只负责 “要画什么”，不负责位置尺寸**

##### 4. 布局（Layout / Reflow）→ 计算几何信息

- 也叫**重排**，核心是计算元素的**盒模型、位置、尺寸**
- 依据渲染树，计算每个元素在视口中的坐标、宽高、排版（流式布局、Flex、Grid 等）
- 只要页面结构 / 尺寸变化，就会触发重新布局，**开销极大**

##### 5. 绘制（Paint / Repaint）→ 填充像素

- 也叫**重绘**，将布局后的元素**填充实际像素**
- 绘制内容：颜色、背景、边框、阴影、文字、图像等
- 绘制是分层进行的，不改变布局只改样式（如颜色）只会触发重绘，**比重排轻量**

##### 6. 合成（Composite）→ 图层合并上屏

- 浏览器将多个绘制好的图层，按层级关系**合成最终画面**，发送给 GPU 显示
- 只触发合成的操作（如 transform、opacity）不经过布局和绘制，开销最小

#### 核心概念：重排 → 重绘 → 合成（性能优先级）

三者触发成本从高到低：

1. **重排（Layout）**触发：修改 DOM 结构、宽高、位置、字体、窗口 resize、获取 offsetWidth 等布局属性影响：后续必须执行重绘 + 合成，**性能开销最大**
2. **重绘（Paint）**触发：修改颜色、背景色、阴影、visibility 等**不影响布局**的样式影响：跳过布局，直接重绘 + 合成，**开销中等**
3. **合成（Composite）**触发：修改 transform、opacity（需开启硬件加速）影响：仅合成图层，开销最小，性能最优

#### 渲染管线的核心优化原则（面试高频）

1. 减少**重排**：批量修改 DOM、使用 class 代替逐一样式修改、避免频繁读取布局属性
2. 优先用**合成属性**：动画用 transform 代替 top/left，用 opacity 控制显隐
3. 降低 CSS 选择器复杂度：减少浏览器匹配 CSSOM 的开销
4. 异步加载 JS：async/defer 避免阻塞 HTML 解析
5. 压缩关键资源：减小 HTML/CSS/JS 体积，缩短首屏渲染时间

## 1. 浏览器为何采用多进程架构？单进程浏览器有什么缺陷？

- 单进程浏览器缺陷：

- 不稳定： 插件崩溃、渲染线程崩溃会导致整个浏览器崩溃。
- 不流畅： 页面上某个脚本死循环（如 JS 死循环）或插件卡死会阻塞整个浏览器，导致所有页面无响应。
- 不安全： 插件或页面脚本可以轻易获取系统任意资源，易被利用进行恶意操作。

- 多进程架构解决方案：

- 解决不稳定： 每个标签页（渲染进程）、每个插件都有独立的进程。一个标签页或插件崩溃，只会影响自身进程，不会影响浏览器主进程和其他页面/插件进程。
- 解决不流畅： 脚本死循环只会阻塞它所在的渲染进程（标签页），其他页面标签运行不受影响。
- 解决不安全： 使用沙箱 (Sandbox) 技术。沙箱可以看作是操作系统给进程加的限制。运行在沙箱中的程序（如渲染进程）被严格限制：无法读写硬盘数据、无法访问敏感的系统资源。当渲染进程需要访问网络或文件系统时，必须通过浏览器主进程进行代理访问，结果再通过 IPC 返回给渲染进程。

## 2. 渲染进程（浏览器内核）包含哪些核心线程？它们各自负责什么？协作关系如何？

- GUI 渲染线程：

- 负责解析 HTML、CSS，构建 DOM 树、CSSOM 树、Render 树 (渲染树)，进行布局和绘制（像素填充），最终渲染出用户可见的页面。
- 它与 JS 引擎线程是互斥的。当 JS 引擎线程在执行 JS 脚本时，GUI 渲染线程会被挂起（被“冻结”），必须等到 JS 引擎空闲时才能执行。所以 JS 执行时间过长会阻塞页面渲染（导致卡顿），需要尽量控制 JS 体积和执行时间。

- JS 引擎线程 (如 V8)：

- 负责解析和执行 JavaScript 代码。V8 引擎本身是在 JS 引擎线程上运行的。
- 一个渲染进程通常只有一个 JS 引擎线程。JS 引擎线程与 GUI 渲染线程互斥。

- 定时器触发线程：

- 负责管理 `setTimeout` 和 `setInterval`。JS 引擎是单线程的，如果定时器的计时在 JS 引擎线程上运行，阻塞性的任务会影响计时的准确性。因此需要单独的线程来计时和触发。
- 定时器到达指定时间后，计时线程会通过事件触发线程将定时器的回调函数添加到任务队列中。

- 事件触发线程：

- 负责管理任务队列（Task Queue）。当一个事件需要被处理时（如定时器触发、异步请求完成、用户点击、I/O 操作完成），这个线程会把事件对应的回调函数添加到任务队列的队尾。
- 等待 JS 引擎空闲时（即执行栈为空），事件循环机制会将任务队列中排在最前面的任务取出执行。

- 异步 HTTP 请求线程 (如 XMLHttpRequest)：

- 当发起网络请求（如 `XMLHttpRequest` 或 `Fetch`）时，浏览器会新开一个线程来处理这个请求。
- 该线程负责检测请求的状态变化（如完成、失败）。当状态发生变化并且请求对象设置了回调函数（如 `onreadystatechange`）时，该线程会将这个回调函数作为一个事件通知事件触发线程，由它添加到任务队列中等待 JS 引擎执行。

## 3. 什么是事件循环 (Event Loop)？宏任务 (macrotask) 和微任务 (microtask) 有什么区别？执行顺序是什么？

- 事件循环 (Event Loop): 是浏览器或 Node.js 解决 JS 单线程运行时异步任务不阻塞主线程的机制。核心思想是：JS 引擎有一个主线程负责执行代码（形成执行栈）和一个Event Loop 线程负责与其它线程（I/O、计时器等）通信和管理事件队列。
- 运行流程：

- 执行一个宏任务： 从任务队列中取出最老的一个任务（通常是`<script>`标签本身的代码）开始执行（同步代码）。
- 清空微任务队列： 该宏任务执行结束前或后（V8 引擎通常在该宏任务结束后立即清空微任务队列），JS 引擎会检查微任务队列（Microtask Queue）。如果微任务队列中有任务，JS 引擎会按添加顺序一次性全部执行完，直到微任务队列清空。
- UI渲染（如需要）： 浏览器可能会根据策略进行 UI 渲染（将渲染结果提交到 GPU 绘制）。注意：渲染在微任务之后，下一个宏任务之前。
- 进入下一轮循环： 开始执行下一个宏任务（从任务队列取下一个）。再次回到步骤 2（清空微任务队列）。如比无限循环。

- 宏任务 (macrotask) vs 微任务 (microtask):

- 来源：

- 宏任务：主要由浏览器发起（脚本代码、UI 交互）。
- 微任务：主要由 JavaScript 自身发起（Promise、MutationObserver）。

- 执行时机：

- 微任务：在本轮事件循环的宏任务执行完、UI 渲染前，一次性执行完所有微任务。
- 宏任务：在下一轮事件循环开始时执行。

- 常见类型：

- 宏任务： `<script>`整体代码、`setTimeout`、`setInterval`、`setImmediate`(Node)、I/O (鼠标事件、键盘事件、文件读写)、UI Rendering (UI 渲染本身不是宏任务，但触发的回调如 `resize` 是)、`postMessage`。
- 微任务： `Promise.then()`/`Promise.catch()`/`Promise.finally()` 的回调、`process.nextTick`(Node, 优先级最高)、`MutationObserver` 的回调（已废弃的 `Object.observe`）。

- 核心差异： 一个事件循环周期中，当执行栈为空时，会先检查并清空所有微任务，然后才会取下一个宏任务执行（也可能先做 UI 渲染）。微任务具有“插队”的能力。

## 4. 详细描述从在浏览器地址栏输入 URL 到页面最终渲染完成的整个过程（涵盖导航阶段和渲染阶段）。

- 🅰️ 导航阶段 (Network & Process)：

- 用户输入 & URL 处理：

1. 用户在地址栏输入内容（URL 或搜索关键字）。
2. 如果是搜索内容： 浏览器使用默认搜索引擎合成新的带搜索关键字的 URL。
3. 如果是 URL： 浏览器根据规则（如检查协议）将其组装成完整的 URL（可能补全 `http://` 或 `https://`）。

- 查找缓存：

1. 网络进程检查本地缓存（包括浏览器缓存、Service Worker 缓存等）。
2. 检查资源是否新鲜（如通过 `Cache-Control`/`Expires` 检查强缓存，通过 `If-Modified-Since`/`If-None-Match` 检查协商缓存）。
3. 若缓存有效，直接使用缓存资源，跳转至准备渲染进程。否则继续。

- DNS 域名解析：

1. 网络进程解析 URL 中的域名对应的 IP 地址。
2. 查询顺序：浏览器缓存 → 操作系统缓存/ hosts 文件 → 本地 DNS 服务器 → 根 DNS 服务器/顶级域 DNS 服务器/权威 DNS 服务器（递归/迭代查询）。

- 等待 TCP 连接 & 三次握手：

1. 浏览器同一域名下最多允许建立约 6 个 TCP 连接。如果当前连接数已满，请求需排队等待。
2. 进行 TCP 三次握手建立可靠连接（避免资源浪费）。

- 发送 HTTP(S) 请求：

1. 建立连接后，浏览器构建 HTTP 请求报文（请求行：方法+URL+HTTP版本、请求头、请求体），通过该连接发送给服务器。
2. HTTPS 连接会先进行 TLS/SSL 握手协商加密参数。

- 服务器处理请求 & 返回响应：

1. 服务器接收请求并处理。
2. 构建 HTTP 响应报文（状态行：状态码+状态描述、响应头、响应体）。

- 浏览器接收响应 & 处理：

1. 网络进程接收响应数据。
2. 检查状态码：

- `301`/`302`：重定向。从响应头 `Location` 中获取新地址，跳转到第一步（处理新 URL）。
- `200 OK`：检查响应头 `Content-Type`。

- 若是 `text/html`：准备渲染 HTML。
- 若是 `application/octet-stream`：触发下载流程。

1. 如果请求资源类型是 HTML，准备进入渲染流程。

- 准备渲染进程：

1. 浏览器进程检查新页面的协议和根域名。
2. 如果与某个已打开的页面相同，则复用该页面的渲染进程。否则，创建一个新的渲染进程。

- 提交文档 (Commit Document)：

1. 浏览器进程通过 IPC 向渲染进程发送“提交文档”的消息。
2. 渲染进程接收到消息后与网络进程建立数据传输连接（管道）。
3. 渲染进程边接收网络进程传输的 HTML 数据，边开始解析。
4. 当渲染进程确认所有基础资源（HTML）已经接收到（不一定完全下载完毕），会通过 IPC 向浏览器进程发送“确认提交”的消息。
5. 浏览器进程收到“确认提交”消息后，更新浏览器界面状态：

- 更新地址栏 URL
- 更新安全状态（如 HTTPS 锁）
- 前进后退历史状态更新
- Web 页面内容开始渲染（此时页面可能还是空白）。

- 🅱️ 渲染阶段 (Rendering)：发生在提交文档后的渲染进程中

- 构建 DOM 树： HTML 解析器将 HTML 字节流转换为 DOM 树 (Document Object Model)。
- 构建 CSSOM 树： CSS 解析器处理 CSS 资源（外部/内联）构建 CSSOM 树 (CSS Object Model)。

1. CSS 会阻塞渲染： 浏览器会阻塞页面的首次渲染直到 CSSOM 构建完成（建议将 CSS 放在 `<head>` 中）。

- 执行 JavaScript:

1. 在构建 DOM 树过程中，遇到 `<script>` 标签（没有 `async`/`defer` 属性）时，会暂停 DOM 树的构建。
2. JS 引擎线程会下载（如果外部 JS）并执行该脚本（期间 GUI 渲染线程被挂起）。
3. JS 可能修改已解析的 DOM 或 CSSOM，因此需要阻塞。
4. JS 会阻塞解析： 建议将非关键 JS 放在底部或用 `async`/`defer` 属性。

- 构建 Render 树 (渲染树 / Layout 树)： 结合 DOM 树 和 CSSOM 树，剔除不可见元素（如 `display: none`, `<head>`），生成描述页面最终可见内容和样式的 Render 树。
- 布局 (Layout / Reflow)： 计算 Render 树中每个节点的精确位置（坐标）和尺寸 (宽、高)。计算相对坐标和绝对坐标。这个过程也称为“回流”。布局是复杂且昂贵的操作。
- 分层 (Layer Creation)： 浏览器根据某些特定规则（如 3D 变换、`position: fixed`、`<video>`、`<canvas>`、`will-change`）将页面划分为多个图层 (Layer)，以便优化性能。
- 绘制 (Paint / Rasterization)： 为每个图层生成一系列的绘制指令列表（记录栅格化的操作），这阶段将 Render 树转换为屏幕上的像素，也称为“光栅化”。该阶段主要进行像素填充（颜色、阴影、渐变、图片绘制等）。
- 合成与显示 (Composite & Display)：

1. 合成： 合成线程将不同图层分割成图块 (tiles)，并将这些图块提交给栅格化线程池。
2. 栅格化： 栅格化线程在 GPU 进程中将图块转换为位图数据并存储在 GPU 内存中。速度非常快（优先栅格化视口附近内容）。
3. 绘制四角描述表： 合成线程收集栅格化完成的图块信息（称为绘制四角描述表 / Draw Quads），描述它们应在屏幕的哪个位置绘制。
4. 最终显示： 合成线程通过 IPC 将这些信息提交给浏览器主进程。浏览器主进程再将其传递给GPU 进程。GPU 进程最终将这些位图信息合成最终的完整图像，并通过显卡硬件显示在屏幕上。

- 最终： 四次挥手断开 TCP 连接（如果 HTTP 头没有指定保持连接 `Connection: keep-alive`）。

## 5. JavaScript 作为单线程语言，如何实现异步操作？核心原理是什么？

- 核心问题： JS 单线程意味着在同一时间只能执行一个任务。如果所有操作（尤其是耗时的如网络请求、定时器）都同步执行，会严重阻塞页面渲染和用户交互（卡死）。
- 解决方案：浏览器提供的多线程环境与协作机制（事件循环）

- 非阻塞异步 API： 浏览器提供了异步 API（如 `setTimeout`, `XMLHttpRequest`, `fetch`, DOM 事件监听等）。当 JS 引擎调用这些 API 时，调用后立即返回（避免阻塞 JS 线程继续执行后续同步代码）。
- 多线程辅助： 浏览器内核为这些异步操作创建对应的辅助线程来完成任务：

1. 浏览器中多个辅助线程：GUI渲染线程、定时器线程、事件触发线程、异步HTTP请求线程、渲染合成线程、GPU线程等。
2. 例如：`setTimeout(callback, 1000)` 调用后，由定时器线程计时 1 秒。
3. 例如：`xhr.send()` 调用后，由异步 HTTP 请求线程处理网络请求。

- 事件队列（任务队列）： 辅助线程完成任务（如计时结束、网络请求返回、用户点击）后，会创建一个事件（如 `timeout`, `readystatechange`, `click`），并将这个事件关联的回调函数通知给事件触发线程，由它添加到对应的任务队列（宏任务队列）中排队。
- 事件循环（Event Loop）： JS 引擎（主线程）不断循环检查两个地方：

1. 执行栈（Call Stack）: 当前是否有同步任务在执行？
2. 任务队列： 是否有待执行的异步任务回调？

- 当前执行栈完全空时，事件循环线程会从任务队列中取出最先进入队列的宏任务（一个）放入执行栈中执行。
- JavaScript 执行该回调函数。

- 如果在该宏任务的执行过程中产生了新的宏任务（如 `setTimeout`）或微任务（如 `Promise.then`），它们会被添加到相应的队列中。
- 当前宏任务执行结束前/后，JS 引擎会清空整个微任务队列（按顺序执行所有微任务）。
- （浏览器可能在此之后进行渲染）。
- 然后进入下一轮循环。如此反复形成 “事件循环”。

- 本质： JS 的单线程（执行栈）负责执行实际的 JS 逻辑，浏览器的多线程负责处理异步操作并将结果（回调函数）通过事件循环机制，在 JS 主线程空闲时安排执行栈执行。

## 6. 沙箱 (Sandbox) 是如何保障渲染进程安全的？

- 沙箱目的： 在渲染进程内部创建一个高度隔离的执行环境，限制渲染进程的能力，防止渲染页面（可能来自不可信来源）的代码对用户操作系统造成安全威胁（如读取文件、安装木马）。
- 实现机制：

- 权限剥离与限制： 沙箱将渲染进程的核心权限极大限制。默认情况下：

1. 无权直接访问本地文件系统： 不能随意读写用户硬盘。
2. 无权直接访问敏感系统资源： 不能访问注册表、其他进程内存等。
3. 无权执行任意系统调用/命令。

- IPC 通信代理：

1. 当渲染进程（页面的 JS）确实需要执行受限制的操作时（例如发送网络请求、保存文件到用户选择的路径、访问 GPU 进行 3D 加速），它不能直接操作。
2. 渲染进程必须通过 IPC (Inter-Process Communication) 向拥有更高权限的浏览器主进程发起请求。
3. 浏览器主进程进行安全策略检查： 主进程接收请求后，会执行严格的安全策略检查（例如同源策略 CORS、文件选择器确认等），决定是否允许该操作。
4. 操作代理与结果返回： 如果允许，浏览器主进程代替渲染进程执行该操作（如进行实际的网络请求或文件保存）。操作完成后，主进程将结果或操作是否成功的状态再次通过 IPC 发送回渲染进程。

- 操作系统级隔离（部分实现）： 现代浏览器沙箱更进一步利用操作系统级别的安全特性（如 Windows 的 Job Objects、Linux 的 Namespaces/Seccomp-bpf、macOS 的 Seatbelt/sandbox-exec），在进程层面施加额外限制，使沙箱逃逸更加困难。

- 总结： 沙箱通过权限限制、代理访问（IPC） 和操作检查，在渲染进程（及其内部的 JS 代码）与用户操作系统/敏感资源之间建立了一道坚固的隔离墙，只允许安全的、可控的操作通过代理通道执行。
