# 其他高频问题

## 前端开发中经常会用到哪些设计模式？说个具体例子

1. **观察者模式（Observer Pattern）**：

- 观察者模式用于实现对象间的一对多依赖关系，当一个对象的状态发生改变时，所有依赖于它的对象都会得到通知并自动更新。
- 实际开发中，可以用来实现事件监听和订阅-发布机制。比如，在前端框架中，Vue 的数据响应式系统就是基于观察者模式实现的。
- 示例：DOM 事件的监听和处理、Redux 中的状态管理等。

2. **单例模式（Singleton Pattern）**：

- 单例模式用于确保一个类只有一个实例，并提供一个全局访问点。
- 实际开发中，可以用来管理全局状态、资源池、配置对象等。比如，管理全局的数据缓存、路由管理器等。
- 示例：全局状态管理库（如 Redux、Vuex）、路由管理器（如 React Router、Vue Router）等。

3. **工厂模式（Factory Pattern）**：

- 工厂模式用于创建对象，但将对象的创建过程封装在工厂类中，客户端无需知道具体的创建细节，只需要通过工厂类来创建对象。
- 实际开发中，可以用来创建具有相似结构的对象，提高代码的灵活性和可维护性。
- 示例：创建不同类型的 UI 组件、创建 XMLHttpRequest 对象等。

4. **适配器模式（Adapter Pattern）**：

- 适配器模式用于解决两个接口不兼容的情况，通过适配器将一个接口转换为另一个接口，使得原本不兼容的类可以协同工作。
- 实际开发中，可以用来兼容不同版本的接口、封装第三方库等。
- 示例：兼容不同浏览器的事件处理、封装不同的数据格式转换器等。

5. **装饰器模式（Decorator Pattern）**：

- 装饰器模式用于在不改变原有对象的结构的情况下，动态地给对象添加额外的功能。
- 实际开发中，可以用来实现插件化的功能、动态扩展对象的功能等。
- 示例：Vue 中的 mixin、React 中的 Higher Order Components（HOC）等。

## 微前端：把“大应用”拆成“可独立演进的小应用”

### 1. 定义

微前端（Micro Frontends）是一种前端架构思想：
把一个大型前端应用拆成多个“小而自治”的子应用，每个子应用都可以独立开发、独立测试、独立部署、独立运行，最后再由主应用统一装配成一个完整产品。

- **主应用**负责“壳子、路由、权限、公共能力”
- **子应用**负责“自己的业务页面和业务逻辑”

当用户访问某个路由时，主应用按规则按需加载对应子应用，把它挂到指定容器里；
子应用之间尽量互不干扰；主应用和子应用可以共享能力，但不能互相污染；
子应用卸载后，能尽量恢复到干净状态；

### 2. 为什么需要微前端

微前端通常是为了解决这些问题：

1. **团队规模大，单体前端协作成本高**
2. **不同业务线迭代节奏不同**，希望独立发布
3. **历史系统重构困难**，希望新旧模块并存
4. **主应用需要统一入口**，但业务模块要尽量独立

所以微前端的价值是：

- 降低大项目耦合
- 提高交付效率
- 支持增量重构
- 让不同团队并行开发

### 3. 它和 SPA、MPA 的区别

- **SPA (Single Page Application)**：一个前端应用，路由切换不刷新页面，体验好，但项目大了之后容易变重、耦合高
- **MPA**：多个页面各自独立，天然隔离，但页面切换要重新请求，体验和协作成本一般
- **微前端**：介于两者之间，既保留前端单页体验，又尽量做到“模块级独立”

### 4. 常见实现方式

- `iframe`
- 路由分发
- Web Components
- JS 动态加载（当前最主流）

JS 动态加载 是最主流的一类方案，而 single-spa 和 qiankun 就是这类方案的典型代表。
**single-spa** 主要负责子应用的路由匹配和生命周期管理，**qiankun** 则是在它基础上增加了样式隔离和 JS 隔离等能力。

## qiankun / single-spa 在干什么

`single-spa` 是微前端的经典方案，核心职责是：

- 根据 URL 匹配子应用
- 管理子应用生命周期
- 在合适的时机执行 `bootstrap`、`mount`、`unmount`

`qiankun` 可以理解成在 `single-spa` 基础上的增强版，它补了两个非常关键的能力：

- **样式隔离**
- **JS 隔离**

这两个能力，基本就是微前端能不能“稳”的关键。

## 为什么微前端一定会遇到隔离问题

多个子应用虽然逻辑上是独立的，但运行时往往共用同一个浏览器环境：

- 同一个 `document`
- 同一个 `window`
- 同一套事件循环
- 同一个页面容器

所以天然会出现：

- CSS 冲突
- 全局变量污染
- 原型链污染
- 弹窗/下拉框/浮层挂载异常
- 子应用卸载后残留状态

因此，微前端真正的难点，不是“加载模块”，而是“**让多个模块在一个页面里安全共存**”。

## 样式隔离：解决 CSS 互相影响的问题

### 1. 样式污染会带来什么问题

如果多个子应用都运行在同一个 `document` 里，它们的 CSS 会直接互相影响，常见现象有：

- A 子应用把 `button` 改成红色，B 子应用也被影响
- 某个子应用写了全局 `body` 样式，整个系统视觉都变了
- 弹窗、下拉框、`modal` 这类挂到 `body` 下的组件样式丢失

所以样式隔离的目标是：

- 子应用样式只作用于自己
- 不污染别的应用
- 尽量不破坏第三方组件的全局能力

### 2. `qiankun` 常见的两种方案

#### 方案一 `shadow dom`

`shadow dom` 是浏览器原生的样式隔离机制。

**优点**

- 天然隔离样式
- 子应用内部样式不会泄漏出去
- 外部样式也很难影响内部

**缺点**

- 隔离太彻底，很多全局能力不好用
- 弹窗、气泡、下拉菜单这类会挂到 `body` 的组件容易失去上下文
- 父应用想统一调整子应用样式会比较困难

#### 方案二 `css prefix` / 作用域前缀

本质上是在子应用样式前面统一加范围标识，让样式只命中自己的容器，比如给选择器加一个类似 `data-qiankun="应用名"` 的作用域。

**优点**

- 轻量
- 兼容性好
- 能解决大多数普通样式冲突

**缺点**

- 不是真正的浏览器级隔离
- 对挂在 `body` 外部的浮层支持一般
- 仍然依赖团队对样式规范的约束

### 3. 为什么很多项目不一定非要用它

现在很多 React / Vue 项目本身已经有比较成熟的局部样式方案，比如：

- `CSS Modules`
- `scoped style`
- `BEM`
- `CSS-in-JS`

这些方案已经能解决大多数“组件内部不互相污染”的问题，并且还能保留全局样式、弹窗挂载、样式传递等能力。

所以在很多真实项目里，**比起额外加一层很重的微前端样式隔离，更重要的是统一样式规范和组件边界**。

## JS 隔离：解决全局环境互相污染的问题

### 1. 为什么需要 JS 隔离

多个子应用共用一个页面时，子应用很容易做这些事：

- 修改 `window.xxx`
- 挂全局变量
- 改原型链
- 绑定全局事件
- 操作 `history`、`location`、`document`

如果不隔离，后加载的子应用可能影响先加载的子应用，卸载后也可能残留副作用。

所以 JS 隔离的核心目标是：

- 子应用运行时不要污染主应用
- 子应用卸载后尽量恢复现场
- 多个子应用切换时状态不要串掉

### 2. `qiankun` 大致怎么做

`qiankun` 会把子应用代码运行在一个相对独立的执行环境里，并通过 `Proxy` 代理 `window`，让子应用“看到”的全局对象和真实 `window` 不完全一样。

这样，子应用中对 `window`、`document`、`location`、`history` 等对象的操作，就可以被拦截、记录、转发或回滚。

你可以把它理解成两层隔离：

1. **作用域隔离**：子应用代码运行在单独的函数/模块作用域里
2. **全局对象隔离**：通过代理 `window`，控制对全局环境的读写

## 沙箱机制：JS 隔离的核心实现

`qiankun` 里常见的沙箱思路可以分成三类：

- 快照沙箱
- 单应用代理沙箱
- 多应用代理沙箱

### 1. 快照沙箱：先拍照，再恢复

快照沙箱的思路很直观：

- 子应用激活时，先记录一份当前 `window` 状态
- 子应用运行期间，如果修改了 `window` 就记录变化
- 子应用失活时，把 `window` 恢复到快照状态

**优点**

- 思路简单，容易理解
- 能做基础的状态恢复

**缺点**

- 需要遍历大量 `window` 属性，性能开销大
- 更适合单实例场景
- 多个子应用同时存在时容易混乱
- 仍然可能留下全局污染

### 2. 单应用代理沙箱：只管一个子应用

单应用代理沙箱的思路是：不每次都扫描整个 `window`，而是专门记录这个子应用改过什么。

通常会记录：

- 哪些属性被改过
- 原始值是什么
- 哪些属性是新加的

这样在子应用卸载时，可以更快回滚这些改动。

**优点**

- 比快照沙箱快
- 不需要频繁扫描整个 `window`
- 适合“同一时刻只运行一个子应用”的场景

**缺点**

- 仍然在改真实 `window`
- 还是会有全局污染风险
- 多子应用并存时容易相互影响

### 3. 多应用代理沙箱：每个子应用都有自己的“虚拟全局”

多应用代理沙箱更适合真正的微前端场景。

它的核心思想是：

- 不让子应用直接操作真实 `window`
- 给每个子应用提供一个独立的代理环境
- 子应用读到的是自己的“虚拟全局对象”
- 写入时只影响当前子应用的代理层

这样可以做到：

- 子应用之间互不污染
- 多个子应用可以同时存在
- 路由切换更安全
- 更符合“多实例运行”的微前端需求

### 4. 一句话记忆

- **快照沙箱**：先拍照，再恢复
- **单应用代理沙箱**：只记录一个子应用改了什么
- **多应用代理沙箱**：每个子应用都有自己的代理隔离层

## iframe vs 微前端：面试很爱问

### iframe 的优点

- 天然隔离，CSS 和 JS 几乎完全隔开
- 实现简单
- 适合强隔离场景

### iframe 的问题

- 路由、通信、性能和体验都比较差
- 弹窗、滚动、适配、登录态共享等问题很多
- 页面像“嵌套网页”，整体体验不够统一

### 微前端的优势

- 比 iframe 更接近原生应用体验
- 可以共享路由、登录态、公共组件、埋点能力
- 兼顾独立开发和统一体验

### 简单结论

- **强隔离、弱交互**：iframe
- **弱隔离、强体验、强协作**：微前端

## qiankun 和 Module Federation 有什么区别

这个问题也很常见，建议这么理解：

### 1. `qiankun`

更偏向“**微前端框架**”。

它关注的是：

- 子应用怎么挂载和卸载
- 路由怎么切换
- 样式怎么隔离
- JS 怎么隔离

核心是“应用级运行时装配”。

### 2. Module Federation

更偏向“**模块共享机制**”。

它允许多个前端应用在运行时共享代码和依赖，动态加载彼此的模块。

核心是：

- 远程加载模块
- 共享依赖
- 去中心化

### 3. 怎么选

- 如果你更关心“子应用整页接入、生命周期、隔离”，更像 `qiankun`
- 如果你更关心“跨项目共享组件、工具、业务模块”，更像 `Module Federation`

它们可以共存，但解决的问题重点不同。

## 面试时可以怎么答

### 1. 微前端是什么

> 微前端是一种把大型前端应用拆成多个独立子应用的架构方案。主应用负责路由分发和容器装配，子应用负责自己的业务逻辑和页面。为了让多个子应用能在同一页面安全共存，还需要做样式隔离和 JS 隔离。

### 2. 为什么要做 JS 隔离

> 因为多个子应用共用一个浏览器环境，如果直接操作 `window`、全局变量或原型链，很容易互相污染。JS 隔离就是通过沙箱机制控制全局读写，尽量保证子应用运行和卸载前后环境一致。

### 3. 为什么样式隔离不是银弹

> 因为很多组件会把弹窗、下拉框、气泡等挂到 `body` 上，而 `shadow dom` 或前缀隔离都可能让这些浮层拿不到正确样式，所以样式隔离要结合组件规范和团队约束一起做。

### 4. 微前端适合什么场景

> 适合大型团队、多业务线协作、历史系统重构、多个子团队独立发布的场景；如果只是一个中小型单体项目，强行上微前端反而会增加复杂度。

## 还可以补充的相关问题

### 1. 微前端适合什么，不适合什么

- 适合：多团队协作、历史包袱重、需要增量重构、需要独立发布
- 不适合：小项目、团队很小、业务简单、交互耦合特别高的场景

### 2. 微前端的常见坑

- 依赖重复加载
- 全局事件没清理
- 弹窗、浮层、Portal 挂载异常
- 路由冲突
- 登录态和权限共享复杂
- 子应用通信设计不清晰

### 3. 子应用之间怎么通信

常见方式有：

- `props` 传参
- 全局状态管理
- `CustomEvent`
- `postMessage`
- `localStorage` / `BroadcastChannel`

### 4. 微前端通信设计要注意什么

- 少用强耦合的直接调用
- 尽量设计稳定的通信协议
- 公共能力放主应用或公共层
- 通信要可追踪、可回收、可测试

### 5. 微前端和组件化的区别

- 组件化解决的是“代码复用”和“局部拆分”
- 微前端解决的是“应用级拆分”和“组织级协作”

也就是说，组件化是代码层面的拆分，微前端是系统层面的拆分。

### 6. 微前端一定要用 `qiankun` 吗

不一定。`qiankun` 只是比较成熟的一种实现。实际项目里也可以用 `single-spa`、`Module Federation`、`iframe` 或者自研方案，关键是看场景和约束。

---

## 一句话总结

微前端的本质，不是简单把页面切碎，而是让多个子应用在一个页面里“**可独立演进、可协同运行、可安全隔离**”。

真正的难点在于：

- 路由装配
- 样式隔离
- JS 隔离
- 通信设计
- 生命周期治理

所以面试里如果要讲透，最好不要只说“把页面拆开”，而是要能讲出：**为什么要拆、怎么拆、怎么隔离、怎么通信、适合什么场景**。

## 两个常见场景题

### 页面 1 与页面 2 如何同步播放歌曲

可以优先回答 `postMessage`、`BroadcastChannel`、`localStorage + storage` 事件，核心是建立跨页面状态同步机制。

### 分页请求乱序怎么解决

核心思路是“只认最后一次请求”：

- 为每次请求生成递增 requestId，只处理最新响应
- 或者使用 `AbortController` 取消旧请求
- 在 React / Vue 中结合状态更新时机，避免过期响应覆盖新数据

## 2 个页面之间通信的方法

1. URL 参数传递：可以通过 URL 的查询字符串（query string）传递参数，例如 [http://example.com/page2?param1=value1&param2=value2。在第一个页面中可以通过链接或](http://example.com/page2?param1=value1&param2=value2。在第一个页面中可以通过链接或) JavaScript 跳转到第二个页面，并在链接或 JavaScript 中添加查询参数。在第二个页面中可以通过解析 URL 的查询参数来获取参数值。
2. Local Storage：可以使用 HTML5 的 Local Storage API 在浏览器中存储一些数据，这些数据可以被同一域名下的其它页面访问到。在第一个页面中可以将数据存储到 Local Storage 中，然后在第二个页面中读取该数据。
3. Session Storage：与 Local Storage 类似，Session Storage 也是 HTML5 提供的 API，不同的是 Session Storage 中保存的数据只能在同一个会话（session）中共享，如果关闭浏览器窗口或标签页，数据就会被清除。在第一个页面中可以将数据存储到 Session Storage 中，然后在第二个页面中读取该数据。
4. Cookies：可以使用 Cookies 在浏览器中存储一些数据，这些数据可以被同一域名下的其它页面访问到。在第一个页面中可以设置一个 Cookie，然后在第二个页面中读取该 Cookie。
5. postMessage API：可以使用 postMessage API 在两个页面之间进行跨域通信。在第一个页面中可以调用 window.postMessage() 方法发送一条消息，然后在第二个页面中通过监听 message 事件来接收该消息。

## SPA 是什么

SPA（Single Page Application，单页应用）是一种应用程序或网站的设计模式，其特点是在单个 HTML 页面上加载所有必要的代码（如 HTML、CSS、JavaScript），并根据用户与应用的交互动态更新页面内容。相对于传统的多页面应用，SPA 通过减少页面加载次数和服务器请求，提高了用户体验和应用性能。

SPA 的优点包括：

- 更快的页面加载速度：只需加载一次 HTML、CSS 和 JavaScript，后续页面跳转时只需要通过 AJAX 请求获取数据并更新页面内容。
- 更好的用户体验：页面间切换无需重新加载，提供了类似于桌面应用的体验。
- 减轻服务器负担：客户端负责渲染页面，服务器只需提供数据。

SPA 的缺点包括：

- 首次加载时间较长：需要加载整个应用的代码和资源。
- 不利于 SEO：由于 SPA 的内容是动态生成的，搜索引擎可能无法正确索引页面内容。
- 框架依赖：通常需要使用前端框架（如 Angular、React 或 Vue）来实现 SPA。

## SPA 首屏优化方式

1. 代码分割：将SPA应用程序的代码分割成多个小块，在需要时动态加载。这样可以将加载时间分散到多个时间点，减少首屏加载时间。
2. 懒加载：仅在需要时动态加载组件和资源，减少初始加载量和首屏加载时间。
3. 预加载：提前加载下一个页面所需的资源，减少页面切换时的加载时间，提高用户体验。
4. 使用CDN：将静态资源（例如JavaScript、CSS、图片等）放到CDN上，可以减少网络延迟和带宽占用，提高资源加载速度。
5. SSR（Server Side Rendering）：使用服务端渲染技术，在服务器端生成HTML页面，减少客户端的加载时间，并提高SEO效果。
6. 静态化页面：将SPA应用程序的某些页面转化为静态页面，通过CDN进行分发，可以减少服务器的压力和流量消耗，提高页面加载速度。

总之，SPA的首屏优化是一个复杂的问题，需要根据应用程序的实际情况和场景选择合适的优化方式。需要综合考虑网络延迟、资源大小、缓存策略、渲染性能等多个因素，以达到最佳的用户体验和性能表现。

- 减小入口文件体积
- 常用的手段是路由懒加载，把不同路由对应的组件分割成不同的代码块，待路由被请求的时候会单独打包路由，使得入口文件变小，加载速度大大增加
- 静态资源本地缓存
- 后端返回资源问题：
- 采用HTTP缓存，设置Cache-Control，Last-Modified，Etag等响应头

- 图片资源压缩
- 开启Gzip压缩
- 使用SSR

## 浏览器引擎的种类

浏览器引擎（也称为浏览器内核）负责解析网页内容并呈现给用户。以下是一些主要的浏览器引擎：

WebKit： WebKit 是一个开源的浏览器引擎，由苹果公司开发。它是 Safari 浏览器的内核，同时也被很多其他浏览器所采用。

Blink： Blink 是一个由谷歌公司开发的开源浏览器引擎，它是 Chrome、Edge 和 Opera 等浏览器的内核。Blink 是从 WebKit 分离出来的，因此与 WebKit 有很多相似之处。

Gecko： Gecko 是 Mozilla 基金会开发的开源浏览器引擎，它是 Firefox 浏览器的内核。

Trident： Trident 是微软开发的浏览器引擎，它曾经是 Internet Explorer 的内核。在微软推出基于 Blink 内核的新版 Edge 浏览器后，Trident 的使用逐渐减少。

## 静态资源怎么存储

静态资源（如 HTML、CSS、JavaScript、图片等）是不需要服务器动态生成的文件。静态资源存储的方式有很多种，以下是一些常见的方法：

文件系统： 静态资源可以直接存储在服务器的文件系统中。当服务器收到请求时，它会根据请求的 URL 查找对应的静态文件并返回给客户端。

CDN： CDN（Content Delivery Network，内容分发网络）是一种分布式网络，它可以将静态资源缓存到离用户较近的服务器上。这样，当用户请求静态资源时，CDN 会从离用户最近的服务器上提供资源，从而减少延迟和提高加载速度。

对象存储： 对象存储是一种云存储服务，它允许开发者将静态资源上传到云端，并通过 URL 访问这些资源。对象存储的优点包括高可用性、可扩展性和低成本。常见的对象存储服务有 Amazon S3、Google Cloud Storage 和 Microsoft Azure Blob Storage 等。

缓存： 缓存是一种存储技术，它可以将常用的静态资源存储在客户端（如浏览器缓存）或者服务器端（如代理服务器缓存、Redis 等）。通过使用缓存，静态资源的加载速度可以得到显著提升，同时还可以减轻服务器的负担。

## 前端埋点的实现

前端埋点是指在前端页面中加入用于监测用户行为的代码，这些代码可以收集用户的操作数据并发送到服务器端。前端埋点是一种常用的用户行为分析方法，可以帮助我们了解用户在使用网站或应用时的真实行为。

前端埋点的实现方式有很多种，常见的方法包括：

- 在页面中加入 JavaScript 代码来监测用户行为，并使用 XMLHttpRequest 或 fetch API 发送数据到服务器。
- 使用第三方的前端埋点工具，例如 Google Analytics、Mixpanel 等。这些工具通常会提供简单的 API，可以让开发人员轻松地在页面中添加埋点代码。
- 使用前端框架（如 React、Vue 等）的插件或扩展来实现埋点功能。这些插件或扩展通常会提供简单的 API，让开发人员能够轻松地在应用中添加埋点功能。

在 React 中实现前端埋点有多种方法，你可以根据自己的需要选择合适的方法。

一种常见的方法是使用 React 的生命周期函数或自定义钩子函数来监测用户行为，并使用 XMLHttpRequest 或 fetch API 发送数据到服务器。例如，你可以在 componentDidMount 生命周期函数中添加代码来监测组件挂载事件，并在组件卸载时使用 componentWillUnmount 生命周期函数来发送数据。

另一种方法是使用第三方的前端埋点工具，例如 Google Analytics、Mixpanel 等。这些工具通常会提供简单的 API，让开发人员能够轻松地在 React 应用中添加埋点功能。

你也可以使用 React 插件或扩展来实现埋点功能。这些插件或扩展通常会提供简单的 API，让开发人员能够轻松地在应用中添加埋点功能。

#### 果金融系统使用定时器每隔一秒钟用 Promise 发送请求，可能会有以下几个问题：

1. 负载增加：由于定时器每隔一秒钟发送请求，如果有大量请求同时发出，可能会导致服务器负载过高，影响系统的稳定性和性能。
2. 安全风险：可能会有攻击者利用这种方式进行 DDoS 攻击，导致服务器资源耗尽，无法正常提供服务。
3. 网络带宽消耗：如果每隔一秒钟发送大量请求，可能会消耗大量的网络带宽，导致网络性能下降。

为了避免这些问题，应该谨慎地考虑使用定时器发送请求的频率，避免发送过多的请求。
