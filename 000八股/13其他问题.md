# 其他高频问题

## 前端开发中经常会用到哪些设计模式？说个具体例子

### 1. 观察者模式（Observer Pattern）

- 用于实现对象间的一对多依赖关系，当一个对象状态变化时，所有依赖它的对象都会得到通知并自动更新。
- 实际开发中常用于事件监听和订阅发布机制。
- 例如：DOM 事件监听、Vue 的响应式系统、Redux 的状态订阅等。

### 2. 单例模式（Singleton Pattern）

- 用于确保一个类只有一个实例，并提供全局访问点。
- 常用于管理全局状态、资源池、配置对象等。
- 例如：Redux、Vuex、路由管理器等。

### 3. 工厂模式（Factory Pattern）

- 用于创建对象，但把创建过程封装起来，客户端不需要关心具体细节。
- 常用于创建结构相似但类型不同的对象。
- 例如：创建不同类型的 UI 组件、封装 XMLHttpRequest 创建过程等。

### 4. 适配器模式（Adapter Pattern）

- 用于解决两个接口不兼容的问题，通过适配器将一个接口转换成另一个接口。
- 常用于兼容不同版本接口、封装第三方库。
- 例如：兼容不同浏览器事件处理、数据格式转换等。

### 5. 装饰器模式（Decorator Pattern）

- 用于在不改变原有对象结构的情况下，动态地给对象添加功能。
- 常用于插件化扩展、增强对象能力。
- 例如：Vue 中的 mixin、React 中的 HOC 等。

## 微前端：把“大应用”拆成“可独立演进的小应用”

### 1. 定义

微前端（Micro Frontends）是一种前端架构思想：
把一个大型前端应用拆成多个“小而自治”的子应用，每个子应用都可以独立开发、独立测试、独立部署、独立运行，最后由主应用统一装配成一个完整产品。

- **主应用**负责：壳子、路由、权限、公共能力
- **子应用**负责：自己的业务页面和业务逻辑

用户访问某个路由时，主应用按规则按需加载对应子应用，并把它挂到指定容器里。
子应用之间尽量互不干扰，主应用和子应用可以共享能力，但不能互相污染。
子应用卸载后，要尽量恢复到干净状态。

### 2. 为什么需要微前端

微前端通常用于解决以下问题：

1. 团队规模大，单体前端协作成本高
2. 不同业务线迭代节奏不同，希望独立发布
3. 历史系统重构困难，希望新旧模块并存
4. 主应用需要统一入口，但业务模块要尽量独立

微前端的价值在于：

- 降低大项目耦合
- 提高交付效率
- 支持增量重构
- 让不同团队并行开发

### 3. 它和 SPA、MPA 的区别

- **SPA（Single Page Application）**：一个前端应用，路由切换不刷新页面，体验好，但项目大了之后容易变重、耦合高。
- **MPA**：多个页面各自独立，天然隔离，但页面切换要重新请求，体验和协作成本一般。
- **微前端**：介于两者之间，既保留前端单页体验，又尽量做到模块级独立。

### 4. 常见实现方式

- `iframe`
- 路由分发
- Web Components
- JS 动态加载（当前最主流）

JS 动态加载是最主流的一类方案，`single-spa` 和 `qiankun` 都是典型代表。

## qiankun / single-spa 在干什么

`single-spa` 是微前端经典方案，核心职责是：

- 根据 URL 匹配子应用
- 管理子应用生命周期
- 在合适的时机执行 `bootstrap`、`mount`、`unmount`

`qiankun` 可以理解成在 `single-spa` 基础上的增强版，它补充了两个非常关键的能力：

- 样式隔离
- JS 隔离

这两个能力，基本就是微前端能不能“稳”的关键。

---

## 为什么微前端一定会遇到隔离问题

多个子应用虽然逻辑上是独立的，但运行时往往共用同一个浏览器环境：

- 同一个 `document`
- 同一个 `window`
- 同一套事件循环
- 同一个页面容器

所以天然会出现：

- CSS 冲突
- 全局变量污染
- 原型链污染
- 弹窗、下拉框、浮层挂载异常
- 子应用卸载后残留状态

因此，微前端真正的难点不是“加载模块”，而是“让多个模块在一个页面里安全共存”。

---

## 样式隔离：解决 CSS 互相影响的问题

### 1. 样式污染会带来什么问题

如果多个子应用都运行在同一个 `document` 里，它们的 CSS 会直接互相影响，常见现象有：

- A 子应用把 `button` 改成红色，B 子应用也被影响
- 某个子应用写了全局 `body` 样式，整个系统视觉都变了
- 弹窗、下拉框、`modal` 这类挂到 `body` 下的组件样式丢失

所以样式隔离的目标是：

- 子应用样式只作用于自己
- 不污染别的应用
- 尽量不破坏第三方组件的全局能力

### 2. `qiankun` 常见的两种方案

#### 方案一：`shadow dom`

`shadow dom` 是浏览器原生的样式隔离机制。

**优点**

- 天然隔离样式
- 子应用内部样式不会泄漏出去
- 外部样式也很难影响内部

**缺点**

- 隔离太彻底，很多全局能力不好用
- 弹窗、气泡、下拉菜单这类会挂到 `body` 的组件容易失去上下文
- 父应用想统一调整子应用样式会比较困难

#### 方案二：`css prefix` / 作用域前缀

本质上是在子应用样式前面统一加范围标识，让样式只命中自己的容器，比如给选择器加一个类似 `data-qiankun="应用名"` 的作用域。

**优点**

- 轻量
- 兼容性好
- 能解决大多数普通样式冲突

**缺点**

- 不是真正的浏览器级隔离
- 对挂在 `body` 外部的浮层支持一般
- 仍然依赖团队对样式规范的约束

### 3. 为什么很多项目不一定非要用它

现在很多 React / Vue 项目本身已经有比较成熟的局部样式方案，比如：

- `CSS Modules`
- `scoped style`
- `BEM`
- `CSS-in-JS`

这些方案已经能解决大多数“组件内部不互相污染”的问题，并且还能保留全局样式、弹窗挂载、样式传递等能力。

所以在很多真实项目里，比起额外加一层很重的微前端样式隔离，更重要的是统一样式规范和组件边界。

## JS 隔离：解决全局环境互相污染的问题

### 1. 为什么需要 JS 隔离

多个子应用共用一个页面时，子应用很容易做这些事：

- 修改 `window.xxx`
- 挂全局变量
- 改原型链
- 绑定全局事件
- 操作 `history`、`location`、`document`

如果不隔离，后加载的子应用可能影响先加载的子应用，卸载后也可能残留副作用。

所以 JS 隔离的核心目标是：

- 子应用运行时不要污染主应用
- 子应用卸载后尽量恢复现场
- 多个子应用切换时状态不要串掉

### 2. `qiankun` 大致怎么做

`qiankun` 会把子应用代码运行在一个相对独立的执行环境里，并通过 `Proxy` 代理 `window`，让子应用“看到”的全局对象和真实 `window` 不完全一样。

这样，子应用中对 `window`、`document`、`location`、`history` 等对象的操作，就可以被拦截、记录、转发或回滚。

可以把它理解成两层隔离：

1. 作用域隔离：子应用代码运行在单独的函数 / 模块作用域里
2. 全局对象隔离：通过代理 `window`，控制对全局环境的读写

---

## 沙箱机制：JS 隔离的核心实现

`qiankun` 里常见的沙箱思路可以分成三类：

- 快照沙箱
- 单应用代理沙箱
- 多应用代理沙箱

### 1. 快照沙箱：先拍照，再恢复

快照沙箱的思路很直观：

- 子应用激活时，先记录一份当前 `window` 状态
- 子应用运行期间，如果修改了 `window` 就记录变化
- 子应用失活时，把 `window` 恢复到快照状态

**优点**

- 思路简单，容易理解
- 能做基础的状态恢复

**缺点**

- 需要遍历大量 `window` 属性，性能开销大
- 更适合单实例场景
- 多个子应用同时存在时容易混乱
- 仍然可能留下全局污染

### 2. 单应用代理沙箱：只管一个子应用

单应用代理沙箱的思路是：不每次都扫描整个 `window`，而是专门记录这个子应用改过什么。

通常会记录：

- 哪些属性被改过
- 原始值是什么
- 哪些属性是新加的

这样在子应用卸载时，可以更快回滚这些改动。

**优点**

- 比快照沙箱快
- 不需要频繁扫描整个 `window`
- 适合“同一时刻只运行一个子应用”的场景

**缺点**

- 仍然在改真实 `window`
- 还是会有全局污染风险
- 多子应用并存时容易相互影响

### 3. 多应用代理沙箱：每个子应用都有自己的“虚拟全局”

多应用代理沙箱更适合真正的微前端场景。

它的核心思想是：

- 不让子应用直接操作真实 `window`
- 给每个子应用提供一个独立的代理环境
- 子应用读到的是自己的“虚拟全局对象”
- 写入时只影响当前子应用的代理层

这样可以做到：

- 子应用之间互不污染
- 多个子应用可以同时存在
- 路由切换更安全
- 更符合“多实例运行”的微前端需求

### 4. 一句话记忆

- **快照沙箱**：先拍照，再恢复
- **单应用代理沙箱**：只记录一个子应用改了什么
- **多应用代理沙箱**：每个子应用都有自己的代理隔离层

## iframe vs 微前端：面试很爱问

### iframe 的优点

- 天然隔离，CSS 和 JS 几乎完全隔开
- 实现简单
- 适合强隔离场景

### iframe 的问题

- 路由、通信、性能和体验都比较差
- 弹窗、滚动、适配、登录态共享等问题很多
- 页面像“嵌套网页”，整体体验不够统一

### 微前端的优势

- 比 iframe 更接近原生应用体验
- 可以共享路由、登录态、公共组件、埋点能力
- 兼顾独立开发和统一体验

### 简单结论

- **强隔离、弱交互**：`iframe`
- **弱隔离、强体验、强协作**：微前端

## qiankun 和 Module Federation 有什么区别

这个问题也很常见，建议这样理解：

### 1. `qiankun`

更偏向“微前端框架”。

它关注的是：

- 子应用怎么挂载和卸载
- 路由怎么切换
- 样式怎么隔离
- JS 怎么隔离

核心是“应用级运行时装配”。

### 2. Module Federation

更偏向“模块共享机制”。

它允许多个前端应用在运行时共享代码和依赖，动态加载彼此的模块。

核心是：

- 远程加载模块
- 共享依赖
- 去中心化

### 3. 怎么选

- 如果更关心“子应用整页接入、生命周期、隔离”，更像 `qiankun`
- 如果更关心“跨项目共享组件、工具、业务模块”，更像 `Module Federation`

它们可以共存，但解决的问题重点不同。

## 面试时可以怎么答

### 1. 微前端是什么

> 微前端是一种把大型前端应用拆成多个独立子应用的架构方案。主应用负责路由分发和容器装配，子应用负责自己的业务逻辑和页面。为了让多个子应用能在同一页面安全共存，还需要做样式隔离和 JS 隔离。

### 2. 为什么要做 JS 隔离

> 因为多个子应用共用一个浏览器环境，如果直接操作 `window`、全局变量或原型链，很容易互相污染。JS 隔离就是通过沙箱机制控制全局读写，尽量保证子应用运行和卸载前后环境一致。

### 3. 为什么样式隔离不是银弹

> 因为很多组件会把弹窗、下拉框、气泡等挂到 `body` 上，而 `shadow dom` 或前缀隔离都可能让这些浮层拿不到正确样式，所以样式隔离要结合组件规范和团队约束一起做。

### 4. 微前端适合什么场景

> 适合大型团队、多业务线协作、历史系统重构、多个子团队独立发布的场景；如果只是一个中小型单体项目，强行上微前端反而会增加复杂度。

---

## 还可以补充的相关问题

### 1. 微前端适合什么，不适合什么

- 适合：多团队协作、历史包袱重、需要增量重构、需要独立发布
- 不适合：小项目、团队很小、业务简单、交互耦合特别高的场景

### 2. 微前端的常见坑

- 依赖重复加载
- 全局事件没清理
- 弹窗、浮层、Portal 挂载异常
- 路由冲突
- 登录态和权限共享复杂
- 子应用通信设计不清晰

### 3. 子应用之间怎么通信

常见方式有：

- `props` 传参
- 全局状态管理
- `CustomEvent`
- `postMessage`
- `localStorage` / `BroadcastChannel`

### 4. 微前端通信设计要注意什么

- 少用强耦合的直接调用
- 尽量设计稳定的通信协议
- 公共能力放主应用或公共层
- 通信要可追踪、可回收、可测试

### 5. 微前端和组件化的区别

- 组件化解决的是代码复用和局部拆分
- 微前端解决的是应用级拆分和组织级协作

也就是说，组件化是代码层面的拆分，微前端是系统层面的拆分。

### 6. 微前端一定要用 `qiankun` 吗

不一定。`qiankun` 只是比较成熟的一种实现。实际项目里也可以用 `single-spa`、`Module Federation`、`iframe` 或自研方案，关键是看场景和约束。

---

## 一句话总结

微前端的本质，不是简单把页面切碎，而是让多个子应用在一个页面里“可独立演进、可协同运行、可安全隔离”。

真正的难点在于：

- 路由装配
- 样式隔离
- JS 隔离
- 通信设计
- 生命周期治理

所以面试里如果要讲透，最好不要只说“把页面拆开”，而是要能讲出：为什么要拆、怎么拆、怎么隔离、怎么通信、适合什么场景。

## 模块联邦是 Webpack 在 v5 提供的核心能力

模块联邦允许多个前端应用在运行时共享代码，并且可以动态加载彼此的模块。
模块联邦的本质是：运行时模块加载 + 共享依赖协商机制。

### 核心能力

- 远程加载模块（Remote Module）
- 共享依赖（Shared Dependencies）
- 去中心化（无需统一构建）

## 两个常见场景题

### 1. 页面 1 与页面 2 如何同步播放歌曲

可以优先回答：`postMessage`、`BroadcastChannel`、`localStorage + storage` 事件。
核心是建立跨页面状态同步机制。

### 2. 分页请求乱序怎么解决

核心思路是“只认最后一次请求”：

- 为每次请求生成递增 `requestId`，只处理最新响应
- 或者使用 `AbortController` 取消旧请求
- 在 React / Vue 中结合状态更新时机，避免过期响应覆盖新数据

---

## 2 个页面之间通信的方法

1. **URL 参数传递**：通过 URL 查询字符串传递参数，例如 `http://example.com/page2?param1=value1&param2=value2`。第一个页面可以通过链接或 JavaScript 跳转到第二个页面，并在链接或 JavaScript 中添加查询参数；第二个页面可以解析 URL 查询参数来获取参数值。
2. **Local Storage**：使用 HTML5 的 Local Storage API 在浏览器中存储数据，同一域名下的其他页面可以访问。
3. **Session Storage**：与 Local Storage 类似，但只能在同一个会话中共享，关闭浏览器窗口或标签页后数据会被清除。
4. **Cookies**：使用 Cookies 在浏览器中存储数据，同一域名下的其他页面可以访问。
5. **postMessage API**：用于两个页面之间的跨域通信，一个页面通过 `window.postMessage()` 发送消息，另一个页面监听 `message` 事件接收。

## 聊重构收益

迁移后项目整体功能应与迁移前保持一致，但依赖的组件包、打包模式、内部逻辑的部分写法等会进行升级重构。

新项目将不会使用 kfx 静态部署，因为 kfx 不支持 SSR，暂时还使用老的部署流水线，后续升级为 SSR 流水线。即使在老的流水线下，Webpack 迁移至 Vite 后，流水线整体时间也降低了 **50%**。

## SPA 是什么

SPA（Single Page Application，单页应用）是一种应用程序或网站设计模式：在单个 HTML 页面上加载所有必要的代码（如 HTML、CSS、JavaScript），并根据用户与应用的交互动态更新页面内容。

相较于传统多页面应用，SPA 通过减少页面加载次数和服务器请求，提高了用户体验和应用性能。

### SPA 的优点

- 更快的页面加载速度：只需加载一次 HTML、CSS 和 JavaScript，后续页面跳转时只需要通过 AJAX 请求获取数据并更新页面内容。
- 更好的用户体验：页面间切换无需重新加载，提供类似桌面应用的体验。
- 减轻服务器负担：客户端负责渲染页面，服务器只需提供数据。

### SPA 的缺点

- 首次加载时间较长：需要加载整个应用的代码和资源。
- 不利于 SEO：内容是动态生成的，搜索引擎可能无法正确索引页面内容。
- 框架依赖：通常需要使用前端框架（如 Angular、React 或 Vue）来实现 SPA。

---

## SPA 首屏优化方式

1. 代码分割：将 SPA 应用程序的代码分割成多个小块，在需要时动态加载，减少首屏加载时间。
2. 懒加载：仅在需要时动态加载组件和资源，减少初始加载量。
3. 预加载：提前加载下一个页面所需资源，减少切换时的等待时间。
4. 使用 CDN：将 JavaScript、CSS、图片等静态资源放到 CDN 上，减少网络延迟和带宽占用。
5. SSR（Server Side Rendering）：在服务器端生成 HTML 页面，减少客户端加载时间，并提升 SEO 效果。
6. 静态化页面：将部分页面转化为静态页面，通过 CDN 分发，减少服务器压力和流量消耗。

### 进一步优化手段

- 减小入口文件体积
- 路由懒加载
- 静态资源本地缓存
- 使用 HTTP 缓存：`Cache-Control`、`Last-Modified`、`ETag`
- 前端合理利用 `localStorage`
- UI 框架按需加载
- 避免组件重复打包
- 调整 CommonsChunkPlugin 配置，如 `minChunks: 3`
- 图片资源压缩
- 开启 Gzip 压缩
- 使用 SSR

## 浏览器引擎的种类

浏览器引擎（也称为浏览器内核）负责解析网页内容并呈现给用户。常见的浏览器引擎包括：

- **WebKit**：由苹果公司开发，是 Safari 浏览器的内核，也被很多其他浏览器采用。
- **Blink**：由谷歌公司开发，是 Chrome、Edge 和 Opera 等浏览器的内核。Blink 是从 WebKit 分离出来的。
- **Gecko**：由 Mozilla 基金会开发，是 Firefox 浏览器的内核。
- **Trident**：由微软开发，曾经是 Internet Explorer 的内核。新版 Edge 基于 Blink 后，Trident 的使用逐渐减少。

## 静态资源怎么存储

静态资源（如 HTML、CSS、JavaScript、图片等）是不需要服务器动态生成的文件。常见存储方式有：

- **文件系统**：直接存储在服务器文件系统中，请求到来时根据 URL 查找并返回对应静态文件。
- **CDN**：将静态资源缓存到离用户更近的节点，减少延迟并提升加载速度。
- **对象存储**：如 Amazon S3、Google Cloud Storage、Microsoft Azure Blob Storage 等，具有高可用性、可扩展性和低成本。
- **缓存**：如浏览器缓存、代理服务器缓存、Redis 等，用于提高加载速度并减轻服务器压力。

## 服务稳定性与降级方案

之前使用 CSR 加载页面的方式，资源加载方式比较简单，服务挂载到 Nginx 上就可以运行，基本无需担心服务稳定性。

切换到服务端渲染后，新增了服务端 Node 层，因此服务端稳定性需要重点关注，需要通过以下方式确保服务可用性，并具备降级为 CSR 渲染的能力。

### Node 服务挂掉时，整体降级

- 在 `bpm-approval-redirect` 项目内，通过 Kconf 配置降级为 CSR 渲染
- Nginx 层处理降级

### 单次请求页面异常

- Next.js 框架处理：当服务端渲染发生异常报错时（包括接口异常、逻辑异常等），页面会直接停止服务端渲染，切换为客户端渲染

## 前端埋点的实现

前端埋点是指在前端页面中加入监测用户行为的代码，这些代码可以收集用户的操作数据并发送到服务器端。

### 常见实现方式

- 在页面中加入 JavaScript 代码来监测用户行为，并使用 `XMLHttpRequest` 或 `fetch` API 发送数据到服务器。
- 使用第三方前端埋点工具，例如 Google Analytics、Mixpanel 等。
- 使用前端框架（如 React、Vue 等）的插件或扩展来实现埋点功能。

### 在 React 中实现埋点

常见做法包括：

- 使用生命周期函数或自定义 Hook 监测用户行为，并通过 `XMLHttpRequest` 或 `fetch` 发送数据。
- 使用第三方埋点工具，如 Google Analytics、Mixpanel。
- 使用 React 插件或扩展提供的 API 实现埋点。

### 定时器每隔一秒用 Promise 发送请求的风险

如果金融系统使用定时器每隔一秒钟用 Promise 发送请求，可能会有以下问题：

1. 负载增加：大量请求同时发出，可能导致服务器负载过高，影响稳定性和性能。
2. 安全风险：可能被攻击者利用进行 DDoS 攻击，导致服务器资源耗尽。
3. 网络带宽消耗：频繁发送大量请求会消耗带宽，导致网络性能下降。

因此，应该谨慎控制请求频率，避免发送过多请求。

## INP 是什么？

INP（Interaction to Next Paint）是衡量页面交互响应体验的性能指标，主要关注用户交互后，到下一次页面绘制完成之间的延迟。

### 怎么统计的？

通常会记录用户一次交互之后，页面响应并完成绘制所需的时间，并在多个交互中取代表值（通常关注较差表现）。

### 为什么会影响交互体验？

因为它衡量的是“我点了之后，页面多久有反应”，直接关系到用户是否觉得卡顿。

现在大厂越来越关注真实用户体验指标，包括：

- LCP
- CLS
- INP
- FID

## Lighthouse Performance 的理解

Lighthouse Performance 本质上是围绕用户加载体验构建的一套指标体系，从“是否看到内容”到“是否可以交互”，再到“是否流畅稳定”，进行全链路衡量。

FCP 和 LCP 解决的是内容呈现问题，SI 描述渲染过程的平滑性，TTI 和 TBT 衡量主线程是否被阻塞并影响交互，CLS 则关注页面布局稳定性。

在工程实践中，优化路径可以抽象为三件事：

1. 优化关键渲染路径
2. 降低 JavaScript 主线程负载
3. 保证布局稳定性
