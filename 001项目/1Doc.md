# Memory
## 逐字解说
L1记忆是滑动窗口，存最近几轮对话的详细内容，保证短期上下文连贯；
L2记忆用Rolling Summary，定期把L1的内容压缩成摘要存起来，管理长期对话。
比如用户连续聊10轮文档创作，L1存最后3轮的具体需求，L2存前7轮的摘要，这样既能在token限制内记住用户习惯，又不会丢失关键信息。

双层记忆不仅记用户输入和最终输出，也会记录Agent间的关键流转信息。
比如L1滑动窗口会存“Planner拆分出Summary和Memorize任务→Worker执行→Critic检查通过”这样的核心步骤，确保短期对话中Agent能衔接之前的分工；
L2的Rolling Summary会压缩这些流转逻辑，比如“用户需要带架构图的总结，系统调用了Summary和Memorize Agent协作完成”，这样长期对话时，Agent依然知道之前是怎么配合的。

## QA
### 1. 如果用户在长对话中突然改变需求，双层记忆机制如何适应这种变化？
当用户改变需求时，L1滑动窗口会优先清空旧需求的详细信息，存入新需求的内容，确保短期协作聚焦当前任务；L2的Rolling Summary会新增一条“用户当前需求变更为XX”的摘要，同时保留之前的个性化偏好，比如用户之前喜欢简洁风格，新需求下依然会沿用这个偏好，不会完全丢失历史习惯。

新需求可以通过用户输入的关键词和意图来判断，比如之前用户一直说“生成论文总结”，突然说“帮我把总结翻译成英文”，这就是需求从“生成”变成“翻译”。这时候L1会优先存“翻译”或“不要架构图”这些新指令，L2保留“用户需要论文总结”的历史任务背景，帮助Agent理解新需求是基于什么任务来的。

### 2. Rolling Summary 用的什么方法？
用的是“任务-动作-结果”三元组压缩法，明确了要保留哪些类型的信息；
比如把“用户让生成论文总结，Planner调用Summary Agent，生成300字带架构图的内容，Critic检查通过”压缩成“任务：论文总结；动作：Summary+Memorize Agent协作；结果：带Mermaid图的300字总结”。
这种方法能直接保留对后续任务有用的核心要素，对包含Agent流转和用户偏好的对话来说，三元组压缩丢失关键信息的概率更低。

压缩时，还会用LLM对对话内容进行语义分析来填充这个框架，优先保留用户明确提到的偏好和任务核心要素，比如用户说“图表要简洁风格”，压缩时会特意保留“简洁风格”这个偏好词。如果遇到模糊信息，会参考历史对话中用户的选择来补充，比如用户之前每次选300字总结，压缩时就默认“结果：300字总结”，避免丢失隐性需求。

### 3. 如果L2的Rolling Summary积累了多条摘要，怎么决定哪条优先被调用？
会根据时间戳和需求相关性排序，最近的摘要优先，同时用LLM计算当前用户需求和每条摘要的语义相似度，相似度高的摘要排在前面，确保优先唤起和当前任务最相关的历史信息。

### 4. 记忆存在什么数据库里？
L1短期记忆可以存在Redis里，因为它需要高频读写，Redis的速度能满足实时对话需求。L2长期记忆的Rolling Summary适合存在PostgreSQL这类关系型数据库，方便按用户ID查询和更新长期偏好、任务历史这些结构化信息。

# RAG
## 逐字解说
RAG 这一块我们也没有做成固定链路，而是做成 Agentic RAG。

也就是说不是所有请求都默认检索，而是由 Retrieval Agent 判断当前任务是否需要 RAG。

如果需要，它会先做 Query Rewrite，把用户问题改写成更适合检索的查询，再执行 BM25 + 向量检索的混合召回，并做 rerank 和上下文压缩，最后把高相关内容注入 Workspace。

因为长上下文场景里，如果直接把 TopK chunk 全塞进去，很容易导致 token 爆炸，所以我们做了 Context Compression，比如 chunk summary、chunk merge 等。

### 1. 分片策略是什么样的？
对于普通文本，采用递归语义切片，先按段落分割，若段落长度超过512 token，再按句子递归拆分，确保每个分片在256-512 token之间，且不拆分完整句子。
对于Markdown格式文本，优先按标题层级分片，同一标题下的内容再按代码块、列表块分割，保证每个分片保留标题上下文和完整语义块，这样向量化时能更准确捕捉内容主题，提升召回相关性。

### 2. 那针对格式化文本，除了按标题和块分片，有没有处理过特殊格式，比如表格、公式或者代码块？
假设处理表格时，会将表头和对应行数据合并成“表头：值”的文本片段，公式保留LaTeX原格式，代码块按函数或逻辑块分割。

处理代码块，按语法树解析，比如借助tree-sitter工具，优先按函数、类这些语法单元切分，每个分片控制在1000到1500字符，同时保留函数签名和注释。

### 3. 来源回填是如何做的？
来源回填就是在RAG生成回答后，把引用的文档分片信息附在回答末尾。具体做法是检索时记录每个相关分片的元数据，生成回答时用这些元数据替换掉回答里的占位符，让用户知道信息来源。

### 4. 索引是如何做的？
索引用BM25关键词索引和向量相似度索引混合，融合时采用RRF方法：对每个分片，分别取BM25排名和向量排名，按1/(排名+60)计算分数后相加，最后按总分取Top结果，能稳定平衡关键词匹配和语义相关性。

### 5. 向量数据库用的什么？
使用 Redis Stack，自带了 RedisSearch 和 RedisJSON 这些模块，能直接存向量，还支持 HNSW 索引和余弦相似度这些常用的向量检索功能，部署和集成也方便，很适合 RAG 这种需要实时检索的场景。

### 6. 如果文档更新频繁，你们是怎么保证Redis里的向量索引和最新文档同步的？是全量重建还是增量更新？
可以设计增量更新策略，当文档有新增或修改时，只对变化的分片重新生成向量，然后调用Redis的FT.ADD或FT.UPDATE命令更新索引，删除旧分片的向量。如果是大量文档更新，可先暂停检索，全量重建索引后再切换，这样能减少对服务的影响。

### 7. 那在检索时，如果用户查询的是长文本问题，比如多轮对话后的复杂问题，向量索引还能准确召回相关分片吗？
可以先对长查询进行语义压缩，提取核心问题生成向量，再去检索。或者用多向量检索，把长查询拆成多个子问题向量，分别召回后合并结果，这样能提升长问题的召回准确性。

### 7. 在RAG系统里，你怎么评估检索到的文档和用户问题的相关性？有没有什么具体的指标或者方法？
召回率看是否能把所有相关分片都找出来，准确率看召回的分片中真正相关的比例。
除了这两个，还可以用NDCG评估排序质量，或者人工标注Top5分片的相关性打分。

### 8. 重排策略
重排阶段可以加入交叉编码器，对召回的Top10分片和用户问题做二次语义匹配，进一步提升相关性。
比如先用RRF拿到10个分片，再用交叉编码器打分排序取Top5，这样兼顾效率和准确性。


# Agents
## 逐字解说
这个部分做了一套文档内的 AI 多智能体协作系统，主要是想解决传统单 Prompt 模式在写作场景下的问题。

整体上设计的是一个Fast / Swarm 双通道架构。

1. Fast 模式主要处理低复杂度、低时延任务，比如翻译、简单润色等。这类请求会直接经过 Router Agent 路由对应到具体的 Worker Agent，然后调用 LLM 生成结果。
2. Swarm 模式主要处理复杂任务，比如长文档总结、Mermaid 生成等。复杂任务会进入多智能体协作链路。

它会先对请求做复杂度评估，比如：输入文本长度、是否需要知识库索引、是否涉及多步骤生成，然后决定走 Fast 还是 Swarm。

如果进入 Swarm 模式，会先由 Planner Agent 做任务拆解。

比如用户说：

‘请根据当前文档生成技术方案，并补 Mermaid 架构图。’

Planner 会把任务拆成几个步骤：

1. 先总结文档核心模块
2. 提取系统结构
3. 生成 Mermaid 草稿
4. 最后统一润色输出

之后任务会进入多个 Worker Agent 并行执行。

比如：

Summary Agent 负责总结
Retrieval Agent 负责 RAG 检索
Mermaid Agent 负责结构图生成
Writing Agent 负责最终文本生成

中间结果不会直接拼 Prompt，而是写入 Shared Workspace，也就是共享工作区，所有 Agent 都可以读取前序结果，而不需要重新理解全文。

生成完成后，还会经过 Critic Agent。

Critic 主要负责：检查结果是否偏题、引用合法性、Markdown 格式等问题。

如果发现问题，会进入 Reflection Loop，再次修正生成结果。

最后由 Merger Agent 做统一结果聚合，并转换成结构化协议返回前端。

因为我们这个场景不是普通聊天，而是编辑器内 AI 操作，所以前端不仅需要拿到文本，还需要知道：

应该如何回填
替换哪里
插入哪里
是否生成 Mermaid 块

基于此，我们设计了一套统一响应协议。

```
                    User Request
                           │
                   Lead Orchestrator
                           │
               Complexity Scorer Agent
                 ┌─────────┴─────────┐
                 │                   │
              FAST                SWARM
                 │                   │
          Router Agent         Planner Agent
                 │                   │
          Single Worker        Task DAG Planning
                 │                   │
                LLM        ┌─────────┼─────────┐
                           │         │         │
                    Retrieval   Writing   Mermaid
                      Agent      Agent      Agent
                           │         │         │
                           └──── Shared Workspace ────┐
                                                      │
                                                Critic Agent
                                                      │
                                                Reflection Loop
                                                      │
                                                Merger Agent
                                                      │
                                                 Final Output
```

## QA 
### 1. 这个框架相比单 Agent 方案，任务处理效率提升了多少？
多 Agent 并行写作能使复杂任务平均处理时间缩短，同时 评审的 Reflection Loop能提高输出正确率，降低 Markdown 格式错误等。

### 2. 为什么选择Planner-Worker-Critic三层架构？
因为科研文档创作任务复杂，Planner负责拆解任务保证逻辑清晰，Worker并行执行提升效率，Critic通过反思循环保证结果质量，三层分工比单一Agent更适配“复杂任务+高精度”的科研场景。

### 3. Blackboard Pattern在高并发下如何保证数据一致性？
我们给共享工作区的每个数据块加了版本号，Agent写入时需校验版本，避免覆盖冲突；同时用读写锁控制访问，Worker读时加共享锁，写时加排他锁，高并发下任务错误率控制在5%以内。

### 4. Critic的Reflection Loop触发阈值是什么？
比如Markdown语法错误≥2处、引用格式错误率＞10%、内容与文档核心主题匹配度＜80%时，会触发反思循环，针对性回溯对应Worker Agent修正。
还能加个收敛检测，要是连续两次反思改进幅度小于5%，也直接停，避免无效循环。

### 5. ReAct范式
简单说就是让模型“思考-行动-观察”循环，先分析任务该做什么，再调用工具执行，然后根据结果调整下一步，像人解决问题一样有逻辑地一步步推进，最后得出答案。

### 6. Memorize Agent如何抽象复杂系统结构生成Mermaid？
先通过Summary Agent的文档总结提取核心模块，再用规则库映射模块关系到Mermaid语法，比如“模块A调用模块B”转化为“ A --> B”，复杂结构拆分为子图表，确保前端渲染清晰。

### 7. 响应式协议如何定义回填位置？
协议包含“操作类型+起始位置+结束位置+内容”，比如“replace:100:200:生成文本”，前端根据起始/结束索引定位文档位置，替换或插入内容，适配编辑器实时更新。

### 8. Worker Agent结果冲突时如何仲裁？
冲突时由Planner Agent根据任务优先级仲裁，比如“生成结构图”任务中，Memorize Agent结果优先级高于Summary Agent，以Memorize结果为准写入共享工作区。

### 9. 基于 ReAct 范式，设置 Reflection Loop 阈值
可以结合任务类型动态设阈值，比如简单的文案生成设3次，复杂的技术文档设5次，超过就终止
