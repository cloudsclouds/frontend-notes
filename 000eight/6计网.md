# 计算机网络

## GET和POST的请求的区别

**GET**：

- 用于从**服务器获取数据**，一般**不会更改**服务器上的资源。
- **参数附加在 URL** 中，因此**数据长度有限**，有 URL 长度限制。
- **速度较快**，因为数据直接在 URL 中传输，适合查询操作。
- **安全性较低**，参数可能暴露在 URL 上。
- 对浏览器缓存友好，可以被缓存。

**POST**：

- 用于**向服务器发送数据**，通常会**更改**服务器上的数据，**比如表单提交**。
- 参数通过**请求体**传输，没有长度限制。
- **相对安全**，参数不会直接暴露在 URL 上。
- 不会被浏览器缓存。
- 处理速度比 GET 略慢，因为涉及请求体数据传输。

## HTTP 1.0 1.1 2.1 http3+UDP

> HTTP/1.0 和 1.1 是通过**请求行中的版本号区分的**；而 HTTP/2 和 HTTP/3 不再依赖请求头，而是在连接建立阶段通过**协议协商来确**定，比如 HTTP/2 通过 TLS 的 ALPN 协商为 h2，而 HTTP/3 基于 QUIC（UDP），在握手阶段就确定协议，同时可以通过 Alt-Svc 响应头告知客户端支持 h3。

**http 1.0:**

特点：

1. 每次请求都要需要**创建新的TCP连接**，请求完成之后断开连接。

**HTTP/1.1**：

1. 支持**长连接**，即**TCP连接上可以（串行）传送多个HTTP请求和响应**。
2. **支持请求流水线，可以同时并行发送多个请求**，**但服务器必须按请求顺序返回响应**。
3. **引入了更多的缓存控制策略**，如Cache-control 和 etag等缓存控制字段。
4. 仍然存**在队头阻塞问题，即一个请求被阻塞，其他请求也会被阻塞。**

**HTTP/2**：

- **多路复用**：**一个 TCP 连接可以并行处理多个请求**，解决**队头阻塞问题。**
- **头部压缩**：使用 HPACK 算法**压缩请求头**，节省带宽。
- **服务器推送**：服务器可以对⼀个客⼾端请求发送多个响应。服务器可以主动推送资源给客⼾端，⽽ ⽆需客⼾端明确地请求，减少了延迟。

> HTTP/2 的服务器推送是指在客户端请求一个资源时，**服务器可以主动将相关资源一并发送给客户端**，属于单向优化机制，主要用于减少请求延迟。
>
> 而 WebSocket 是一种独立协议，建立连接后支持客户端和服务器之间的双向实时通信，适用于聊天、实时数据等场景。两者的核心区别在于通信模式和使用场景。
>
> ```TypeScript
> const socket = new WebSocket('wss://yourserver.com/ws');
>
> socket.send(JSON.stringify({ type: 'chat', data: 'Hello' }));
> ```

- **流量控制**：提供更精细的流量控制。
- **HTTP/2无法解决的问题**：
  - **TCP的固有问题**：虽然解决了应用层的队头阻塞，但由于**底层TCP仍然基于单个连接**，存在**TCP队头阻塞**问题，特别是在高延迟和丢包场景下。{***TCP 层队头阻塞来自 TCP 的按序交付要求，丢包会阻塞整个连接。***}
  - **连接建立时的延迟**：TCP握手依然存在延迟。
- **HTTP/3 + UDP**：
  - **UDP**没有TCP的队头阻塞问题，因此通过使用QUIC协议实现的**HTTP/3**能避免TCP连接中的队头阻塞。
  - **多路复用**：QUIC协议**允许多个流在一个UDP连接中并行传输**，解决了TCP连接中的队头阻塞问题。
  - **更低的连接延迟**：QUIC通过0-RTT连接重用，减少了连接建立的延迟。
  - **更好的移动网络支持**：由于UDP是无连接的，适合移动网络中频繁切换网络的情况。

HTTP 层队头阻塞来自 HTTP/1.1 的串行响应机制，同一连接内的响应必须按顺序返回；TCP 层队头阻塞来自 TCP 的按序交付要求，丢包会阻塞整个连接。HTTP/2 通过多路复用解决了 HTTP 层的队头阻塞，但仍逃不掉 TCP 层的队头阻塞。HTTP/3 基于 QUIC，按流独立传输，从根本上解决了这两类问题。

## HTTP 和 HTTPS 的区别：

HTTPS 是在 HTTP 基础上加了 TLS/SSL 加密层的安全通信协议。

通过 TLS 协议，先用**非对称加密**安全地协商出一个**对称密钥**，然后用这个**密钥进行高效加密通信**，同时通过**证书机制保证服务器身份可信**，防止访问伪造网站；

还用完整性校验（MAC/HMAC/AEAD）防止数据在传输过程中被篡改。

客户端发起请求（ClientHello）

 服务端返回：

- 证书（含公钥）

 客户端：

- 验证证书（CA 签名）
- 生成一个 **随机对称密钥（pre-master secret）**
- 用服务器公钥加密发送给服务器

 服务端：

- 用私钥解密，拿到对称密钥

 双方开始用 **对称加密通信**

## HTTP 状态码

- **1xx**（信息性状态码）：表示请求已被接收，继续处理。
  - 例如：100 Continue
- **2xx**（成功状态码）：表示请求已成功处理。
  - 例如：200 OK、201 Created
- **3xx**（重定向状态码）：表示请求需要进一步的操作（如跳转）。
  - 301 Moved Permanently：请求的资源已经 **永久移动** 到新的 URL，以后所有请求都应该访问新的地址
  - 302 Found：请求的资源 **临时** 位于新的 URL，但以后可能还会恢复原地址
  - 304：客户端之前请求过该资源，并且已经缓存了，服务器判断 **资源自上次访问后没有改变**，**所以告诉浏览器直接使用本地缓存，不需要重新下载**
- **4xx**（客户端错误状态码）：表示请求有错误，由客户端引起。
  - 例如：**400 Bad Request、404 Not Found、403 Forbidden**
- **5xx**（服务器错误状态码）：表示服务器在处理请求时发生错误。
  - 例如**：500 Internal Server Error**、502 Bad Gateway

## 常见的HTTP请求方法

- **GET**：用于请求获取资源。
- **POST**：用于提交数据，通常用于创建或修改资源。
- **PUT**：用于更新资源。
- **DELETE**：用于删除资源。
- **HEAD**：类似于GET，但只请求头部。
- **OPTIONS**：用于获取支持的请求方法或查询服务器支持的选项。
- **PATCH**：用于部分更新资源。

## 与缓存相关的HTTP请求头有哪些

**浏览器缓存的本质是通过复用已下载的资源，避免重复请求，其核心机制分为强缓存和协商缓存两类。**

浏览器请求资源时，缓存判断的优先级是：

1. **发送请求前，浏览器先查找强缓存是否命中：**
  1. **命中 → 直接返回缓存资源（状态码 200, from cache）**
  2. **未命中 → 进入协商缓存阶段**
2. **协商缓存阶段：**
  1. **浏览器携带标识（**`If-None-Match` **或** `If-Modified-Since`**）询问服务器资源是否变更；**
  2. **若未变更 → 返回 304 Not Modified；**
  3. **若变更 → 返回 200 并更新缓存副本。**
3. **强缓存**：浏览器在缓存有效期内 **直接使用本地缓存**，**不发请求**到服务器，响应状态码是 **200**。

强缓存由**服务器通过响应头**控制，关键 HTTP 头有两个：

- **Expires**（ HTTP/1.0）： 服务器返回资源的 **“过期时间”**（绝对时间，格式如 `Expires: Wed, 21 Oct 2025 07:28:00 GMT`），在此之前不会再向服务器请求。
- **Cache-Control**（ HTTP/1.1）： 通过 **“相对时间”** 和**更灵活的缓存控制**，优先级高于 `Expires`，比如 `max-age` 指定缓存时间，`no-store` **不使用强缓存，每次请求都必须进入协商缓存流程**（让服务器判断是否复用缓存）等。`Cache-Control: max-age=31536000, public`

1. **协商缓存**：**需服务器判断是否复用缓存：** 缓存过期或没有强缓存时，浏览器**会向服务器发送请求**，通过 `ETag` 或 `Last-Modified` 判断资源是否变化；没变化返回 **304**，有变化才返回新资源。
  1. `ETag` / `If-None-Match`（ HTTP/1.1）：**由服务器生成文件内容的唯一标识，通过资源唯一标识**判断内容是否变化。
  2. `Last-Modified` / `If-Modified-Since`（ HTTP/1.0）：通过**最后修改时间判断是否更新。**
    如果资源未变，服务器返回 304 Not Modified，复用本地缓存，节省带宽。**

**如何让应用正确使用缓存**

**静态资源（JS / CSS / 图片）文件名哈希（如** `main.abc123.js`**）**

```JavaScript
location /static/ {
    add_header Cache-Control "max-age=31536000, immutable";
}
```

**动态接口数据：对接口数据使用协商缓存保证实时性。**

```JavaScript
Cache-Control: no-store
```

| 头字段               | 作用              | 示例                                    |

### **2. 请求头（Request Headers）**

#### **（1）客户端信息**


|                     |                          |                                      |
| ------------------- | ------------------------ | ------------------------------------ |
| 头字段                 | 作用                       | 示例                                   |

### 请求头**面试常问问题总结**

- **Content-Type 和 Accept 区别？**
`Content-Type` 表示请求体的格式，`Accept` 表示期望服务器返回的格式。
- **If-Modified-Since 和 If-None-Match 区别？**
前者基于时间（Last-Modified），后者基于资源标识（ETag），后者更精确。
- **Referer 的作用？**
表示请求来源页面 URL，用于防盗链、流量分析，但受 `Referrer-Policy` 控制，HTTPS → HTTP 时可能被移除。

## 跨域与同源策略

> 1. 什么是同源策略？
>
> 同源策略是浏览器的一个安全机制。
>
> 同源要求三个部分完全一致：
>
> - 协议（http / https）
> - 域名（[www.xxx.com](http://www.xxx.com)）
> - 端口号（8080、3000 等）
>
> 只要有一个不一样，就是跨域。
>
> **浏览器限制跨域：主要是为了防止 CSRF、XSS、窃取用户信息，保证安全。**
>
> 1. 什么是跨域？
>
> 当前端地址和接口地址不同源时，**浏览器就会拦截请求**，报 CORS 错误。
>
> 注意：
>
> - **请求发出去了**
> - **服务器也响应了**
> - **但浏览器拦截了响应，不给前端**
>
> **这就是跨域。**
>
> 1. 跨域解决方案（高频）
>
> **① CORS（最常用、后端配置）**
>
> 1️⃣ 简单请求（直接发）
>
> 满足以下条件：
>
> - 方法：GET / POST / HEAD
> - 请求头是“安全头”（如 `Content-Type` 仅限三种）
>   - `text/plain`
>   - `application/x-www-form-urlencoded`
>   - `multipart/form-data`
>
> 简单请求不会触发预检请求，但服务器仍然需要返回 CORS 相关响应头，比如 Access-Control-Allow-Origin，否则浏览器会拦截响应，前端无法获取数据。
>
> 2️⃣ 复杂请求（预检 OPTIONS）
>
> 不满足上述条件 → 就是复杂请求
>
> 比如：
>
> - PUT / DELETE
> - JSON（`application/json`）
> - 自定义 header
>
> **服务器在响应头里加：**
>
> Access-Control-Allow-**Origin**: [https://example.com](https://example.com)
>
> Access-Control-Allow-**Methods**: GET, POST
>
> Access-Control-Allow-**Headers**: Content-Type
>
> 浏览器看到这些头，就允许跨域。
>
> **② JSONP（老方案，只支持 GET）**
>
> **利用 script 标签不受跨域限制的特点。**
>
> 前端传一个回调函数名，后端返回函数调用包裹的数据。
>
> 缺点：只 GET、不安全、不支持 POST。
>
> **③ 代理服务器（开发最常用）**
>
> 开发环境用：
>
> - Webpack DevServer
> - Vite proxy
>
> 原理：
>
> **浏览器 → 本地代理（同源） → 服务器**
>
> **代理服务器转发请求，服务器之间没有跨域。**
>
> **④ Nginx 反向代理（生产常用）**
>
> **用 Nginx 把接口路径转发到后端服务。**
>
> 对前端透明，前端感觉就是同源。

- **同源策略**：浏览器为保证安全性，规定不同域之间的网页不能互相访问。只有在**协议、域名、端口**都相同的情况下，才能进行交互。
- **跨域问题**：当一个网页试图访问另一个域的资源时，就会遇到跨域问题。常见的跨域解决方法包括：
  - **CORS（跨源资源共享）**：通过设置服务器的`Access-Control-Allow-Origin`头部，允许特定域的访问。
  - **JSONP（用于GET请求）**：通过`<script>`标签**绕过浏览器的同源策略**。 浏览器的同源策略对 `<script>` 标签 **不做限制**，可以加载任意域名下的 JS 文件 ，**前端**创建一个 `<script>` 标签，请求目标接口，并在 URL 里带上一个回调函数名：

```HTML
<script src="https://api.example.com/user?callback=handleResponse"></script>
```

**后端**收到请求后，不是返回 JSON 数据，而是返回一段 JS 代码，比如：

```HTML
handleResponse({ name: "Ying", age: 22 });
```

这段 JS 被浏览器直接执行，调用了前端定义的 `handleResponse` 函数，前端就拿到了数据。

- **WebSocket**：通过WebSocket协议进行跨域通信。WebSocket 通过在 HTTP 握手后升级为 **TCP 长连接**协议 `Connection: Upgrade` `101 Switching Protocols`，在同一个连接上允许客户端和服务器独立、同时发送数据，从而实现真正的全双工通信。
- **代理**：通过同源的中间服务器转发请求，开发中常见于**本地代理配置**。
- **部署时用 Nginx / API 网关做代理，统一转发后端接口。** 前端访问的始终是“同源”的中间服务器（比如本地的开发服务器），由它帮你去请求目标跨域接口。浏览器认为请求的都是“同域”，所以不会触发跨域限制。

```JavaScript
// vite.config.js
export default {
  server: {
    proxy: {
      '/api': {
        target: 'https://other-domain.com', // 目标地址
        changeOrigin: true,                 // 修改请求头的 Origin
        rewrite: path => path.replace(/^\/api/, '') // 重写路径
      }
    }
  }
}
```

## TCP三次握手与四次挥手

### 1. 三次握手（建立连接）：

1. **SYN**：客户端向服务器发送一个SYN包（同步序列号，用于建立连接），表示请求建立连接，并发送自己的初始序列号`x`。
2. **SYN-ACK**：服务器收到SYN包后，回复SYN-ACK包， 表示同意建立连接，并告知ACK 确认包x+1，自己的初始序列号`y`
3. **ACK**：客户端收到SYN-ACK包后，回复ACK包 y+1 给服务器，**表示确认收到了服务器的响应。**  客户端进入 **ESTABLISHED** 状态，服务器收到这个 ACK 后也进入 **ESTABLISHED** 状态。

✅ **连接建立成功**。

**为什么是三次而不是两次？**

- 因为需要**双向确认**（客户端能发，服务器能收；服务器能发，客户端也能收）。
- 两次握手无法确保服务器的 SYN+ACK 包是否被客户端成功接收，可能导致“**半连接**”问题（服务器以为连接建立了，客户端却没收到，资源被浪费）。

### 2. 四次挥手（断开连接）：

1. **FIN**：客户端发送FIN包，表示 自己已经**没有数据要发送了**，请求关闭客户端 。
2. **ACK**：服务器收到FIN包后，回复ACK包。 表示“我知道你要关闭了，但我还可能有数据要发”。
3. **FIN**： 当服务器也发送完所有数据后，向客户端发送 FIN 包 ， 请求关闭服务器 。
4. **ACK**：客户端收到服务器的FIN包后，回复ACK包， **客户端等待一段时间（2MSL）后彻底关闭，**服务器收到这个 ACK 后直接关闭连接（CLOSED）。

**为什么要四次？**

- 因为 TCP 是**全双工协议**，断开连接必须由双方分别发出 FIN 并得到确认。
- 服务器在收到客户端的 FIN 时，可能还有数据没发完，因此不能和 ACK 合并发送，导致需要分两步完成（ACK + FIN）（第二次挥手+第三次挥手）。
- 假设服务器在第二次挥手时，**把 ACK 和 FIN 合并**，即客户端发 FIN → 服务器直接回 FIN+ACK → 客户端回 ACK。这样确实只用了三次，但前提是 **服务器在收到客户端 FIN 时，自己已经没有数据要发了**。
在实际 TCP 通信中，**这种情况不一定满足**，所以 TCP 协议设计时，**将 ACK 和 FIN 分开**，保证可靠传输。

👉 **TIME_WAIT 的意义**

- 确保客户端发送的最后一个 ACK 能被服务器收到；
- 避免旧连接的数据包在网络中延迟，影响新的连接（防止“延迟的重复分组”）。

## DNS解析的过程

1. **浏览器缓存**：首先查看浏览器的DNS缓存中是否有该域名的解析记录。
2. **操作系统缓存 （含 hosts）** ：如果浏览器没有缓存，再查看操作系统的DNS缓存。
3. **本地DNS服务器查询**：如果操作系统缓存中也没有，查询配置的本地DNS服务器。
4. **递归查询**：**根域名服务器**：本地DNS服务器首先向根域名服务器查询，根服务器知道顶级域（TLD）服务器的地址。
5. **顶级/TLD服务器**：根域名服务器返回TLD服务器的地址，本地DNS服务器向TLD服务器查询（[如.com](http://如.com)、.cn）。
6. **权威DNS服务器**：TLD服务器返回最终域名的权威DNS服务器地址，本地DNS服务器向权威DNS服务器查询该域名的IP地址。
7. **返回结果**：解析完成后，本地DNS服务器将结果返回给客户端， 最终浏览器拿到 IP 并进行后续连接。

## **Cookie、Session、 LocalStorage、SessionStorage、IndexedDB**

### 面试官问：“跨域请求能带 cookie 吗？”

你可以答：

> 默认不能，必须同时满足：
>
> 1. 前端设置 withCredentials / credentials
> 2. 服务端允许 Access-Control-Allow-Credentials
> 3. cookie 设置 SameSite=None + Secure

### **Cookie 与 Session 最大的区别**：

Cookie 数据存在**客户端**，Session 数据存在**服务器**；Cookie 可**被篡改**，Session 更**安全**。

### **LocalStorage 与 Cookie 的区别**：

LocalStorage 容量大，不随请求发送，适合存缓存数据；Cookie 容量小，会自动发给服务器。

### **LocalStorage 与 SessionStorage 的区别**：

LocalStorage 跨标签页共享，永久保存；

SessionStorage 仅当前标签页有效，关闭即清除。

1. cookie： 早期用于在**客户端存储少量数据**，以及**维持用户登录状态** （比如存储 session_id）大小限制：一般 ≤ 4KB（每个域名）。**每次请求都会带上（除非** `HttpOnly`**），可以设置过期时间。**

> **Cookie 的属性主要包括：**
>
> Expires 和 Max-Age 用于控制过期时间，
>
> Secure 表示只在 HTTPS 下传输，
>
> HttpOnly 防止 JavaScript 访问提升安全性，**不会通过可会断脚本访问**，只有http请求会携带这个cookie，帮助防止跨站脚本攻击。
>
> SameSite 用于控制跨站请求是否携带 Cookie。
>
> 其中 Max-Age 的优先级高于 Expires，SameSite 是防止 CSRF 攻击的重要手段。

1. Session 会话：存储在**服务器端**的一种用户状态管理方式。用户第一次登录后，服务器会创建一个 session（一般保存在内存或数据库），并返回对应的 session_id 给客户端。客户端通常用 Cookie 存这个 session_id，之后请求携带它，服务器就能找到该用户的 session。
2. LocalStorage （本地存储） ：HTML5 提供的**本地永久存储**方案。只在客户端保存，**不会随请求发送**。不适合存储敏感信息（比如 token 明文），因为前端能轻易读取。可以与 token 鉴权结合，比如保存 JWT。大小限制：一般为 5MB 左右（远大于 Cookie）有效期：永久保存，除非手动清除安全性：可被同源脚本读取，需防 XSS
3. SessionStorage （ 会话存储 ）：与 LocalStorage 类似，也是 HTML5 提供的**本地存储**，但**生命周期是“页面会话”**。**页面关闭即销毁**，即使同一个域名在新标签页打开，也不会共享。有效期：浏览器标签页关闭即清除跨页面：❌ 不同标签页、窗口之间不共享使用场景：表单临时保存、单页面应用临时状态、避免刷新丢失数据
4. IndexedDB：浏览器内置的**本地 NoSQL 数据库**。支持存储大量结构化数据（对象、文件等），支持事务。

## **非对称加密和对称加密的过程**

对称加密是指加密和解密使用相同的密钥。这意味着发送方和接收方需要共享同一个密钥来加解密数据。

非对称加密使用一对密钥——**公钥**和**私钥**。公钥和私钥是配对的，**公钥用于加密，私钥用于解密。加密过程与解密过程分别由不同的密钥完成。**

1. **密钥生成**：
  1. **每个用户都有一对密钥：公钥**和**私钥**。公钥是公开的，任何人都可以知道；而私钥是保密的，仅由用户自己持有。
2. **加密**：
  1. 发送方使用接收方的公钥对明文进行加密。由于公钥是公开的，任何人都可以用公钥加密数据，但只能用对应的私钥解密。
3. **数据传输**：加密后的密文通过网络传输给接收方。
4. **解密**：
  1. 接收方使用自己的私钥对密文进行解密，恢复出原始的明文数据。只有拥有私钥的人才能解密数据，即使是用公钥加密的密文。

## 网络模型

OSI 七层模型（理论模型）

从下到上依次是：

**物理层（Physical Layer）**

- 功能：传输比特流（0 和 1），定义物理接口、传输媒介、电压、速率等
- 示例：网线、光纤、网卡、Hub

**数据链路层（Data Link Layer）**

- 功能：建立可靠的链路，进行帧传输、错误检测和流量控制
- 示例：以太网、MAC 地址、交换机

**网络层（Network Layer）**

- 功能：选择路径，实现不同网络之间的互联（路由）
- 示例：IP 协议、路由器

**传输层（Transport Layer）**

- 功能：**端到端通信**，保证数据可靠传输
- 示例：TCP（可靠）、UDP（不可靠）

**会话层（Session Layer）**

- 功能：建立、管理和终止会话
- 示例：RPC、NetBIOS

**表示层（Presentation Layer）**

- 功能：数据格式转换、加密/解密、压缩
- 示例：JPEG、ASCII、SSL

**应用层（Application Layer）**

- 功能：直接为用户提供服务
- 示例：HTTP、FTP、SMTP、DNS

**TCP/IP 模型（实际使用更广）**

TCP/IP 模型简化为 4 层或 5 层（和 OSI 对应关系略有不同）：

**链路层 / 网络接口层**

- 对应 OSI 的物理层 + 数据链路层

**网络层 / Internet 层**

- 对应 OSI 的网络层
- 核心协议：IP

**传输层**

- 对应 OSI 的传输层
- 核心协议：TCP、UDP

**应用层**

- 对应 OSI 的会话层、表示层、应用层的功能
- 核心协议：HTTP、FTP、SMTP、DNS

## TCP vs UDP

**1️⃣ 连接方式**

- **TCP**
  - **面向连接**：通信前必须先建立连接（三次握手），结束时四次挥手
  - 保证双方“握手成功”后才能传输数据
- **UDP**
  - **无连接**：直接发送数据，不需要建立连接
  - 没有握手过程，发送快但不可靠

**2️⃣ 可靠性**

- **TCP**
  - **可靠传输**：保证数据按顺序到达，丢包会重传
  - 提供流量控制和拥塞控制
- **UDP**
  - **不可靠传输**：不保证数据顺序，也不重传
  - 发送速度快，但可能丢包或乱序

**3️⃣ 传输方式**

- **TCP**
  - 面向 **字节流**（stream）
  - 数据被拆分成数据段（segment），接收方按顺序重组
- **UDP**
  - 面向 **报文（datagram）**
  - 一次发送一个完整数据包，包大小有限制（一般 ≤ 64 KB）

**5️⃣ 应用场景**

- **TCP**
  - 对**可靠性要求高**的场景：网页浏览（HTTP/HTTPS）、文件传输（FTP）、邮件（SMTP）
- **UDP**
  - 对**速度和实时性要求高**、可容忍少量丢包的场景：视频会议、在线游戏、DNS 查询、VoIP

> UDP 本身是不可靠的，但可以在应用层通过引入**序列号、ACK 确认、超时重传、滑动窗口**等机制来实现可靠传输。这些机制本质上类似于 TCP 的实现，例如通过序列号保证顺序，通过 ACK 和重传保证数据不丢失，通过窗口机制提高传输效率。在实际中，像 QUIC 协议就是基于 UDP 实现可靠传输的典型例子。

## 前端安全

XSS 是跨站脚本攻击，攻击者通过**注入恶意 JS 代码**，**在用户浏览器执行**，可能窃取 Cookie、篡改 DOM 或劫持操作。

 防护方法包括：

1. **输出编码**：**对用户输入进行 HTML/JS/URL 转义**，避免直接 innerHTML，浏览器不会把转义后的字符当作可执行代码，从而防止执行恶意 JS。
2. CSP（Content Security Policy）通过**限制资源加载来源**，本质是“白名单机制”，**限制脚本来源，从浏览器层面阻止 XSS 执行。**
3. Cookie 设置为（HttpOnly）

CSRF **利用用户在浏览器的登录状态**（Cookie、Token），**在第三方网站或脚本中发起伪造请求**，从而执行用户未授权的操作（如转账、修改密码）。

前端防护方法：

**使用 CSRF Token**：请求携带服务端生成的 token，服务端校验。

**SameSite Cookie**：限制第三方站点不能发送 Cookie。

## JWT

> “JWT 是一种自包含的 **JSON Web Token**，用于**客户端和服务器之间的身份认证**。
>
> 它由 **Header、Payload 和 Signature** 组成。
>
> 登录成功后**服务器生成 JWT 并返回给客户端**，客户端每次请求带上 token，服务器通过**签名验证和过期时间**校验来识别用户身份。
>
> JWT 的优点是无状态、易于分布式扩展，但 **Payload 信息不加密，敏感信息不能放在里面**，需要注意签名算法和过期策略。”

JWT 是一种轻量级、基于 JSON 的**自包含令牌**，用于在客户端和服务器之间安全地传递身份信息。

JWT 是一个**字符串**，由 **3 部分** 组成，使用 `.` 分隔：

```JavaScript
<Header>.<Payload>.<Signature>
```

Header（头部）：描述 token 类型和签名算法

{

 "alg": "HS256",

 "typ": "JWT"

}

Payload（负载）：存放声明（Claims），如用户 ID、过期时间、权限

{

 "sub": "1234567890",

 "name": "Alice",

 "iat": 1680000000,

 "exp": 1680003600

}

Signature（签名）：用 **Header + Payload + Secret** 生成 ，用于验证 token 是否被篡改

HMACSHA256(base64UrlEncode(Header) + "." + base64UrlEncode(Payload), secret)

### JWT 工作流程

1️⃣ 用户登录

1. 用户提交用户名/密码
2. 服务器验证成功
3. 生成 JWT（Header + Payload + Signature）
4. 返回给客户端（通常放在 `Authorization: Bearer <token>` 或 localStorage / cookie）

2️⃣ 客户端请求接口

1. 每次请求都携带 JWT（Header / Cookie / Authorization）
2. 服务器拿到 JWT：
  1. Base64 解码 Header 和 Payload
  2. 用 Secret 验证 Signature
  3. 检查 `exp` 是否过期
3. 验证通过 → 执行接口逻辑
4. 验证失败 → 返回 401 / 403

3️⃣ 无状态机制

- JWT 自带用户信息和权限
- 服务端无需存储 session，方便 **分布式/微服务**
- 安全依赖于 **Secret** 或 **私钥（RSA/ECDSA）**

### Jwt vs cookie+session

**Cookie + Session 是“服务端存状态”，JWT 是“客户端存状态”。**

1️⃣ Cookie + Session

风险：

- CSRF（跨站请求伪造）

解决：

- SameSite
- CSRF Token

2️⃣ JWT

风险：

- XSS（token 被偷）
- token 一旦泄露 → 可直接使用

解决：

- **不存 localStorage（推荐 httpOnly cookie）**
- 设置过期时间（exp）
- 使用 refresh token

**Session 是服务端存储用户状态，安全但扩展性差（分布式需要共享 session，如 Redis）；JWT 是客户端自包含认证信息，无状态、易扩展（浏览器 → 任意服务器 → 校验 JWT → OK），但需要额外机制保证安全和失效控制。**
