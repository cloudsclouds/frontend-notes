# AI

## 1. AI 使用的体验，如何才能使得产出更符合预期？
在开发中用 AI 时，明确需求描述很重要，不能过于笼统，而是尽可能详细。

给 AI 提供上下文也关键，比如把项目中已有的组件结构、接口文档片段发给它。

另外，对 AI 生成的代码要自己过一遍，结合业务场景调整，毕竟 AI 可能忽略项目特有的约束。

## 基础概念

#### 什么是 LLM？

**LLM (Large Language Model)** 是基于 Transformer 架构的大规模预训练语言模型，通过海量文本数据学习语言规律和世界知识。

#### 什么是 Prompt Engineering？有哪些策略？

**Prompt Engineering** 是通过设计和优化输入提示词，引导 LLM 生成高质量输出的技术。

**核心就是 4 件事：**

**1️⃣ 明确任务（Task）：告诉模型你要干嘛**

> “帮我总结这篇文章” “写一个Vue组件”

**2️⃣ 给上下文（Context）：提供背景信息**

> “这是一个面试场景”“目标读者是初学者”

**3️⃣ 规定输出格式（Format）：控制结果长什么样**

> “用表格输出”“分点说明”“用逐字稿形式回答”

**4️⃣ 加约束（Constraints）：限制模型乱发挥**

> “不超过200字”“必须包含代码示例”“不要使用专业术语”

**常见提示词技巧**

**1. Few-shot（少样本提示）：**给例子让模型模仿，模型更容易学会规律（Zero-shot learning、one-shot learning、few-shot learning）

输入：1+1=2 输入：2+2=4 输入：3+3=？

**2. Chain of Thought（思维链）：**让模型一步步推理，对算法题 / 推理题非常有用

> “请一步一步分析再给出答案”

**3. 角色设定（Role Prompting）：**让输出更专业、更符合场景

> “你是一名资深前端面试官，请评价这段代码”

**4. 分步骤任务（拆解复杂问题）**

❌ 一次性问：写一个电商系统

✅ 拆解

1. 设计数据库
2. 写接口
3. 写前端页面

#### 什么是 Token？上下文长度如何影响性能？

**Token** 是 LLM 处理文本的最小单位，可以是单词、子词或字符。

**Token 特点：**

- 英文：1 token 约等于 0.75 个单词
- 中文：1 个汉字约等于 1-2 tokens
- 通过 BPE (Byte Pair Encoding) 等算法切分

**上下文长度影响：**

|       |                            |
| ----- | -------------------------- |
| 上下文长度 | 影响                         |
| 太短    | 无法处理长文档、多轮对话               |
| 适中    | 平衡性能与成本                    |
| 太长    | 注意力稀释、推理成本增加、KV Cache 内存压力 |

**优化策略：**

- **滑动窗口**：分段处理长文本
- **摘要压缩**：提取关键信息
- **RAG 检索**：只加载相关片段

#### Temperature / Top-p / Top-k 区别？

这三个参数控制 LLM 输出的随机性和多样性。

1. **temperature**：**控制模型输出的“随机性 / 创造性”**

大模型在生成文本时，会**对每个候选词计算概率**，然后“采样”： Temperature 会**调整这些概率的分布。**

温度低 👉 更“保守”、更确定；温度高 👉 更“发散”、更有创意

1. **Top-k：**只**从概率最高的 k 个词中采样**

Top-k = 5，只在概率最高的5个词中选，限制胡乱生成，提高稳定性

1. **Top-p：** 从“**累计概率达到 p 的候选集合**”中采样，不固定数量，而是“概率覆盖范围”

Top-p = 0.9，选最可能的一批词，直到概率加起来 ≥ 0.9

|                          |                      |      |     |
| ------------------------ | -------------------- | ---- | --- |
| 参数                       | 作用                   | 取值范围 | 典型值 |
| Temperature              | 控制概率分布的平滑程度          | 0-2  | 0.7 |
| Top-p (nucleus sampling) | 从累积概率前 p 的 token 中采样 | 0-1  | 0.9 |
| Top-k                    | 从概率最高的 k 个 token 中采样 | 1-无穷 | 50  |

**使用场景：**

- **Temperature 0.1-0.3**：代码生成、数学计算**（确定性高）**
- **Temperature 0.7-1.0**：创意写作、头脑风暴**（多样性高）**
- **Top-p 0.9 + Top-k 50**：平衡质量与多样性

#### 什么是 幻觉？如何缓解？

**Hallucination（幻觉）** 指 LLM 生成**看似合理但实际错误或无依据的**内容。

**幻觉类型：**

- **事实性幻觉**：编造不存在的事实
- **忠实性幻觉**：与输入上下文不一致

**缓解策略：**

|        |                       |
| ------ | --------------------- |
| 方法     | 原理                    |
| RAG 检索 | 提供外部知识支撑              |
| 事实核查   | 多模型交叉验证               |
| 约束生成   | 限制输出范围（如 JSON Schema） |
| 置信度校准  | 让模型表达不确定性             |
| 人类反馈   | RLHF 训练减少幻觉           |

#### **完整调用链**

用户输入 → Agent 规划 → RAG 检索知识 → Function Call 调用工具 → Skills 组合能力 → MCP 通信 → 返回结果

> 用户输入后由 Agent 进行意图解析和任务规划，根据需求先做 RAG 知识检索，再匹配对应的 Skills 能力，随后通过 Function Call 调用工具，并基于 MCP 协议完成跨服务通信，最后聚合结果返回给用户。

## Transformer

### Transformer 架构是什么？

**Transformer** 是 2017 年 Google 提出的基于注意力机制的神经网络架构，完全摒弃了 RNN/CNN，成为现代 LLM 的基础。

**核心组件：**

|                      |                 |
| -------------------- | --------------- |
| 组件                   | 作用              |
| Embedding            | 将 token 转为向量    |
| Positional Encoding  | 注入位置信息          |
| Multi-Head Attention | 多头自注意力，捕捉不同维度关系 |
| Feed-Forward Network | 前馈网络，非线性变换      |
| Layer Norm           | 层归一化，稳定训练       |
| Residual Connection  | 残差连接，缓解梯度消失     |

**架构特点：**

- Encoder-Decoder 结构（GPT 只用 Decoder）
- 并行计算（相比 RNN 的串行）
- 长距离依赖建模能力强

### Self-Attention 如何计算？Q / K / V 是什么？

**Self-Attention 计算流程：**

1. **线性变换**：输入 X 分别乘以 WQ、WK、WV 得到 Q、K、V
2. **计算注意力分数**：Attention(Q, K, V) = softmax(QK^T / sqrt(d_k)) * V

**Q / K / V 含义：**

|           |     |                    |
| --------- | --- | ------------------ |
| 矩阵        | 含义  | 作用                 |
| Q (Query) | 查询  | 表示当前 token 想查询什么信息 |
| K (Key)   | 键   | 表示每个 token 能提供什么信息 |
| V (Value) | 值   | **实际传递的信息内容**      |

**为什么除以 sqrt(d_k)：**

- **防止点积结果过大**，**导致 softmax 梯度消失**
- d_k 是 key 向量的维度

### 为什么 Attention 比 RNN 强？

|       |           |             |
| ----- | --------- | ----------- |
| 特性    | RNN       | Attention   |
| 并行性   | 串行计算      | 完全并行        |
| 长距离依赖 | 梯度消失，难以捕捉 | 直接连接，距离无关   |
| 计算复杂度 | O(n) 每步   | O(n^2) 整体   |
| 可解释性  | 弱         | 强（注意力权重可视化） |

**Attention 优势：**

- 任意两个 token 直接交互
- 不受序列长度影响（理论上）
- 多头机制捕捉不同子空间信息

### KV Cache 是什么？为什么能优化推理？

**KV Cache** 是在自回归生成过程中**缓存已计算 token 的 Key 和 Value 矩阵**，避免重复计算。

**推理阶段特点：**

- 生成第 n 个 token 时，需要关注前面所有 n-1 个 token
- 如果不缓存，每次都要重新计算前面所有 token 的 K、V

**优化效果：**

- 时间复杂度：从 O(n(3) 降到 O(n)2)
- 内存开销：增加约 2  *n*  d_model 的缓存

**优化策略：**

- **PagedAttention**：vLLM 提出的分页管理
- **MQA/GQA**：多查询/分组查询注意力，减少缓存
- **量化压缩**：INT8/INT4 降低缓存大小

### 推理阶段和训练阶段有什么区别？

|                |            |                 |
| -------------- | ---------- | --------------- |
| 阶段             | 训练         | 推理              |
| 目标             | 更新模型参数     | 生成输出            |
| 计算方式           | 并行处理整个序列   | 自回归逐个生成         |
| Attention Mask | 因果掩码（只看前面） | 同左              |
| KV Cache       | 不需要        | 必须使用            |
| 批处理            | 固定长度 batch | 动态长度，需要 padding |
| 精度要求           | FP16/BF16  | 可量化到 INT8/INT4  |

## RAG 检索增强生成

## 基础理解

**什么是 RAG？解决什么问题？**

**RAG (Retrieval-Augmented Generation)** 是将外部知识检索与大语言模型生成相结合的技术框架。

**解决的问题：**

|       |               |
| ----- | ------------- |
| 问题    | 说明            |
| 知识时效性 | LLM 训练数据有截止日期 |
| 幻觉问题  | 模型编造不存在的信息    |
| 领域知识  | 专业领域知识不足      |
| 可溯源性  | 无法追溯信息来源      |

**为什么 RAG 能减少幻觉：**

- 将检索到的真实文档作为上下文
- 限制模型基于提供的证据生成
- 支持引用溯源

**RAG vs Fine-tuning？**

|      |         |             |
| ---- | ------- | ----------- |
| 维度   | RAG     | Fine-tuning |
| 数据需求 | 无需训练    | 需要标注数据      |
| 成本   | 低（检索成本） | 高（训练成本）     |
| 时效性  | 实时更新知识库 | 需要重新训练      |
| 灵活性  | 高       | 低           |
| 适用场景 | 知识频繁更新  | 特定任务风格      |

**选择建议：**

- **知识问答 → RAG**
- **风格迁移 → Fine-tuning**
- **两者结合 → 先 RAG 检索，再用微调模型生成**

## RAG Pipeline

**RAG 全流程是怎样的？标准 Pipeline：**

```Plain
文档加载 → 文本分割 → Embedding → 向量存储 → 检索 → 重排序 → 生成
```

**各环节详解：**

|              |                         |
| ------------ | ----------------------- |
| 环节           | 技术要点                    |
| 文档加载         | PDF、Word、网页等多格式解析       |
| 文本分割 (Chunk) | 按段落、句子、token 数分割        |
| Embedding    | 使用 BERT、OpenAI 等模型向量化   |
| 向量存储         | FAISS、Milvus、Pinecone 等 |
| 检索           | 相似度搜索、混合检索              |
| 重排序 (Rerank) | Cross-encoder 精排        |
| 生成           | 将检索结果拼入 Prompt          |

## Chunk 怎么切？

**切分策略：**

|      |          |           |
| ---- | -------- | --------- |
| 策略   | 适用场景     | 优缺点       |
| 固定长度 | 通用场景     | 简单，可能切断语义 |
| 递归切分 | 结构化文档    | 保留段落/句子边界 |
| 语义切分 | 长文档      | 按语义主题分割   |
| 重叠切分 | 需要上下文连续性 | 增加冗余，提高召回 |

**最佳实践：**

- Chunk 大小：256-512 tokens（视模型上下文而定）
- 重叠：20-30% 防止信息丢失
- 保留元数据：来源、标题、时间等

| ---------------------------- | -------------- | ------ |

- 语言匹配：中英文分别选优化模型
- 领域匹配：医学/法律可选领域专用模型
- 成本考虑：开源 vs 商业 API

## BM25 vs 向量检索 vs Hybrid Search？

|        |           |          |       |
| ------ | --------- | -------- | ----- |
| 方法     | 原理        | 优势       | 劣势    |
| BM25   | 基于词频的稀疏向量 | 精确匹配、可解释 | 语义理解弱 |
| 向量检索   | 稠密向量相似度   | 语义理解、容错性 | 计算成本高 |
| Hybrid | 两者结合      | 兼顾精确与语义  | 系统复杂  |

**Hybrid Search 公式：**

```Plain
最终分数 = alpha * 向量分数 + (1 - alpha) * BM25分数
```

**典型配置：**

- 关键词查询（如 ID、代码）→ BM25 权重高
- 自然语言问题 → 向量检索权重高

## 如何优化召回率 vs 精确率？

> **召回率：在所有“真正相关”的结果中，你找回了多少，有没有“漏掉”重要结果。**
>
> **精确率：在你找出来的结果中，有多少是真正相关的，有没有“垃圾结果”。**

**召回率优化：**

- **扩大检索数量**（Top-K 从 5 增加到 20）
- **多路召回**：向量 + 关键词 + 图谱
- Query 扩展：同义词、相关词

**精确率优化：**

- **重排序（Rerank）模型精排**
- 过滤低相似度结果
- 引入用户反馈学习

**平衡策略：**

- 两阶段检索：**粗排（召回）**+ **精排（精确）对已经检索出来的一批候选结果，再用更“精细”的模型重新排序，选出更相关的结果。**
- 动态阈值：根据 query 类型调整

|              |          |
| ------------ | -------- |
| **Query 类型** | **策略**   |
| 简单关键词        | BM25     |
| 长问题          | 向量检索     |
| 模糊 query     | TopK 更大  |
| 高价值 query    | 强 rerank |

## 如何做 Rerank？

1. **召回候选（Recall / Retrieval）**
  1. 先从整个库里用快速方法找到 **Top-N** 候选（比如 Top-100）
  2. 方法可以是关键词匹配、embedding 检索、BM25 等
  3. 目的是保证高召回率，把可能相关的内容都抓出来
2. **Rerank（重排序 / Fine-grained Ranking）**
  1. 对候选列表逐个打分，用更精细的模型计算 **相关性分数**
  2. 模型可以是 **BERT/RoBERTa** 类的语义匹配模型，或者专门训练的排序模型
  3. 通过分数重新排序，把最相关的结果排在前面
3. **选取最终结果（Top-K）**
  1. 从重新排序后的列表取 Top-K（比如 Top-5）作为最终输出
  2. 可以送入生成模型做回答或推荐

**Rerank 方法：**

|               |                        |               |
| ------------- | ---------------------- | ------------- |
| 方法            | 原理                     | 代表模型          |
| Cross-encoder | Query 和 Doc 一起编码，计算相关性 | BERT-reranker |
| ColBERT       | 晚期交互，平衡效果与效率           | ColBERT       |
| LLM Rerank    | 用 LLM 判断相关性            | GPT-4 判断      |

## 多文档冲突怎么处理？

**冲突类型：**

- 事实矛盾：不同文档给出不同答案
- 时效冲突：新旧版本信息不一致
- 视角冲突：不同立场/观点

**处理策略：**

- 时效优先：优先使用最新文档
- 置信度加权：根据来源权威性加权
- 多答案呈现：列出不同观点让用户选择
- 一致性检测：LLM 检测并指出冲突

## 长文档如何处理？

**处理方案：**

|            |             |      |
| ---------- | ----------- | ---- |
| 方案         | 原理          | 适用   |
| 分层摘要       | 先分段摘要，再整体摘要 | 超长文档 |
| Map-Reduce | 分块处理，聚合结果   | 批量处理 |
| 滑动窗口       | 滑动检索相关片段    | 流式处理 |
| 图谱构建       | 提取实体关系建图    | 复杂关联 |

## GraphRAG 是什么？

**GraphRAG** 是将知识图谱与 RAG 结合的技术，先构建实体关系图谱，再基于图谱检索。

**与传统 RAG 区别：**

|      |        |             |
| ---- | ------ | ----------- |
| 特性   | 传统 RAG | GraphRAG    |
| 数据结构 | 平面向量   | 图结构         |
| 关系捕捉 | 弱      | 强（显式关系）     |
| 多跳推理 | 困难     | 天然支持        |
| 构建成本 | 低      | 高（需要抽取实体关系） |

**适用场景：**

- 复杂多跳问答
- 关系密集型领域（医疗、法律）

## Agent 智能体

## 基础问题

**什么是 Agent？和普通 LLM 的区别？**

**Agent** 是能够感知环境、自主决策、执行动作并完成目标的智能系统。

**与 LLM 区别：**

|     |         |                |
| --- | ------- | -------------- |
| 特性  | LLM     | Agent          |
| 能力  | 文本生成    | 规划 + 记忆 + 工具使用 |
| 交互  | 单轮/多轮对话 | 与环境持续交互        |
| 主动性 | 被动响应    | 主动规划执行         |
| 工具  | 无       | 可调用外部工具        |

## MCP 与工具链

## MCP 基础

在大模型应用中，我理解 MCP 的本质不是一个具体框架，而是一种“**上下文协议化**”的设计思想。

它解决的是 **prompt 拼接不可控的问题**，通过**结构化 context**、**标准化 tool 调用以及流程编排，**把原本隐式的模型交互变成可管理、可扩展的系统。

在前端侧，我主要参与的是**会话状态管理、上下文裁剪优化以及 tool 调用**的 UI 映射，并结合流式响应提升交互体验。本质上是把模型行为转译成用户可感知的界面状态，同时保证性能与可维护性。

#### MCP 是什么？解决了什么问题？

**MCP (Model Context Protocol)** 是 Anthropic 提出的开放协议，标准化 AI 模型与外部工具/数据源的连接方式。

**解决的问题：**

- 工具接入碎片化：每个工具不同接口
- 上下文传递困难：模型与工具数据格式不统一
- 生态封闭：各平台工具不互通

**核心价值：**

- 一次接入，多处使用
- 标准化工具描述
- 安全可控的权限管理

#### MCP 和传统 API 调用有什么区别？

|     |           |           |
| --- | --------- | --------- |
| 维度  | 传统 API    | MCP       |
| 协议  | HTTP/REST | 标准化协议     |
| 描述  | 文档说明      | Schema 定义 |
| 发现  | 人工对接      | 自动发现      |
| 上下文 | 手动传递      | 协议内置      |
| 安全  | 各自实现      | 统一权限      |

#### MCP vs Function Call？

|     |                        |                       |
| --- | ---------------------- | --------------------- |
| 特性  | Function Call          | MCP                   |
| 层级  | 模型能力                   | 通信协议                  |
| 作用  | 模型输出调用意图               | 标准化工具通信               |
| 关系  | MCP 可以使用 Function Call | Function Call 不依赖 MCP |

**关系：**
MCP 是更高层的协议，Function Call 可以是 MCP 的实现方式之一。

## 蒸馏模型

**目标**：把一个**大模型（Teacher，老师模型）学到的知识“蒸馏”到一个小模型（Student，学生模型）**，让小模型在性能接近大模型的前提下，更轻量、速度快、占用资源少。

大模型（Teacher）在训练数据上生成 **soft target**（通常是经过 softmax 的概率分布，而不是单一的 one-hot 标签）。

小模型（Student）训练目标是同时拟合原始真实标签 **和** 大模型的输出概率分布。

损失函数通常是加权的：

L = \alpha \cdot L_{hard} + (1-\alpha) \cdot L_{soft}

- L_{hard} ：交叉熵损失对真实标签
- L_{soft} ：KL散度损失对教师模型输出的软标签

## 训练、预训练、微调

**训练 = 用数据让模型学会做任务的过程**

**预训练 = 用海量通用数据，让模型学会“语言能力”**

**微调 = 在预训练基础上，用“小而精”的数据，让模型适应特定任务**

## 微调方式：

1. **SFT（监督微调）**

👉 给标准答案，输入：问题，输出：标准回答

1. **RLHF（基于人类反馈的强化学习）**

Step1：收集数据（prompt）

Step2：模型生成多个回答

Step3：人工排序（标注）

Step4：训练奖励模型（Reward Model）

Step5：强化学习优化（PPO等）

1. **LoRA （Low-Rank Adaptation（低秩适配））**

**低成本微调：**不改整个模型，只训练很少参数”的微调方法

大模型里有很多线性层：y = W × x，W 是一个很大的矩阵

LoRA做了什么？不直接改 W，而是加一个“增量”：W′=W+BA

W：原始参数（冻结 ❌不训练） ；B、A：小矩阵（要训练 ✔️）

为什么叫“低秩”？因为：BA 的维度很小（rank 很低）
