# 工程化

## Tree Shaking

**Tree Shaking** 是一种 **静态分析并移除未使用代码（Dead Code Elimination）** 的技术，主要用于减少**最终打包体积**。

基于 **ES Module 的静态特性，**构建工具（如 Webpack、Rollup、Vite）在编译阶段 **分析** `import` / `export` 依赖关系，从而确定哪些代码是被使用的，哪些是未被引用的，然后在打包时将未使用的代码移除。

## **核心流程**

1. **构建依赖图：**基于 **ES Module** 的静态导入特性（`import/export`），构建工具在构建阶段即可生成完整的 **模块依赖图**。
2. **AST 静态分析：**工具会将源代码解析为 **AST（抽象语法树）**，并标记哪些导出（`export`）被使用。
3. **标记未使用的代码：**构建工具遍历 AST，对每个 `export` 做标记：如果有引用 → **used；**如果无引用 → **unused**。
4. **删除未使用代码（DCE）：**在打包阶段，构建工具根据标记移除未使用的导出。**UglifyJS / Terser** 等压缩工具会根据标记的 dead code 进行消除。

### 为什么 Tree Shaking 依赖 ES Module

**ES Module 的特点**：

- **静态依赖分析**：`import` 和 `export` 语句在编译阶段就能确定模块依赖图。

**CommonJS 的问题**：

- `require()` 是动态调用，依赖无法在编译阶段完全确定，构建工具无法准确识别未使用代码。

### **构建工具如何实现 Tree Shaking**

通常在生产模式下，**Webpack、Rollup 或 Vite 都会自动开启 Tree Shaking 来优化最终体积。**

### **如何检查 Tree Shaking 是否生效？**

使用 Webpack 的 `stats` 分析工具，或者打包后查看是否移除了未使用代码。也可以通过 `Bundle Analyzer` 检查最终包体积和模块依赖。

### **Tree Shaking 一定能移除所有未使用的代码吗？**

不一定。

如果模块**存在副作用**，比如修改全局变量、注册事件、执行初始化逻辑、引入 CSS 等，构建工具为了保证运行结果正确，就不会删除该模块。

此外，如果 **Babel 把 ESM 转成 CommonJS**，或者存在动**态 import / require**，也会影响 Tree Shaking 的静态分析能力。

副作用（side effects）：

> **一个模块被 import 后，你没有使用它导出的任何变量，它依然会执行一些对外部环境产生影响的代码。**
>
> 因为构建工具无法确定“删掉这个模块会不会影响运行结果”，所以它就不敢删除。
>
> 设置`sideEffects: false` 是 package.json 中给打包工具的优化提示，表示项目中的模块默认没有副作用，可以安全地进行 Tree Shaking。

## Webpack

### Webpack 的作用

> Webpack 是一个用于现代 JS 应用程序的**静态模块**打包工具，它的核心作用可以概括为三点：
>
> 1. **模块打包（Module Bundling）**
>
> - 将项目中散落的 JS、CSS、图片等资源模块打包整合，保证依赖引用正确。
>
> 1. **编译兼容（Loader）**
>
> - 利用 Loader 机制，Webpack 能够编译转换浏览器无法直接识别的文件类型，比如 `.ts`、`.less`、`.vue`。
>
> 1. **功能扩展（Plugin）**
>
> - 通过 Plugin 机制，Webpack 提**供按需加载、代码压缩、环境变量注入**等高级功能。

### Webpack 的构建流程

> Webpack 的构建流程可以分为几个核心阶段。
>
> 首先是**初始化阶段**，Webpack 会**读取配置文件并创建 Compiler 对象**。
>
> 接着进入**编译阶段：**
>
> 从 **entry 入口文件**开始**递归解析**所有依赖模块。
>
> 在解析过程中，Webpack 会通过 **Loader 对不同类型的文件进行转换**，比如将 CSS 或 TypeScript 转换成浏览器可识别的 JavaScript，同时通过 **AST 分析构建依赖关系**。
>
> 然后**根据依赖关系生成 chunk**，实现代码拆分。
>
> 最后进入**输出阶段**，**将打包后的资源输出到指定目录**。
>
> 在整个流程中，**Plugin 会通过钩子机制介入各个阶段**，对构建过程进行扩展和优化。

- **Webpack5 优化**：持久化缓存、资源模块替代 file-loader、Tree-shaking 增强、模块联邦、提升构建速度；
- **模块联邦**：

> 模块联邦是 Webpack5 提供的一种运行时模块共享机制，可以从远程应用动态加载模块，实现跨项目共享代码，常用于微前端架构，相比 npm 的构建时依赖，它更加灵活但也带来版本和网络上的复杂性。

### Loader vs Plugin

> Loader 主要用于**处理非 JavaScript 文件**，本质**是一个函数**，它在**模块加载时对文件内容进行转换**，通过**module.rules配置使用**，例如 babel-loader、sass-loader，
>  Plugin 是用来**扩展 Webpack 功能**的，本质**是一个类**，通过**监听 Webpack 的生命周期钩子来介入整个构建流程，通过插件机制绑定到Webpack实例上**，比如**代码压缩、生成 HTML、定义环境变量等**。
>  简单来说，Loader 解决“文件转换问题”，Plugin 解决“功能扩展问题”

**Loader：**

- `babel-loader`：把 ES6 转换成 ES5
- `ts-loader` / `esbuild-loader`：处理 TypeScript
- `sass-loader` / `less-loader`：样式预处理，将 SCSS/SASS/Less 代码转换成 CSS
- `file-loader`：把文件输出到文件夹中，在代码中通过相对 URL 去引用输出的文件 (处理图片和字体)
- `url-loader`：与 file-loader 类似，区别是用户可以设置一个阈值，大于阈值会交给 file-loader 处理，小于阈值时返回文件 base64 形式编码 (处理图片和字体)
- `i18n-loader`: 国际化

**Plugin：**

- `HtmlWebpackPlugin`：生成 HTML 并注入打包资源
- `DefinePlugin`：注入全局变量
- `BundleAnalyzerPlugin`：可视化打包体积分析
- `MiniCssExtractPlugin`：提取 CSS
- `CopyWebpackPlugin`：复制静态文件

```JavaScript
├── src/
│   ├── index.js
```

```JavaScript
npm init -y
npm install webpack webpack-cli --save-dev
npm install style-loader css-loader --save-dev
```

1. 编写入口文件、 Webpack 配置文件

```JavaScript
const path = require('path')

module.exports = {
  mode: 'development',

  entry: './src/index.js',   // 从入口（entry）开始构建依赖图

  output: {   // 输出
    filename: 'bundle.js',
    path: path.resolve(__dirname, 'dist'),
  },

  module: {
    rules: [
      {
        test: /\.css$/,
        use: ['style-loader', 'css-loader'],  // Loader 是 从右往左执行
      },
    ],
  },

  plugin: [
      new HtmlWebpackPlugin({
          template: path.resolve(__dirname, 'public/login.html'), // 模块文件
          filename: path.resolve(__dirname, 'dist/login/index.html') //输出文件
      })
  ]
}

```

1. 执行打包

```JavaScript
npm run 自定义命令（定义在 package.json 中）
```

最终输出为：

```JavaScript
dist/bundle.js
```

### Webpack HMR

**HMR = Hot Module Replacement**

HMR，也就是热模块替换，是指**在不刷新页面的情况下更新代码模块。它的实现原理是 Webpack Dev Server 会和浏览器建立一个 WebSocket 连接**，**当本地文件发生变化时，Webpack 对模块重新编译打包，通过**`websocket` **告知浏览器执行热更新逻辑。**

浏览器接收到更新信息后，会`http`请求去服务器端获取新的模块资源，并通过 HMR API 将旧模块替换为新模块，而不会刷新整个页面。

这样可以保留应用状态，提高开发效率。本质上，HMR 的核心就是只更新变化的模块，而不是重新加载整个应用。

### 提高 Webpack 打包速度

1. 优化loader配置
2. 优化resolve.alias 起别名
3. 使用cache-loader 进行缓存
4. HappyPack 启动多线程
5. 合理使用 sourceMap

**优化loader配置，精准匹配文件，缩小文件范围**

在使用`loader`时，可以通过配置`include`、`exclude`、`test`属性来精准地匹配哪些文件应用`loader`。

使用include来指定编译文件夹，exclude排除指定文件夹。

```JavaScript
module.exports = {
  module: {
    rules: [
      {
        // 如果项目源码中只有 js 文件就不要写成 /\.jsx?$/，提升正则表达式性能
        test: /\.js$/,
        // babel-loader 支持缓存转换出的结果，通过 cacheDirectory 选项开启
        use: ['babel-loader?cacheDirectory'],
        //使用include来指定编译文件夹只对项目根目录下的 src 目录中的文件采用 babel-loader
        include: path.resolve(__dirname, 'src'),
                          //使用exclude排除指定文件夹
                          exclude: /node_modules/,
      },
    ]
  },
};
```

**优化 resolve.alias 起别名**

alias给一些常用的路径起一个别名，特别当我们的项目目录结构比较深的时候，一个文件的路径可能是./../../的形式

通过配置alias以减少查找过程

```JavaScript
module.exports = {
    ...
    resolve:{
        alias:{
            "@":path.resolve(__dirname,'./src')
        }
    }
}
```

> **alias 发生在 Loader 之前、打包之前、代码执行之前**

**使用 cache-loader 缓存**

在一些性能开销较大的`loader`之前添加`cache-loader`，以将结果缓存到磁盘里，显著提升二次构建的速度。

注意：保存和读取这些缓存文件也会有一些时间开销，所以只对性能开销较大的`loader`使用此`loader`

```Python
module.exports = {
    module: {
        rules: [
            {
                test: /\.ext$/,
                use: ['cache-loader', ...loaders],
                include: path.resolve('src'),
            },
        ],
    },
};
```

**HappyPack 启动多线程**

HappyPack 可以将 Loader 的同步执行转换为并行的，这样就能充分利用系统资源来加快打包效率了。

这是一个Webpack插件，它可以将模块的加载和构建过程分解为多个子进程，从而实现多线程构建。

```JavaScript
module: {
  rules: [
    {
      test: /.js$/,
      loader: 'happypack/loader?id=js',
      include: [resolve('src')],
      exclude: /node_modules/
    }
  ]
},
plugins: [
  new HappyPack({
    id: 'js',//与rule中指定的id一致
    // 开启四个线程
    threads: 4,//必须配置，用于指定开启几个 worker，建议设置成 os.cpus().length - 1
    verbose: true,//是否输出详细的日志信息
    loaders: ['babel-loader?cacheDirectory'],//如何处理 js 文件，如将其转成 es5 代码
  })
]
```

**合理使用Source map**

在开发环境下，因为需要调试源代码，一般会开启Source Map。

而在生产环境下，为了保护代码的安全且获得更好的性能，我们一般不开启Source Map。

Source map 的模式

1. 开发环境下开启：在开发环境下启用Source map可以帮助开发者快速定位错误和调试代码。

- 但是在构建生产代码时应该禁用，确保代码被混淆和压缩。

1. 使用`Cheap`模式：开启Cheap模式可以大幅度减少构建时间，它只会标记行而不会标记列，因此精度较低，但对于代码调试已经足够。
2. 使用`eval`模式：开启eval模式可以进一步缩短构建时间，因为它会将Source map嵌入到eval中，从而避免了Source map文件的生成和加载。
3. 使用`inline`模式：将Source map嵌入到打包文件中，可以避免Source map文件的下载，但会增加打包文件的大小。
4. 使用`external`模式：将Source map文件分离到单独的文件中，可以避免打包文件过大，同时也便于对Source map文件进行管理。

### **Webpack5 优化**

1. 持久化缓存:

Webpack 5 引入了内置的持久化缓存功能，可以显著提高构建性能。通过缓存模块和生成的 chunk，重复构建的速度会更快。

1. Tree-shaking 增强：

通过静态代码分析的方式，识别出未使用的模块、函数、变量等，并将其从最终的构建输出中删除。

1. 模块联邦：

指 多个独立的构建可以组成一个应用程序，这些独立的构建之间不应该存在依赖关系，因此可以单独开发和部署它们。

它允许一个 JavaScript 应用动态地运行来自另一个应用的代码，无需重新编译。它允许不同团队独立开发和部署应用的各个部分。

## Vite

### Vite 的构建流程

> Vite 的构建流程分为两个阶段：开发阶段和生产构建阶段。
>
> **在开发阶段**，**Vite 不会像 Webpack 一样进行整体打包**，而是**基于浏览器原生的 ES Module，实现按需编译。它启动一个本地开发服务器，然后利用 esbuild** （是用go写的，主要用于 JavaScript 和 TypeScript 的编译、打包和压缩，速度比webpack快很多）**对依赖进行预构建**，**把 CommonJS 转换成 ESM**，同时**合并模块以减少请求数**。
>
> 浏览器在加载页面时，请求哪个模块，Vite 就实时编译并返回哪个模块，从而实现非常快的启动速度。transform 过程中会调用插件体系，比如 **.vue 文件会通过 Vue 的编译器转成 JS**。 同时它**通过 WebSocket 实现 HMR**，只更新变更的模块，而不是整页刷新。
>
> 在**生产环境**下，执行 build 时，Vite 会**调用 Rollup**，从 index.html **作为入口开始构建依赖图**。然后进行模块分析和代码分割，比如动态 import 会被拆成多个 chunk。同时进行 Tree Shaking，移除未使用代码。最后对 CSS、图片等资源做优化，并生成带 hash 的静态文件输出到 dist 目录。

## 构建工具：Vite vs Webpack

> 1. **构建方式不同**
>
> Webpack 是一个**静态模块打包器**。它的核心思想是将所有资源（包括 JS、CSS、图片等）视为模块，通过**递归解析依赖，构建一个完整的依赖图**。在开发环境下，Webpack **需要先打包好所有的资源**，然后才能启动开发服务器，项目越大，启动速度越慢。
>
> Vite 它是**基于现代浏览器原生 ES 模块**的支持，开发阶段利用浏览器直接请求模块，实现了**按需编译**。Vite 启动时非常快，不需要预先打包整个项目。**当浏览器请求某个模块时**，Vite 会动态地将该模块编译并返回。**但生产环境仍需要打包优化**，因此内部使用 **Rollup** 进行构建。
>
> 1. **热更新机制不同**
>
> Webpack **需要重新编译模块，**虽然`webpack`现在有了缓存机制，但还是无法从根本上解决这个问题。
>
> `Vite`项目中，监听到文件变更后，会用`websocket`通知浏览器，重新发起新的请求，只对该模块进行重新编译，然后进行替换。
>
> 1. **适用范围**
>
> Webpack 更适合于**大型、复杂**的企业级项目，或者需要**高度定制化构建配置**的项目。而 Vite 则非常适合于**现代、中小型**项目，尤其是使用 Vue、React 等主流框架的项目。

## 模块化

模块是为 Node.js 打包 JS 代码的原始方式。

在 Node.js 中， 每个文件都被视为一个单独的模块。

模块化能提高代码复用性，按需加载，独立作用域；使用标准语法导入和导出使用。

- **Webpack/Rollup/Vite**
  - 通过依赖图分析模块关系，打包成浏览器可执行的文件。
- **Tree Shaking**
  - 按需引入，移除未使用的代码，减小体积。
- **代码分割**
  - 动态 `import()` 实现懒加载，优化首屏性能。

### **CommonJS**

导出：`module.exports = {}`

导入：`require('模块名或路径')`

模块在运行时加载，整个模块生成一个对象，然后从对象上读取方法。

### **ES6 Module**

静态化模块，编译时就能确定依赖关系，通过 `export`/`import` 显式指定输出和输入，支持 Tree-shaking。

1. **export（命名导出导入）**

导出：`export 修饰定义语句`

导入：`import { 同名变量 } from '模块名或路径'`

1. **可以导出多个变量/函数/类**，每个导出都有自己的名字。
2. 导入时必须使用**相同的名字**（可以使用 `as` 别名重命名）。
3. 适合导出模块中多个功能或工具函数。

```JavaScript
// math.js
export const pi = 3.14159;
export function sum(a, b) { return a + b; }
export class Circle {}

// 导入
import { pi, sum, Circle } from './math.js';

// 可以重命名
```

### **CommonJS VS ESmodule**

> 1. **用法不同：**
>
> - **CommonJs** 是 Node **最早引入的模块化方案**，采用 **require 和 module.exports** 实现模块的导入和导出
> - **ES module** 是 **ES6 官方标准**，使用**import/export** 关键字实现模块的导入和导出。
>
> 1. **加载方式不同：**
>
> - **运行时加载:** CommonJS 模块就是对象；即在**执行时是先加载整个模块，生成一个对象**，然后再**从这个对象上面读取方法**，这种加载称为“运行时加载”。
> - **编译时加载**: ES module 模块不是对象，实现编译时静态加载，**在编译阶段就确定依赖关系，只加载需要的导出，形成静态依赖图。并且支持 tree-shaking。**

## dev server 常见配置

> host: 设置开发服务器的主机地址，默认是 [localhost](http://localhost)。

幽灵依赖捏

幽灵依赖是指在package.json文件中没有直接声明，但在项目中被使用的依赖。这种情况通常发生在项目中使用了另一个依赖，而该依赖本身又依赖于某些其他的依赖，但在项目中并没有直接使用这些依赖的情况下。由于这些依赖没有在package.json文件中声明，在使用PNPM等包管理工具时可能会被视为无用依赖而被删除，导致项目无法正常运行。

依赖安装方式

NPM和Yarn都会在node_modules目录下为每个包单独安装一份依赖，而PNPM则会共享依赖，将所有包的依赖都安装在全局目录中，并通过符号链接的方式链接到各个包中。这种方式可以大大减少磁盘空间的占用，加快依赖的安装速度。

安装速度

PNPM相对于NPM和Yarn来说，会更快一些，因为它能够更高效地利用磁盘空间和缓存，减少重复下载和解压依赖的时间。

对于本地包的支持

npm会将本地包的完整拷贝放置在项目的node_modules目录下。而PNPM通过符号链接的方式将本地包链接到项目中，只需要在全局目录中安装一次本地包，就可以在多个项目中共享使用。

PNPM相对于NPM和Yarn来说，更容易支持本地包。PNPM支持将已经存在于本地文件系统中的包链接到项目中，这些包可以是已经存在于其他项目中的，或者是手动下载的。这种方式可以方便地进行本地包的开发和测试。

PNPM在处理本地包依赖关系时，可以自动检测并安装本地包所依赖的其他包，而NPM需要手动将本地包的依赖添加到package.json文件中。

## 性能优化方案

## 一、渲染优化

### 虚拟列表（Virtual List）

**适用场景**：长表格、数据卡片流、日志列表等，数据量从几千到几万条。

**核心原理**：

- 浏览器性能瓶颈主要在于 DOM 数量，虚拟列表的核心是“只渲染可视区域的元素”。
- 利用滚动容器高度与单个元素高度的比值动态计算应渲染的数据索引区间，替换滚动事件中默认全量更新逻辑。

**技术实现方式**：

- 自研：结合滚动事件与 `requestAnimationFrame` 动态挂载、卸载 DOM 节点。
- 使用库：如 `vue-virtual-scroller`、`react-window`。

**注意事项**：

- 列表项高度固定更容易实现；
- 高度不一致时需做预估与动态调整，可能需要提前测量 DOM 高度或设置“锚点缓存”。

### 骨架屏 + 分块渲染（Chunk Rendering）

**适用场景**：首屏数据大、组件嵌套复杂，首次渲染白屏时间长。

**实现方式**：

- 骨架屏：使用 `Skeleton` 组件（Ant Design / Element UI等）或 Tailwind CSS 占位样式模拟数据区块。
- 分块渲染：将数据切成小段，配合 `requestIdleCallback`、`setTimeout` 分帧插入 DOM，避免主线程长时间阻塞。

**效果评估指标**：

- First Contentful Paint（FCP）
- Time To Interactive（TTI）

## 二、数据可视化优化

### 图表数据量优化

**问题背景**：一次性渲染上万条图表数据，会导致 UI 卡顿、图表组件崩溃（尤其是 SVG-based 图表）。

**解决方案**：

- 数据预聚合：在接口层或前端缓存层对数据做聚合处理（如 10000 条取每 100 条平均值，降维为 100 个点）
- 数据抽样：配置图表库的 `sampling` 属性，仅展示关键走势
- 数据懒加载：缩放/滑动图表时按需请求局部数据

**图表库建议**：

- ECharts：`dataZoom` + `sampling` + `progressive` 属性组合
- AntV G2 / G2Plot：配合 Web Worker 做数据处理，或转用 Canvas 渲染
- 大数据场景下尽量避免使用 SVG 图表，推荐 Canvas 或 WebGL 图表（如 [Deck.gl](http://Deck.gl)、L7）

## 三、交互性能优化

### 1. 高频事件优化（如 scroll、resize、input）

**问题**：

- 高频触发事件容易引起主线程阻塞，导致掉帧和 UI 卡顿。

**解决方案**：

- 函数节流（throttle）或防抖（debounce），控制事件触发频率；
- 使用 `requestAnimationFrame` 替代 `setTimeout`，在下一帧渲染前执行逻辑。

### 2. DOM 操作合并（Batching）

**问题**：重复、分散的 DOM 操作会引发多次重排（reflow）和重绘（repaint）

**解决方案**：

- 使用文档片段 `DocumentFragment` 或 `requestIdleCallback` 合并更新操作；
- 在状态变更中优先集中管理（如使用 Pinia store 触发统一 UI 更新）

## 四、网络与状态管理优化

### 1. 请求合并与缓存

**优化点**：

- 使用 Axios 二次封装，添加请求缓存层（如 memory cache / LRU cache）；
- 请求重试机制：接口失败后自动重试（带延时指数退避）；
- 合并请求：多个组件共享相同请求时，仅发一次请求。

### Web Worker 进行数据预处理

**原理**：

- 将数据排序、筛选、搜索等计算任务转移到子线程中执行，主线程只负责 UI 更新。

**使用方式**：

```JavaScript
const worker = new Worker('./filter-worker.js');
worker.postMessage(largeData);
worker.onmessage = ({ data }) => {
  this.filteredList = data;
};
```

## 五、构建与加载优化

### 懒加载与代码分割

**实现方式**：

- 路由懒加载：`import(/* webpackChunkName: "home" */ './Home.vue')`
- 异步组件加载：结合 `defineAsyncComponent`

**结果评估**：

- Bundle 大小减少；
- 首屏加载速度提升（如 3.6s → 1.7s）

### Tree Shaking 与第三方库裁剪

- 避免全量引入（如 `import _ from 'lodash'` → `import cloneDeep from 'lodash/cloneDeep'`）
- 移除未使用的模块；
- 配合 Webpack 插件如 `TerserPlugin`，移除 dead code。

## 六、性能监控与预警体系

**建议工具**：

- Sentry / 腾讯蓝鲸 / 阿里ARMS：监控 JS 报错、页面卡顿、加载耗时；
- 自研打点系统 + Lighthouse 报告 + Performance API：持续跟踪 FCP、TTI、LCP、CLS 等核心指标。

## 前端当中图片格式各自的使用场景，在项目中如何压缩图片来优化性能

### 一、前端常见图片格式及使用场景

### **1. JPEG / JPG**

- **特点**：有损压缩，体积小，色彩丰富，支持 24 位真彩色，不支持透明。
- **使用场景**：**照片、色彩丰富的背景图**（如 Banner、大图）。
- **优点**：压缩率高，体积小。
- **缺点**：有损压缩导致细节丢失，不适合图标或文字。

### **2. PNG**

- **特点**：无损压缩，支持透明（Alpha 通道），支持 8 位、24 位、32 位色。
- **使用场景**：**图标、Logo、透明背景图片**。
- **优点**：清晰度高，支持透明。
- **缺点**：体积大，不适合大图。

### **3. GIF**

- **特点**：支持动画，256 色，支持简单透明。
- **使用场景**：**动图、表情包**。
- **优点**：支持动画，兼容性强。
- **缺点**：色彩少，文件较大，效果有限。

### **4. WebP**

- **特点**：Google 推出的新格式，支持有损/无损压缩，支持透明和动画。
- **使用场景**：**替代 JPEG / PNG**，特别是在移动端和性能敏感场景。
- **优点**：比 JPEG、PNG 体积更小（30%-80%）。
- **缺点**：部分旧版浏览器不支持（可通过 polyfill）。

### **5. SVG**

- **特点**：基于 XML 的矢量图，可无限缩放，代码描述图形。
- **使用场景**：**Logo、Icon、简单图形**。
- **优点**：体积小，支持 CSS、JS 操作，可缩放不失真。
- **缺点**：不适合复杂图像（照片）。

### **6. AVIF**

- **特点**：新一代图片格式，压缩比优于 WebP，支持透明。
- **使用场景**：**极致性能优化的场景**。
- **优点**：更小体积，图像质量更好。
- **缺点**：浏览器兼容性尚未完全普及。

### 二、项目中压缩图片优化性能的策略

### **1. 构建时图片优化**

- **使用工具**：
- `image-webpack-loader`（Webpack 插件）
- `gulp-imagemin`（Gulp 任务）
- `svgo`（SVG 压缩）
- **做法**：
- **JPEG 使用 mozjpeg**，**PNG 使用 pngquant**，开启压缩选项。
- 在打包时自动压缩，减少体积。

### **2. 格式优化**

- 优先使用 **WebP 或 AVIF** 代替 JPEG/PNG（配合 `<picture>` 标签做回退）。

```
<picture>
  <source srcset="image.avif" type="image/avif">
  <source srcset="image.webp" type="image/webp">
  <img src="image.jpg" alt="example">
</picture>
```

### **3. 按需加载（懒加载）**

- 使用 `loading="lazy"` 或 Intersection Observer API。

```
<img src="image.jpg" loading="lazy" alt="example">
```

### **4. 压缩传输**

- 开启 **Gzip 或 Brotli 压缩**，减少网络传输体积。

### **5. CDN 处理**

- 使用 CDN **按需裁剪 + 压缩 + WebP 转换**，比如阿里云 OSS、七牛云。
- 可根据 `Accept` 头自动返回最佳格式。

### **6. 运行时压缩（上传前）**

- 使用 **前端压缩库**：
- `compressorjs`（JPEG 压缩）
- `pica`（图片缩放）
- `browser-image-compression`（支持多格式）

示例：

```
import imageCompression from 'browser-image-compression';

const options = {
  maxSizeMB: 1,
  maxWidthOrHeight: 1920,
  useWebWorker: true
};

const compressedFile = await imageCompression(file, options);
```

**实践总结**：

- 大图：WebP / AVIF 优先，JPEG 次选。
- 图标：SVG 优先，小图用 PNG。
- 动图：用 video 或 Lottie 替代 GIF。
- 构建阶段压缩 + CDN 转码 + 懒加载。
