# 操作系统

## 1. 浏览器与操作系统之间的关系

### （1）进程与线程管理

- Chrome 多进程架构：Browser / Renderer / GPU / Utility 等进程
- 每个进程的创建、销毁、调度均由操作系统完成（Windows 内核、Linux 内核）。
- 渲染进程中的 Script Thread / Compositor Thread / Raster Thread 都对应真实的 OS 线程。

### （2）内存分配

- 浏览器通过 OS 的虚拟内存接口（如 malloc, VirtualAlloc, mmap）向系统申请大块区域
- JS 引擎（如 V8）在内部再建立自己的 heap、栈、young space、old space
- 页面压力大会触发 OS 换页（page fault），进而导致明显卡顿

### （3）网络能力

浏览器无法直接发 TCP 包，而是通过操作系统的网络栈：

- DNS 解析由 OS resolver 或 stub resolver 完成
- TCP 三次握手、滑动窗口、拥塞控制均由 OS 内核处理
- 浏览器只是对 socket 做读写，并把事件派发到 EventLoop

### （4）图形绘制

- DOM → Layout → Paint → Rasterize → Composite → Surface
- 最终通过 GPU 进程调用操作系统的图形 API：

- Windows：DirectX
- macOS：Metal
- Linux：OpenGL / Vulkan

- 最后由 OS 的窗口管理器（DWM、Quartz）合成到屏幕

浏览器几乎所有能力都是对操作系统 API 的高层包装。

## 2. JS 的单线程模型与操作系统的多线程如何协作？

### 核心：**JS 主线程永远只有一个，但浏览器会使用多个系统线程来完成异步任务，然后通过事件循环把结果返回。**

### （1）JS 单线程的原因

- 避免多线程同时修改 DOM 导致状态一致性问题
- 使开发模型更简单：无锁、无竞争

### （2）浏览器如何用多线程协作

虽然 JS 是单线程，但浏览器内核会启动多个线程来执行耗时任务：

|                 |                                 |
| --------------- | ------------------------------- |
| 任务              | 实际执行位置                          |
| setTimeout      | 定时器线程（Timer Thread）*不是 JS 执行线程* |
| 网络请求            | Network Thread，真正用 socket 的是 OS |
| 文件读取            | OS 的 File I/O 线程                |
| 图片解码            | Image Decode Thread             |
| Layout / Raster | 由独立 Compositor/Raster Thread 完成 |
| Web Worker      | 独立的系统线程                         |

当这些线程任务完成时，会把事件加入主线程的 EventLoop 中排队。

### （3）EventLoop 的意义

EventLoop 只是调度中心，不执行异步操作本体。
异步操作本体由浏览器线程或 OS 线程完成。

这是前端理解性能和异步的基础。

## 3. 浏览器为何禁止 JS 直接访问文件系统？

原因是绝对的 **安全模型设计**。

### （1）如果 JS 可以读本地文件，会发生什么？

任何网站都可以：

- 读取你的 SSH 密钥
- 读取你的微信/浏览器密码数据库
- 读取公司源代码、文档
- 写入木马
这几乎是致命漏洞。

### （2）浏览器的安全沙箱

浏览器对页面 JS 设置了非常强的限制：

- 页面 JS 无法调用系统 API
- 无法访问真实的文件路径
- 无法访问其他应用的进程、内存
- 无法进行任意网络访问（必须遵守 CORS）

### （3）允许的方式必须获得用户授权

如：

- <input type="file">（需要用户选择）
- File System Access API（必须弹窗授权）
- sandboxed OPFS（隔离的虚拟文件系统）

核心理念是：网页是不可信的代码，绝不能给它操作系统权限。

## 4. 进程与线程的区别与前端场景对应

## （1）概念区分

|      |             |              |
| ---- | ----------- | ------------ |
| 维度   | 进程（Process） | 线程（Thread）   |
| 资源拥有 | 拥有独立内存空间    | 共享进程内存       |
| 通信成本 | 高（IPC）      | 低（共享内存）      |
| 稳定性  | 一个崩溃不影响其他   | 一个崩溃可能拖垮整个进程 |
| 调度单位 | 进程          | 线程           |

## （2）浏览器中的体现

|                            |                     |
| -------------------------- | ------------------- |
| 前端场景                       | 对应操作系统概念            |
| Chrome 的每个 Tab 一个渲染进程      | OS 的独立 process      |
| JS 主线程                     | 渲染进程内的一个 OS thread  |
| WebWorker                  | 渲染进程中新创建的 OS thread |
| Compositor / Raster Thread | 浏览器内部多线程渲染          |
| Node.js 的 Worker Threads   | OS threads          |

## （3）为什么浏览器使用多进程？

- 单个页面崩溃不会影响其他页面
- 提升安全性（沙箱）
- 利用多核 CPU

## （4）为什么 JS 本身不允许多线程访问 DOM？

避免出现：

- 写时竞争（race condition）
- 锁问题（mutex、deadlock）
- 内存一致性模型复杂化

## 5. 页面卡顿的根本原因

页面卡顿并不是“JS 多，而是主线程被阻塞

## 主要原因：

### （1）JS 执行时间过长（最常见）

例如：

- 大量数据处理
- 复杂循环
- JSON 超大对象解析
- 加密算法、图像处理
- ~~**解决方案：WebWorker/WASM（展望的时候才能说**~~

### （2）频繁 DOM 修改导致 Layout / Reflow

构建 DOM 树、计算布局、合成绘制需要耗时。

### （3）渲染层压力过大

- 大量动画
- 超大图像
- Canvas 连续高频重绘
- 纹理过大导致 GPU 内存爆掉

### （4）内存泄漏

导致：

- 浏览器内存占用持续上升
- OS 触发 Swap/page fault → 卡顿非常明显
尤其是长时间运行的管理后台。

### （5）阻塞 I/O

如：

- localStorage 是同步 API
- 同步 XHR 造成主线程 freeze

这些必须避免。

## 6. 死锁的面试题

前端不直接使用锁，但死锁概念仍经常会被问。

### **（1）什么是死锁？**

**多个线程/任务彼此等待对方释放资源，最终所有人都无法继续。**

经典条件（同时满足才会死锁）：

**互斥**（资源只能一个线程使用）

**占有并等待**

**不可抢占**

**循环等待**

### **（2）前端实际中可能出现的死锁场景**

**场景 1：WebWorker 间消息循环导致互相等待**

Worker A 等 B 回消息

Worker B 又等 A 回消息

双方都卡死

### **场景 2：SharedArrayBuffer + Atomics**

JS 中唯一可能“锁住”的情况

例如线程 A 持有锁不释放，线程 B 一直阻塞 Atomics.wait

### **场景 3：Service Worker 与页面之间循环请求**

页面等待 SW 返回

SW 又同步等待页面某个事件

→ 卡死

### **场景 4：Promise 死锁（任务永远不 resolve）**

最常见于编码错误：

```ts

await new Promise(() => {}) // 永不 resolve
```

导致：
后续 async 逻辑全部卡住
EventLoop 空转

### **场景 5：同步 XHR + 主线程资源竞争**
主线程执行同步 XHR
XHR 需要主线程处理事件
双方等待 → 页面完全假死

### **（3）如何回答如何避免死锁？**
可以直接背这个：
尽量避免共享可写状态
使用不可变数据结构
避免长时间持有资源
Worker 间通信要有超时机制
避免使用同步 XHR
避免写永不 resolve 的 Promise
